# 保姆级大修指南

这是一份给“完全不懂这两个仓库”的实现指南。

它同时覆盖两个仓库：

- `wanxiangshu`：多代理插件运行时，负责 review、KG、nudge、方法论工具、历史折叠与消息投影。
- `wanxiangzhen`：opencode 协调器插件，负责 DAG、worktree、slave 启动、ff-only 合并、事件回放。

这份指南的目标不是“讲概念”，而是告诉你要改什么、为什么改、先改谁、后改谁、哪些旧分支不能再留。

## 先看结论

1. `multi-frontmatter` 不是新真相，也不是旧真相的替代品。
2. 它是“与原有工具输入、工具输出、LLM 输入输出、历史、回放、投影、宿主 wire format 等效的事实源表达”。
3. 原有事实源仍然有效，不能删、不能降级、不能用“只有单块 front matter 才算真相”这种老模型。
4. `wanxiangshu` 和 `wanxiangzhen` 都要按这个模型改。
5. `wanxiangzhen` 还要顺手收紧依赖：安装它的人必须先有 `wanxiangshu`，所以它可以把“万象术不存在”的 fallback 分支删掉。

## 读者先搞清楚的背景

### 1. 什么是 `wanxiangshu`

`wanxiangshu` 是 F# → Fable → JS 的多代理插件运行时。它把稳定语义放在 `Kernel/`，把副作用边界放在 `Shell/`，再分发到 `Opencode/`、`Mux/`、`Omp/` 三个宿主适配层。

它最重要的长期事实源是“历史”。会话历史、review 历史、todo 历史、KG 事件历史都比当前内存状态更可信。

### 2. 什么是 `wanxiangzhen`

`wanxiangzhen` 是 opencode 协调器插件。它不是 review 引擎，真正的 review 语义依赖 `wanxiangshu` 的 `/loop`。

它负责：

- 把需求拆成 DAG
- 生成 slave worktree
- 启动 slave opencode 进程
- 收集 slave 提交
- 做 ff-only 合并
- 把事件写回 coordinator 的 master session 历史

### 3. 为什么这次要大修

因为当前两仓都把“历史事实”拆在很多地方：

- 单块 front matter
- 工具输入/输出
- LLM 输入/输出
- session 历史
- replay / projection
- 宿主 wire format

`BUGFIX.md` 已经把方向改成：`multi-frontmatter` 可以作为等效事实源表达，但原有真相源仍然有效。现在要做的是把这个原则写成人人都能照着改的大修路线。

真正的起点不是“换一种格式”，而是 `compaction` 之后要把被压掉的事件锚点补回来。

压缩会让会话里的原始历史变短，review、with-review、KG、backlog、会话状态机都可能失去恢复上下文。修法不是把压缩结果当新真相，也不是靠内部假投影补洞，而是在 `compaction` 完成后触发一次真实的 `prompt()`，让 LLM 收到一条普通消息，再用 `multi-frontmatter` 把历史上被压掉的锚点重新挂回去。

这里补的是“真实可写入历史的消息”，不是伪造的回放，也不是一次性多次 prompt 的碎片化注入。一个 prompt 里装多块 front matter，才是这次的主轴。

## 这次大修的总原则

### 1. 单 prompt，多 front matter

如果一条事实需要进入历史，优先考虑：

- 一个 `prompt()`
- 一段正文
- 多块 front matter

不要先发多个 prompt 再拼结果。多 prompt 会引入回复打断、顺序漂移、历史撕裂和额外状态机。

更具体地说：`compaction` 结束后应该由一个真实的 `prompt()` 承担“补锚点”职责，把被压缩掉的工具输入、工具输出、LLM 输入输出、review 状态、KG 锚点等重新写成可消费的多块 front matter。

### 2. 事实源不改名，只改编码

工具输出、工具输入、LLM 输入输出、历史、回放、投影、wire format 都还存在。

`multi-frontmatter` 只是把同一批事实换一种能被所有消费者理解的编码。

### 3. 先改解析，再改消费者，再改宿主壳

不要一上来就改最外层 hook。先把 core parser 变成多块 front matter，再让所有只读消费者接受它，最后再收 shell / host codec。

### 4. 先保真，再简化

`wanxiangzhen` 之所以能简化，是因为它可以假设 `wanxiangshu` 已经安装。

这不是“把 fallback 写得更优雅”，而是“把 fallback 删除”。

## `wanxiangshu` 的大修路线

### 第 1 步：先改核心解析器

目标文件：`src/Kernel/PromptFrontMatter.fs`

要做的事：

- 从“只认第一块 front matter”改成“能读多块 front matter”
- 保留旧单块输入的兼容行为
- 让解析结果保持稳定、可穷举、可测试

这里是整个迁移的第一落点。只要这里还只能读单块，后面所有消费者都只能继续打补丁。

### 第 2 步：改只读消费者

这些文件都要改成接受多块 front matter 作为等效事实源：

- `src/Kernel/LoopMessages.fs`
- `src/Kernel/ReviewReplayPolicy.fs`
- `src/Kernel/BacklogProjectionCore.fs`
- `src/Kernel/ToolOutputInfo.fs`
- `src/Kernel/KnowledgeGraph/Job.fs`
- `src/Kernel/KnowledgeGraph/Projection.fs`
- `src/Kernel/ReviewPrompts/*`
- `src/Kernel/CapsFormat.fs`
- `src/Kernel/SearchPrompts.fs`
- `src/Kernel/SubagentPrompts.fs`

这一步的原则很简单：

- 读历史时，不要假设只有一块 front matter
- 读历史时，不要假设只有 tool output 或 LLM output 才是真相
- 读历史时，要把 `multi-frontmatter` 当成能表达同一事实的合法输入

### 第 3 步：改宿主边界与 codec

这些文件负责把外部对象变成内部事实，或者把内部事实重新编码回宿主：

- `src/Opencode/SessionIo.fs`
- `src/Omp/SessionCompacting.fs`
- `src/Omp/HookExecute.fs`
- `src/Omp/SessionLifecycleHooks.fs`
- `src/Omp/ToolResultEvent.fs`
- `src/Shell/OpencodeSessionEventCodec.fs`
- `src/Shell/HostMessagePartCodec.fs`
- `src/Shell/MessageTransformCommon.fs`
- 各宿主消息 codec

这里要特别注意：

- 不要把 `multi-frontmatter` 当成“只在 replay 用”的临时替身
- 不要把 compaction 的局部切片当成唯一事实
- 不要把宿主 wire format 绑死成单块 YAML
- 不要在 compaction 后用内部强写回替代真实 `prompt()`；锚点必须重新进入普通历史

### 第 4 步：补测试

优先看这些测试：

- `tests/LoopMessagesTests.fs`
- `tests/ReviewPromptsFormatTests.fs`
- `tests/ReviewTestsPrompts.fs`
- `tests/WorkBacklogTests.fs`
- `tests/ToolOutputInfoTests.fs`
- `tests/OmpSessionCompactingTests.fs`
- `tests/OpencodeSessionEventCodecTests.fs`
- `tests/OpencodeSessionEventCodecTestsSession.fs`
- `tests/MessagingTests.fs`

要锁住的行为：

- 单块 front matter 还能读
- 多块 front matter 也能读
- replay / projection 结果一致
- compaction 后的语义不丢
- 旧历史不需要重写就能被新解析器理解

### 第 5 步：补文档

要改：

- `wanxiangshu/README.md`
- `wanxiangshu/PRD.md`

这两份文档至少要补三件事：

1. 什么是 `multi-frontmatter`
2. 为什么它和原真相源等效
3. 迁移时哪些文件先改、哪些文件最后改

## `wanxiangzhen` 的大修路线

### 先说安装前提

`wanxiangzhen` 以后要把 `wanxiangshu` 当硬前提。

也就是说：

- 用户安装 `wanxiangzhen` 前，必须先安装 `wanxiangshu`
- `wanxiangzhen` 不再保留“没有万象术也能半残运行”的主路径
- 如果缺依赖，应该尽早失败，而不是在主流程里散落 fallback

这会直接简化 `wanxiangzhen` 的实现。

### 第 1 步：删掉依赖探测的主分叉

重点看这些地方：

- `src/Shell/ConfigReader.fs` 的 `detectVibeFs`
- `src/Shell/CoordinatorOps.fs` 里的 `vibeFs` 分支
- `src/Kernel/SquadPrompts.fs` 里 `if vibeFsDetected then ... else ...`

如果万象术一定存在，这类“有就 A、没有就 B”的逻辑就该收紧。目标不是保留两个世界，而是只保留一个世界。

### 第 2 步：把历史写成第一真相

重点文件：

- `src/Kernel/SquadEvent.fs`
- `src/Shell/EventCodec.fs`
- `src/Shell/CoordinatorRuntime.fs`
- `src/Shell/SessionIo.fs`

这里的原则和 `wanxiangshu` 一样：

- 历史是事实
- 内存 DAG 是投影
- `multi-frontmatter` 是可等效的历史编码

当前 `EventCodec.fs` 只按第一块 front matter 解析，这种写法后续要能读多块；否则后面一旦把多事件压进单条消息，历史回放就会丢事实。

### 第 3 步：把 prompt 注入和事件写回统一成一条路

重点文件：

- `src/Shell/CoordinatorRuntime.fs`
- `src/Shell/CoordinatorOps.fs`
- `src/Shell/SlaveRuntime.fs`

这里要做的不是“写更多 prompt”，而是让注入、提交、回放、清理都服务于同一个历史模型。

建议优先遵守这条原则：

- 一次注入尽量用一条 `prompt()`
- 如果要写多个事实，先考虑多 front matter
- 不要把 `multi-prompt` 当成默认方案

### 第 4 步：把 `wanxiangshu` 作为必需依赖写进文档

要改：

- `wanxiangzhen/PRD.md`
- 新增 `wanxiangzhen/README.md`

`wanxiangzhen/README.md` 现在不存在，建议新建。

建议它至少讲清楚：

- 先装 `wanxiangshu`
- 再装 `wanxiangzhen`
- coordinator / slave 是什么
- master session 历史是 SSOT
- 为什么 review 依赖 `wanxiangshu`

`PRD.md` 则负责写清楚完整规则和文件级实现顺序。

### 第 5 步：补测试

优先看这些测试：

- `tests/EventCodecTests.fs`
- `tests/EventReplayTests.fs`
- `tests/SessionIoTests.fs`
- `tests/CoordinatorOpsTests.fs`
- `tests/ExtendedCoordinatorOpsTests.fs`
- `tests/SlaveRuntimeTests.fs`
- `tests/OpencodePluginE2eTests.fs`
- `tests/MockE2eTests.fs`

要锁住的行为：

- 事件编码 / 解码能吃多块 front matter
- 历史重放不丢事件
- 没有 `wanxiangshu` 时尽早报错
- 有 `wanxiangshu` 时走简化主路径

## 建议的执行顺序

1. 先改 `wanxiangshu/src/Kernel/PromptFrontMatter.fs`
2. 再改 `wanxiangshu` 的只读消费者
3. 再改 `wanxiangshu` 的 Shell / host codec
4. 再补 `wanxiangshu` 的测试
5. 再改 `wanxiangshu/README.md` 和 `wanxiangshu/PRD.md`
6. 再改 `wanxiangzhen` 的历史 / 编解码 / 注入链
7. 删除 `wanxiangzhen` 的 fallback 分支
8. 新建或补全 `wanxiangzhen/README.md`
9. 再改 `wanxiangzhen/PRD.md`
10. 最后补两仓测试

---

## 最容易犯的错

- 把 `multi-frontmatter` 当成新真相，误删原有工具 / LLM 语义
- 只改 parser，不改消费者，导致 replay 和 live 行为不一致
- 只改文档，不改测试，最后没人知道语义有没有锁住
- 在 `wanxiangzhen` 里继续保留“万象术不存在也能跑”的 fallback
- 把多块 front matter 拆成多次 prompt，导致回复打断和历史撕裂

---

## 一句话收尾

这次大修的本质不是“加一个新格式”，而是：

> 让 `multi-frontmatter` 成为所有历史事实的等效编码，同时保留原有真相源；`wanxiangshu` 先完成语义升级，`wanxiangzhen` 再在“万象术必定存在”的前提下收紧实现。
