# `/plan` 算法与流程的保姆式速查

这份速查是给**真正要动手实现的人**看的。

它的目标只有一个：

> 让实现者不需要反复翻 `THATREPO`、`方案.md`、`附录.md`，就能按步骤把 `/plan` 做出来。

它不是概念说明书，也不是设计哲学文档。它是一份**流程清单 + 调用顺序 + 伪代码 + 接线说明**。

---

## 1. 一句话理解这次 `/plan`

用户输入：

```text
/plan <需求>
```

系统做的事：

1. 把需求标准化
2. 先生成少量假设问题
3. 并行跑多个“纯消息分支初稿”
4. 每个分支接受一次 critique（DFS 压力）
5. 每个分支再生成局部 solution pool（BFS）
6. 每个分支基于 draft + critique + pool 做一次显式重组/修订
7. 让第二模型评估修订后的 branches
8. 合成一份最终计划 Markdown
9. 只在最后写一次文件 `PLAN-<hex4>.md`

**分支不执行、不写盘、不改工作区。第一版也不能退化成 best-of-N。**

---

## 2. 你要实现的最终效果

实现完成后，用户应该看到：

- `/plan` 可以跑
- 过程中会有“正在生成分支”“正在评估”“正在生成计划文件”的状态
- 最后落盘一个计划文件
- 文件名形如：`PLAN-7f3a.md`
- 这个文件里有：
  - 需求
  - 不确定性
  - 多分支对比
  - judge 结论
  - 最终计划
  - 验收标准
  - 风险与回退

---

## 3. 总体架构

你这次要做的是三层：

### 3.1 Kernel（纯函数层）

负责：

- 规范化需求
- 定义类型
- 构造 prompt
- 解析分支 JSON
- 解析 judge JSON
- 渲染最终 Markdown
- 生成文件名

### 3.2 Opencode（宿主适配层）

负责：

- 拦截 `/plan`
- 组装上下文
- 调 Kernel
- 调模型
- 调写工具落盘

### 3.3 Mux（宿主适配层）

负责：

- 提供 `/plan` slash command
- 用同一个 Kernel 协议跑分支
- 用同一个 judge 协议评估
- 最后落盘计划文件

---

## 4. 核心概念对应

| 概念 | 这次 `/plan` 里的对应 |
|---|---|
| Strategy-as-lens | 不同分支 lens |
| Hypothesis Packet | 需求侦察结果 |
| Structured Solution Pool | 每个 branch 的局部 BFS 候选池 |
| PQF | 分支保留/淘汰判断 |
| Final Judge | 第二模型评估器 |
| Memory Bank | 上一轮简要摘要（可选） |
| EDFS | 第一版做单次闭环版，未来扩成多轮 |
| Recombination | critique + pool 之后的 branch revision |

## 5. 端到端流程图

```text
用户输入 /plan 需求
        ↓
解析需求文本
        ↓
Kernel: normalizeRequirement
        ↓
Kernel: buildPlanHypotheses
        ↓
Kernel: buildPlanLenses
        ↓
并行生成 5 个初始 branch drafts
        ↓

        ↓

        ↓

        ↓

        ↓

        ↓
Kernel: parsePlanJudgeResponse
        ↓

        ↓
Host: write PLAN-<hex4>.md
        ↓

```

## 7. Kernel 里必须有的类型

下面这些类型，建议直接照着做。

```fsharp
type PlanRequest = {
    requestId: string
    rawRequirement: string
    normalizedRequirement: string
    branchCount: int
    branchModelName: string
    judgeModelName: string
    outputFileName: string
    workspaceRoot: string
    existingContext: string option
}

type PlanLens =
    | DirectDelivery
    | ArchitectureFirst
    | RiskFirst
    | SimplificationFirst
    | CounterexampleFirst
    | CrossDomainFirst
    | ConstraintFirst

type PlanHypothesis = {
    hypothesisId: string
    text: string
    targetBranchIds: string list
}

type PlanBranchCandidate = {
    branchId: string
    lens: PlanLens
    title: string
    candidatePlanMarkdown: string
    candidatePlanSummary: string
    keyAssumptions: string list
    keyRisks: string list
    validationChecks: string list
    selfCritique: string
    confidence: float
}

type PlanJudgeDecision = {
    winnerBranchId: string
    keptBranchIds: string list
    rejectedBranchIds: string list
    judgeReasoning: string
    mergeNotes: string list
}

type PlanRunResult = {
    request: PlanRequest
    hypotheses: PlanHypothesis list
    branches: PlanBranchCandidate list
    decision: PlanJudgeDecision
    finalMarkdown: string
    finalFileName: string
}
```

---

## 8. Kernel 里必须有的函数

### 8.1 需求标准化

```fsharp
normalizeRequirement : string -> string
```

作用：

- 去掉多余空格
- 统一换行
- 保持内容原意

---

### 8.2 文件名格式化

```fsharp
formatPlanFileName : string -> string
```

输入示例：

- `7f3a`

输出：

- `PLAN-7f3a.md`

---

### 8.3 Lens 生成

```fsharp
buildPlanLenses : PlanRequest -> PlanLens list
```

建议默认 5 个：

- `DirectDelivery`
- `ArchitectureFirst`
- `RiskFirst`
- `SimplificationFirst`
- `CrossDomainFirst`

如果需求特别偏工程，也可以替换最后一个为：

- `ConstraintFirst`

---

### 8.4 假设生成

```fsharp
buildPlanHypotheses : PlanRequest -> PlanLens list -> PlanHypothesis list
```

作用：

- 提前找需求里的未知数
- 让分支不用盲跑

建议至少生成三类假设：

1. 需求是否真的清晰
2. 约束是否冲突
3. 有没有隐藏风险

---

### 8.5 分支 prompt 生成

```fsharp
buildPlanBranchPrompt : PlanRequest -> PlanLens -> PlanHypothesis list -> string
```

这一步会给每个分支生成独立 prompt。

每个 prompt 要求分支输出结构化 JSON，至少带：

- 分支标题
- 计划正文
- 风险
- 验证点
- 自我批判
- confidence

---

### 8.6 分支 JSON 解析

```fsharp
parsePlanBranchResponse : string -> Result<PlanBranchCandidate, string>
```

要能解析：

- 纯 JSON
- 带 markdown code fence 的 JSON
- 前后有多余说明的 JSON

---

### 8.7 Judge prompt 生成

```fsharp
buildPlanJudgePrompt : PlanRequest -> PlanBranchCandidate list -> string
```

judge prompt 要强调：

- 只比较候选本身
- 不要被文风骗
- 不要只挑最长的
- 不要只挑最抽象的
- 不要只挑最保守的

---

### 8.8 Judge JSON 解析

```fsharp
parsePlanJudgeResponse : string -> Result<PlanJudgeDecision, string>
```

judge 输出建议包含：

- winnerBranchId
- keptBranchIds
- rejectedBranchIds
- judgeReasoning
- mergeNotes

---

### 8.9 最终 Markdown 渲染

```fsharp
renderPlanMarkdown : PlanRunResult -> string
```

这是最终落盘文本的唯一生成函数。

### 8.10 第一版闭环还必须补的函数

```fsharp
buildPlanCritiquePrompt : PlanRequest -> PlanBranchCandidate -> string
parsePlanCritiqueResponse : string -> Result<PlanBranchCritique, string>

buildPlanPoolPrompt : PlanRequest -> PlanBranchCandidate -> PlanBranchCritique -> string
parsePlanPoolResponse : string -> Result<PlanPoolEntry list, string>

buildPlanRevisionPrompt : PlanRequest -> PlanBranchCandidate -> PlanBranchCritique -> PlanPoolEntry list -> string
parsePlanRevisionResponse : string -> Result<PlanBranchRevision, string>
```

如果没有这组函数，第一版就还是 best-of-N。

## 9. 最终 Markdown 文件结构

推荐固定成下面这些章节：

```markdown
# PLAN-7f3a

## 需求

## 侦察到的不确定性

## 分支总览

## 分支候选对比

## Judge 结论

## 最终计划

## 实施步骤

## 验收标准

## 风险与回退

## 未决问题

## 附录：被淘汰分支摘要
```

### 每章写什么

#### 需求

- 直接放用户原始需求

#### 侦察到的不确定性

- 哪些地方还不明确
- 哪些约束可能冲突
- 哪些前提需要确认

#### 分支总览

- 每个 lens 简介
- 每个分支的短说明
- 每个分支的 confidence

#### 分支候选对比

- 每个候选为什么保留
- 为什么淘汰别的候选
- 各自风险是什么

#### Judge 结论

- winner 是谁
- 为什么它最好
- 为什么没选其他分支

#### 最终计划

- 直接写可执行计划

#### 实施步骤

- 按先后顺序列步骤

#### 验收标准

- 明确能检查的标准

#### 风险与回退

- 如果失败怎么办

#### 未决问题

- 哪些问题还要人工确认

#### 附录：被淘汰分支摘要

- 保留一点 portfolio 透明度
- 但不要写成日志

## 10. 分支输出的 JSON schema

每个分支都应该输出类似下面的结构：

```json
{
  "branchId": "b1",
  "lens": "ArchitectureFirst",
  "title": "先把骨架立住",
  "candidatePlanMarkdown": "# ...",
  "candidatePlanSummary": "...",
  "keyAssumptions": ["..."],
  "keyRisks": ["..."],
  "validationChecks": ["..."],
  "selfCritique": "...",
  "confidence": 0.72
}
```

### 这几个字段为什么必须有

- `candidatePlanMarkdown`：不是想法，而是草案
- `keyAssumptions`：告诉后续实现者最关键的前提
- `keyRisks`：告诉后续实现者哪里会崩
- `validationChecks`：告诉后续实现者怎么验收
- `selfCritique`：保留 build-and-break 的风格
- `confidence`：辅助 judge，但不能单独决定结果

---

## 11. Judge 的 JSON schema

第二模型的输出建议长这样：

```json
{
  "winnerBranchId": "b3",
  "keptBranchIds": ["b3", "b1"],
  "rejectedBranchIds": ["b2", "b4", "b5"],
  "judgeReasoning": "...",
  "mergeNotes": ["...", "..."]
}
```

### judge 需要比较什么

- 谁最贴题
- 谁最能落地
- 谁最少空话
- 谁最能覆盖风险
- 谁最能指导真正实施

### judge 不应该比较什么

- 谁更像“大模型口吻”
- 谁写得更长
- 谁更会包装
- 谁更像最终答案本身

## 12. 轻量版流程伪代码

下面是实现者最需要的骨架。

```text
on /plan <需求>:
    1. normalized = normalizeRequirement(需求)
    2. request = create PlanRequest
    3. lenses = buildPlanLenses(request)
    4. hypotheses = buildPlanHypotheses(request, lenses)
    5. for each lens in lenses:
           prompt = buildPlanBranchPrompt(request, lens, hypotheses)
           call model -> rawBranchJson
           parse -> branchDraft
    6. for each branchDraft:
           critiquePrompt = buildPlanCritiquePrompt(request, branchDraft)
           call model -> rawCritique
           parse -> branchCritique
    7. for each branchDraft + branchCritique:
           poolPrompt = buildPlanPoolPrompt(request, branchDraft, branchCritique)
           call model -> rawPool
           parse -> branchPool
    8. for each branchDraft + branchCritique + branchPool:
           revisionPrompt = buildPlanRevisionPrompt(...)
           call model -> rawRevision
           parse -> branchRevision
    9. judgePrompt = buildPlanJudgePrompt(request, branchRevisions)
   10. call second model -> rawJudgeJson
   11. parse -> decision
   12. finalMarkdown = renderPlanMarkdown(result)
   13. write PLAN-<hex4>.md
   14. return success message
```

## 13. Opencode 接线顺序

### 13.1 入口

在 Opencode 里，`/plan` 应该走 slash command 或 command hook。

### 13.2 宿主职责

Opencode 侧只做这些：

- 取输入文本
- 调 Kernel 生成 plan protocol
- 调模型生成 branch drafts
- 调模型生成 critiques
- 调模型生成局部 pools
- 调模型生成 revisions
- 调第二模型 judge
- 走 write 工具落盘

### 13.3 不要做的事

- 不要让分支直接调用写工具
- 不要让分支真实执行命令
- 不要让分支直接写文件夹里的代码

### 13.4 最终物化

最后只允许一次写盘：

```text
PLAN-<hex4>.md
```

### 14.1 入口

### 14.2 复用 Kernel

- markdown renderer

### 14.3 最终写盘

Mux 最后也只做一次写盘，不做分支真实执行。

## 15. 最重要的实现策略

### 15.1 先把分支做成消息分支

每个分支就是一次模型调用，输入同一个需求，但 lens 不同。

### 15.2 第一版必须包含一次 branch-local DFS

每个 branch 都要：

- 先写 draft
- 再接受 critique
- 再做 revision

### 15.3 第一版必须包含一次 branch-local BFS

每个 branch 都要生成 2~3 个局部 pool entries。

### 15.4 第一版必须包含一次显式重组

revision 不能只是“根据 critique 重写”，而要显式吸收 pool 中的候选片段或替代思路。

### 15.5 再把 judge 做成二次模型调用

judge 不要用本地规则硬选。

让第二模型看结构化候选，输出结构化评估。

### 15.6 最后一次性落盘

这点不要拆。

只有最终 plan 才能写文件。

## 17. 建议的验收标准

如果实现完了，至少要满足：

1. 输入 `/plan <需求>` 后，会生成多个结构化 drafts。
2. 每个 draft 都会经历一次 critique。
3. 每个 branch 都会生成局部 BFS pool。
4. 每个 branch 都会经历一次显式重组/修订。
5. judge 产出 winner 和合并建议。
6. 最终会生成一个 `PLAN-<hex4>.md`。
7. 过程中没有分支真实写盘。
8. 过程中没有分支真实执行。
9. Opencode 和 Mux 都能走通。

## 19. 一句提醒

这次一定要记住：

> `/plan` 不是“多代理写作”，也不是“best-of-N”；它是“多分支认知搜索 + branch-local DFS/BFS + 显式重组 + 第二模型评估 + 单次文件物化”。

如果实现过程中开始出现“每个分支都在改实际文件”，那就已经偏离这次设计了。
