# REF63: 上下文窗口管理策略

## 1. 问题

多轮迭代导致历史消息不断累积，超出模型上下文窗口。

## 2. Contextual 的压缩策略

### 消息历史阈值
每 10 轮触发一次压缩：
1. 调用记忆代理生成压缩摘要
2. 替换各代理的消息历史：
   - 保留：初始需求、记忆摘要、最新上下文
   - 丢弃：旧轮次的完整对话

### 压缩后的消息结构
```
mainGeneratorMessages = [
    初始需求,
    记忆摘要("有效/无效的方法"),
    最新上下文: "",
    最新一轮的主生成 + 批判消息,
    最新的综合批判
]
```

## 3. Deepthink EDFS 的上下文管理

### 策略层面
| 技术 | 说明 |
|------|------|
| 精选上下文 | 每个代理只收到其角色需要的最小上下文 |
| 历史窗口 | 修正器最多看 5 条历史 |
| 池历史窗口 | 方案池最多看 5 条池历史 |
| 跨策略快照 | 只看其他策略的最新修正/批判 |
| 记忆银行 | 5 条以上的历史压缩为记忆 |

### 上下文仓库结构
```
其他策略（轻量）：
  Strategy 2: 策略文本 + 最新修正 + 最新批判
  Strategy 3: 策略文本 + 最新修正 + 最新批判

当前策略（完整）：
  Strategy 1: 策略文本 + 记忆银行 + 最新修正/批判
            + 最近 5 条历史
            + 最新方案池
```

## 4. 最坏情况估计

```typescript
// 修正器上下文（第 10 轮）：
// 每个修正+批判 ≈ 20k tokens
// 4 个其他策略 × 20k = 80k
// 当前策略 5 条历史 = 100k
// 方案池 ≈ 15k
// 总计 ≈ 195k tokens
```

超过 256k 窗口时，历史被记忆银行替换（回到 ~155k）。
