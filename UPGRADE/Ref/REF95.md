# REF95: 关键设计决策记录

## 1. 为什么不直接复制 THATREPO 的执行层

核心原因：分支并发写盘会导致状态污染。
- 分支 A 的写入会改变分支 B 看到的世界
- 批判的对象的工件可能已被其他分支修改
- 记忆不可重放（取决于执行顺序）
- PQF 可能评估的是混写后的结果

## 2. 为什么保持内存化

- 分支图的自然载体是内存状态机，不是真目录
- 共享写盘天然破坏分支独立性
- 正确的方向是"内存工件"，不是"并发改盘"

## 3. 为什么只保留 `/plan` 一个命令

- 用户不应学习 `/plan-strategy`, `/plan-judge`, `/plan-pqf`...
- 搜索协议应隐藏在系统内部（类似编译器内部 pass）
- 单命令更容易保证状态一致性和执行顺序

## 4. 为什么第一版不能退化成 best-of-N

THATREPO 的精髓不是"多候选选一"，而是：
- DFS 式的 branch-local 深挖（draft → critique → revision）
- BFS 式的局部候选扩张（solution pool）
- 显式重组/移植/交叉吸收（recombination）

## 5. 为什么批判和修正必须分离

- 批判负责诊断（只找问题）
- 修正负责修复（只改方案）
- 分开可以避免代理被自己的输出偏差影响

## 6. 为什么假设测试者应该隔离

- 测试者只看到 Core Challenge + 一条假设
- 看不到策略、分支历史或其他假设
- 防止测试结果被上下文污染
- 确保测试结果的客观性
