# REF20: Deepthink 模式——修正代理 (Correction/Self-Improvement Agent)

## 1. 角色定位

修正代理（又称解决方案修正精炼代理）是分支迭代中产出工作产品的权威代理。它接收现有工作产品、诊断压力、积累的分支情报和分配的策略，产出下一个完整的修正工件。

## 2. 核心原则

- **修正不等于对前一个答案的忠诚**: 前一个工作是可编辑的材料
- **隔离协议**: 最终工件必须自包含——删除所有隐藏上下文后依然完整可读
- **忠实与合规契约**: 开头的修改承诺必须与最终工件完全一致
- **反锚定**: 主动抵抗与前一个答案保持一致的倾向

## 3. 输出结构

```
Revision approach: (2-3段)
  第1段: 承认关键批判、说明修复深度、确定是否改变结论
  第2-3段: 说明从假设包/解决方案池/其他策略吸收了什么
  （可选）
Corrected artifact: (完整工件)
  完整的、自包含的最终工件
```

## 4. 修复深度决策

| 修复深度 | 适用场景 |
|----------|----------|
| 局部修复 | 缺陷是孤立的（代数错误、单一边界情况） |
| 组件重建 | 多个失败由同一子系统/推理块引起 |
| 完全重建 | 批判攻击根本假设、目标、表示或最终结论 |

## 5. 解决方案池参与

- 必须检查每个池候选（但可选择不采用）
- 在修正承诺中说明采用了哪个池想法及其原因
- 不要无差别粘贴——集成必须无缝

## 6. 验证协议

提交前内部审计：
- 每个承诺都有实现
- 最终结论正确
- 已知反例和边界情况已处理
- 全局一致性（代码接口一致、证明依赖有效）
