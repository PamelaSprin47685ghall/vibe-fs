# 行为退化总览和未来如何补全

## 1. 这份文档是干什么的

前面三份文档分别解决的是：

- `方案.md`：怎么实现第一版 `/plan`
- `附录.md`：`THATREPO` 的概念背景是什么
- `算法与流程的保姆式速查.md`：实现时按什么步骤照着写

但还缺一份很关键的东西：

> **第一版 `/plan` 相比 `THATREPO` 到底退化了哪些行为？将来如果仍然只保留 `/plan` 这一个命令，又如何把所有 `THATREPO` 技术逐步补全回来？**

这份文档就是专门回答这个问题的。

它的核心立场非常明确：

1. 第一版 `/plan` 不是简单 best-of-N，而是**轻量版、纯消息、内存化、单次闭环的 DFS/BFS/重组引擎**。
2. 这只是为了降低风险和缩短第一阶段实现路径。
3. **长期目标不是停留在轻量版，而是在“只保留 `/plan` 一个命令”的前提下，把 `THATREPO` 的技术几乎全部内化到 `/plan` 流程里。**
4. 这个未来版本仍然应该是**内存化系统**，而不是多分支共享真实文件夹写盘系统。

也就是说：

> `/plan` 未来不是一个“生成计划文件的小功能”，而应该成为本项目内部的**单入口多阶段认知搜索引擎**。

---

## 2. 为什么必须写“行为退化总览”

如果没有这份文档，外部实现者和后续维护者很容易犯两个错误：

### 错误一：把第一版当成最终版

也就是把现在的轻量版理解成：

- 多想几个方案
- 让第二模型选一个
- 写个 Markdown 完事

这样会导致整个设计停留在“多候选 brainstorming 工具”层面，而无法发展成真正接近 `THATREPO` 的认知搜索协议。

### 错误二：未来补全时方向跑偏

最常见的跑偏方式是：

- 一上来做多命令体系
- 一上来让分支真实写盘
- 一上来把 branch lifecycle 散到很多 slash command 里
- 一上来加一堆宿主层状态，而不是先补 Kernel 协议

这都违背了当前确定的路线：

- **只保留 `/plan` 一个命令**
- **所有核心能力都内嵌进 `/plan` 的内部阶段**
- **全程保持内存化搜索**
- **最终只在结束时物化**

所以，必须把“现阶段退化了什么”写清楚，同时把“未来如何补全”写成一条干净路线。

## 3. 当前第一版 `/plan` 的行为定义

为了明确退化点，先把第一版 `/plan` 的行为说清楚。

第一版 `/plan` 做的是：

1. 接收一个需求文本
2. 标准化需求
3. 生成少量高价值不确定性假设
4. 生成若干个分支镜头（lenses）
5. 让多个分支各自返回结构化初始计划候选（draft）
6. 对每个 draft 做一次 critique
7. 对每个 branch 做一次局部 solution pool 扩张（BFS）
8. 让每个 branch 基于 critique + pool 做一次 revision / recombination
9. 让第二模型对 revisions 做 judge
10. 生成一个最终计划 Markdown
11. 最后一次性落盘 `PLAN-<hex4>.md`

它**不做**：

- 多轮 correction / critique loop
- 分支版本化演化
- memory bank 递归压缩
- solution pool 的历史滚动
- hypothesis heartbeat 的多轮刷新
- PQF 驱动的 branch replacement
- final judge 的长期搜索收敛
- branch-local artifact snapshots
- 影子文件树

所以，第一版的本质是：

> **“单次闭环的内存化 DFS/BFS/重组搜索 + judge 收敛 + 单次物化”**

它不是：

> **“完整多轮 EDFS 搜索系统”**

## 4. 当前相对 `THATREPO` 的退化项总览

下面按概念逐项列退化。

---

### 4.1 Strategy-as-lens：已保留，但还没有完整生命周期

#### `THATREPO` 的原始能力

在 `THATREPO` 中，strategy 不只是多分支标题，而是：

- 贯穿整个 branch 生命周期的搜索框架
- 影响 execution、critique、solution pool、memory、PQF
- 是 branch identity 的核心

#### 第一版 `/plan` 的退化

第一版里，lens 已经影响：

- 初始 draft
- critique 语境
- 本地 pool 的扩张方向
- revision 的重写方式

它还没有做到：

- 跨多轮保留 branch identity
- 在后续 correction / critique / hypothesis refresh 中持续施加约束

#### 未来补全方向

未来 `/plan` 内部要把 lens 升级成真正的：

- `PlanBranchIdentity`
- `branchVersion`
- `branchMemory`
- `branchLifecycle`

即使对外仍然只有 `/plan` 一个命令，内部也要把每个分支当成有持续身份的对象，而不是一次性消息。

---

### 4.2 Hypothesis Packet：已保留，但还没有完整 heartbeat 机制

#### `THATREPO` 的原始能力

在 `THATREPO` 中，hypothesis 不只是前置问题，而是：

- 可被独立测试
- 会形成 packet
- packet 会 selective injection 到不同 branch
- 每隔两轮还会 heartbeat 刷新

#### 第一版 `/plan` 的退化

现在至少应做：

- 初始假设侦察
- 假设结果对初始 draft 的上下文影响

没有做：

- 假设的独立测试回路
- selective injection 的动态细化
- hypothesis heartbeat
- 假设的历史演化与替换

#### 未来补全方向

未来 `/plan` 内部要把 hypothesis 变成真正的阶段系统：

1. 先生成 hypothesis
2. 再对 hypothesis 进行独立 testing
3. testing 结果形成 packet
4. packet selective 注入不同 branch
5. 每隔固定步数根据 branch critique/history 重新生成 hypothesis

也就是说，将来 `/plan` 里应该存在：

- `PlanHypothesis`
- `PlanHypothesisTest`
- `PlanHypothesisPacket`
- `PlanHypothesisRound`
- `PlanHypothesisHeartbeat`

这些都保存在内存态里，不需要落盘成中间文件。

---

### 4.3 EDFS：当前只保留了“单次闭环版”

#### `THATREPO` 的原始能力

在 `THATREPO` 中，`Evolving Depth First Search` 的关键不是“深搜”三个字，而是：

- branch 会经历多轮 correction / critique
- branch 会保留历史
- branch 可以被 PQF 判死并替换
- branch 有 version
- 整个系统是持续演化的

#### 第一版 `/plan` 的退化

当前版本不应是完全没有 EDFS，而应至少有：

- 初始 draft
- critique
- revision

也就是说，当前应该是**单次闭环版 EDFS**，而不是多轮 EDFS。

#### 未来补全方向

未来仍然只保留 `/plan` 一个命令，但内部要把 `/plan` 变成一个**多阶段 loop 引擎**：

```text
/plan
  -> init strategies
  -> init hypotheses
  -> branch proposals
  -> critique
  -> correction
  -> pool expansion
  -> PQF
  -> heartbeat hypotheses
  -> loop
  -> final judge
  -> materialize PLAN-<hex4>.md
```

注意：

- 用户仍然只输入一次 `/plan`
- 中间所有阶段都在内存里发生
- 外部看起来仍然是“一个命令”

也就是说，将来不是新增 `/critique-plan`、`/judge-plan`、`/branch-plan` 之类命令，而是把这些步骤全部内嵌进 `/plan` 里。

---

### 4.4 Structured Solution Pool：当前应有局部池，但没有滚动池历史

#### `THATREPO` 的原始能力

solution pool 在 `THATREPO` 里是滚动的：

- 每轮都生成新的 pool
- pool 与 critique/history/memory 强耦合
- pool 本身也会成为后续 correction 的输入

#### 第一版 `/plan` 的退化

当前版本不应只是“全局候选集合”，而应至少包含：

- 每个 branch 的局部 BFS pool
- 局部 pool 参与该 branch 的 revision

没有历史，没有递进，没有“围绕当前 branch 状态持续扩张 BFS 邻域”的能力。

#### 未来补全方向

未来 `/plan` 内部应当维护：

- `currentBranchPool`
- `branchPoolHistory`
- `crossBranchPoolSummary`

每轮 correction 完成后，都重新生成 pool。

但是这些 pool 仍然应该：

- 只存在于内存中
- 只在最终 `PLAN-<hex4>.md` 中作为总结性内容出现

绝不能变成“每轮生成一个真实文件夹快照”。

---

### 4.5 Recursive Memory Bank：当前只有摘要，没有真正递归压缩

#### `THATREPO` 的原始能力

Memory Bank 在 `THATREPO` 里会：

- 接收前一轮 memory
- 接收最新的若干 history window
- 生成新的统一 memory

这是一种真正的递归压缩。

#### 第一版 `/plan` 的退化

当前版本最多只是：

- 记录本轮 assumptions / risks / rejected reasons

还没有真正形成：

- “探索史压缩器”

#### 未来补全方向

未来 `/plan` 应该在内存里维护：

- `PlanMemorySnapshot`
- `PlanMemoryBank`
- `PlanMemoryUpdate`

它记的内容不应是“答案”，而应是：

- 哪些策略重复失败
- 哪些假设已被证伪
- 哪些结构化风险在反复出现
- 哪些候选总是空泛但写得好看
- 哪些 lens 对某类任务尤其有效

这样 `/plan` 在未来可以逐渐从“一次性规划器”进化成“带长期经验的规划器”。

---

### 4.6 PQF：当前只保留了 judge 内的轻量筛选

#### `THATREPO` 的原始能力

PQF 不是选 winner，而是决定：

- 哪条 branch 还该不该继续活

#### 第一版 `/plan` 的退化

现在 judge 会做一点：

- 保留哪些分支
- 淘汰哪些分支

但它还是一次性的，不是 branch lifecycle 级的。

#### 未来补全方向

未来 `/plan` 内部的 judge 要拆成两个层次：

1. **PQF Layer**
   - 哪些 branch 应该继续
   - 哪些 branch 应该替换
   - 哪些 branch 只是重复镜像

2. **Final Judge Layer**
   - 最后哪条 branch 的最终计划最优

也就是说，将来 judge 不再只是“从 5 个里选 1 个”，而是：

- 在多轮内部迭代中持续做 branch governance

---

### 4.7 Final Judge Isolation：当前基本保留，但未来要更严格

#### 当前状态

第一版里，judge 只看结构化候选，这基本符合 `Final Judge Isolation`。

#### 未来补全方向

未来要更严格区分：

- `branch working state`
- `judge visible state`

也就是 judge 不该直接看到：

- 所有中间噪声
- 所有分支内部 scratch notes
- 所有历史细枝末节

它只该看到：

- 经过 Kernel 筛整后的 candidate snapshot

因此未来 `/plan` 内部要补一个中间层：

- `PlanJudgeView`

专门负责把庞杂的 branch state 压缩成 judge 可看的视图。

---

### 4.8 Build-and-Break：当前应由 critique + pool + revision 共同承担

#### 当前状态

现在每个候选会带：

- `selfCritique`
- `keyRisks`
- `validationChecks`

这保留了一点 build-and-break 味道。

#### 退化点

还没有做到：

- 专门的 adversarial branch
- 反向分支与构造分支的组合策略
- 反例候选与正向候选的成对出现

#### 未来补全方向

未来 `/plan` 的 branch set 应该显式区分：

- constructive branches
- adversarial branches
- synthesis branches
- replacement branches

这样 build-and-break 才不是“每个分支顺手自我批评一句”，而是系统层面的组合设计。

## 5. 未来版本要补齐的全部 `THATREPO` 技术

下面直接给出未来 `/plan` 的完整目标图景。

注意：

- **仍然只保留 `/plan` 一个命令**
- **仍然保持内存化**
- **仍然避免多分支真实共享写盘**

### 5.1 完整未来版 `/plan` 的阶段图

```text
/plan <需求>
    ↓
Phase 0: Request normalization
    ↓
Phase 1: Strategy generation
    ↓
Phase 2: Hypothesis generation
    ↓
Phase 3: Hypothesis testing
    ↓
Phase 4: Selective injection
    ↓
Phase 5: Initial branch proposals
    ↓
Phase 6: Critique pass
    ↓
Phase 7: Correction pass
    ↓
Phase 8: Structured solution pool expansion
    ↓
Phase 9: Branch memory update
    ↓
Phase 10: PQF branch governance
    ↓
Phase 11: Hypothesis heartbeat refresh
    ↓
Phase 12: 下一轮 branch iteration
    ↓
Phase 13: Final judge
    ↓
Phase 14: Single materialization to PLAN-<hex4>.md
```

这就是目标形态。

它本质上是：

- 一个命令
- 内部多阶段
- 多轮搜索
- 最后收敛

## 6. 为什么未来也必须保持“只一个命令”

这是重要约束，不是 UI 偏好。

### 6.1 用户视角应该极简

用户不应该学习一整套：

- `/plan-strategy`
- `/plan-judge`
- `/plan-pool`
- `/plan-memory`
- `/plan-pqf`

这种多命令体系会暴露系统内部结构，增加认知负担。

### 6.2 搜索协议本来就应该隐藏在系统内部

就像编译器不会要求用户手动执行寄存器分配、SSA、优化 pass 一样。

`/plan` 应该是：

- 一个顶层入口
- 内部自动调度完整协议

### 6.3 单命令更容易做状态完整性

如果分散成多个命令，会很快出现：

- 中间状态丢失
- 命令顺序错误
- 用户跳过某步
- branch history 不一致

把所有行为都封装进 `/plan`，可以保证：

- 状态由系统管理
- 顺序由系统控制
- 版本由系统控制

---

## 7. 为什么未来仍然必须保持“内存化”

### 7.1 因为 branch graph 不是文件系统

branch graph 是：

- 认知状态
- 候选工件状态
- critique 历史
- hypothesis 轮次
- PQF 决策
- memory 压缩结果

这些东西的自然载体是：

- 内存态 state machine

而不是：

- 一堆真实目录

### 7.2 共享写盘天然破坏分支独立性

一旦未来补全过程里让分支共享写盘，马上会坏：

- branch identity
- critique objectivity
- memory replayability
- judge isolation

所以无论将来功能补到多深，都不要把 branch runtime 建立在共享真实目录上。

### 7.3 正确的未来方向是“内存工件”，不是“并发改盘”

未来如果一定要让 branch 更接近真实执行，也应该是：

- branch 产出 in-memory artifact
- branch 产出 virtual patch
- branch 产出 shadow snapshot

而不是：

- branch 直接去改仓库根目录

## 8. 未来补全过程的正确路线

下面给一个建议的四阶段路线。

---

### 阶段 A：当前轻量版（但不是 best-of-N）

#### 目标

- 先验证 strategy / hypothesis / branch candidate / judge 有没有价值

#### 能力

- 单次闭环版 DFS：draft -> critique -> revision
- 每个 branch 的局部 BFS pool expansion
- 一次显式 recombination / correction
- 第二模型评估 revisions
- 最终 Markdown 落盘

#### 特征

- 无 branch history 滚动
- 无 memory recursion
- 无 PQF lifecycle
- 无 heartbeat

这是现在的第一版。

---

### 阶段 B：多轮内存 branch proposals

#### 目标

让 `/plan` 不再是单轮，而是多轮内存化 branch refinement。

#### 要补的能力

- `PlanBranchState`
- `PlanBranchIteration`
- `PlanCritique`
- `PlanCorrection`
- `PlanBranchHistory`

#### 行为变化

分支不再只跑一次，而是：

- proposal
- critique
- correction
- proposal again

#### 仍然保持

- 不写盘
- 不真实执行
- 不共享工作区

---

### 阶段 C：完整 hypothesis + pool + PQF

#### 目标

把 `THATREPO` 的结构性技术基本补齐。

#### 要补的能力

- hypothesis testing stage
- selective injection stage
- structured solution pool history
- recursive memory bank
- PQF keep/update branch governance
- judge view isolation

#### 这时 `/plan` 内部就已经很像 `THATREPO`

只是仍然是：

- memory-native
- message-native
- single materialization at the end

---

### 阶段 D：内存工件化

#### 目标

让分支不仅返回文本候选，还返回更精细的工件对象。

#### 可补的能力

- `PlanArtifact`
- `PlanPatchCandidate`
- `PlanSectionReplacement`
- `PlanValidationArtifact`
- `PlanJudgeView`

#### 这一步最接近 “全部 `THATREPO` 技术”

但依然不应该：

- 改共享真实目录

它应该做到：

- 分支有自己的内存工件世界
- 最后由一个 materializer 选中并写一个文件

---

## 9. 未来版 `/plan` 的内存状态应该长什么样

这部分不是现在就要实现，但未来一定要往这里靠。

建议最终形成一个大的内存态：

```fsharp
type PlanSessionState = {
    request: PlanRequest
    strategyRounds: PlanStrategyRound list
    hypothesisRounds: PlanHypothesisRound list
    branches: PlanBranchRuntime list
    branchPools: PlanPoolRound list
    memoryBank: PlanMemoryBank option
    pqfRounds: PlanPqfRound list
    judgeRounds: PlanJudgeRound list
    finalDecision: PlanJudgeDecision option
    finalMarkdown: string option
}
```

### 关键理解

这整个 `PlanSessionState` 都应该：

- 存在于内存
- 可序列化
- 可导出到最终计划中

但不需要每轮都写磁盘。

## 10. 未来补全时最容易犯的错误

### 错误一：把补全理解成“加更多分支”

补全不是让分支数从 5 变 10。

真正的补全是：

- 增加 branch lifecycle
- 增加 hypothesis testing
- 增加 pool rolling
- 增加 memory recursion
- 增加 PQF

### 错误二：把补全理解成“让分支去执行”

这是最危险的误解。

未来补全的重点不是：

- branch can execute

而是：

- branch has richer internal state and richer artifacts

### 错误三：把 `/plan` 做成多个子命令系统

不允许演化成：

- `/plan-next`
- `/plan-continue`
- `/plan-pqf`
- `/plan-judge`

这些都应该是 `/plan` 内部阶段。

### 错误四：把 memory 写成总结答案

memory 记的是：

- 探索史
- 失败史
- 不变量
- 风险模式

不是“当前最佳答案摘要”。

### 错误五：把 judge 变成文风比赛

judge 以后越强，越要看：

- 结构化状态
- 候选工件
- 风险与验证

而不是看：

- 哪段 prose 更像大模型写的咨询报告

---

## 11. 最终理想形态

未来真正完整的 `/plan`，应该达到这个状态：

- 用户仍然只输入一次 `/plan <需求>`
- 系统内部自动完成：
  - strategy generation
  - hypothesis generation
  - hypothesis testing
  - selective injection
  - initial branch proposal
  - critique
  - correction
  - solution pool BFS expansion
  - memory bank recursive update
  - PQF 分支治理
  - heartbeat hypothesis refresh
  - final judge isolation
- 全程 branch state 都在内存中
- 只有最后物化 `PLAN-<hex4>.md`

如果能做到这点，那么虽然对外看起来只是一个 `/plan` 命令，内部实际上已经拥有：

> `THATREPO` 的几乎全部认知搜索技术。

而且比直接照搬执行层更适合本项目。

---

## 12. 一句结论

第一版 `/plan` 只是一个退化但安全的入口；未来真正要做的，不是增加更多命令，也不是让分支去改真实文件夹，而是把 `THATREPO` 的全部搜索协议**压缩进这一个命令的内存化阶段系统里**。
