# `/plan` 轻量版多分支引擎实现方案

## 1. 目标

这个方案的目标不是把 `THATREPO` 里的所有执行层能力直接搬过来，而是尽可能原汁原味地移植它的**概念结构**，同时避开你已经指出的最大堵点：

- 不让多个分支直接写同一个文件夹。
- 不让分支真实执行代码或真实改盘。
- 不让分支之间互相污染状态。
- 不让系统一开始就陷入复杂的工作树、并发写盘、回滚和冲突合并。

所以这次实现的 `/plan <需求>` 不是“执行器”，而是一个**纯消息分支 + 第二模型评估 + 最终单次落盘**的轻量版规划引擎。

它应该保留 `THATREPO` 的这些核心概念：

- `Strategy-as-lens`
- `Build-and-break` 的候选组合
- `Hypothesis Packet` 的前置侦察
- `Structured candidate pool`
- `Recursive memory` 的压缩思想（但这里先做轻量版，可只保留一轮历史摘要）
- `PQF` 式的分支保留/淘汰判断
- `Final Judge` 的独立评估

但它要去掉这些高风险部分：

- 分支真实落盘
- 分支真实执行工具
- 分支共享工作区写入
- 分支之间共享同一份可变文件状态

最终产物只有一个：`PLAN-<hex4>.md`。

---

## 2. 设计原则

### 2.1 纯内核优先

所有可复用逻辑先放到 `Kernel`。

`Kernel` 只负责：

- 定义类型
- 组装 prompt
- 解析分支返回
- 做 judge 合并
- 渲染最终 markdown
- 生成文件名

`Kernel` 不负责：

- 调用模型
- 访问文件系统
- 发 slash command
- 落盘
- 选择宿主平台行为

### 2.2 分支只回消息，不做动作

每个分支只返回结构化消息。

分支不能：

- 写文件
- 跑 shell
- 改工作区
- 调工具
- 做真实实现

分支能：

- 提出不同策略镜头
- 产出不同计划候选
- 自我批判
- 标记风险
- 标记需要外部验证的点

### 2.3 第二模型负责评估，不负责执行

第二模型只做 judge。

它要比较的是：

- 哪个分支最贴题
- 哪个分支覆盖面最好
- 哪个分支最可落地
- 哪个分支风险最低
- 哪个分支最有实施价值

它不能：

- 直接改代码
- 直接跑工具
- 直接写盘

### 2.4 最终只允许单次物化

只有最后被选中的计划，才允许调用写工具，把内容写成 `PLAN-<hex4>.md`。

这一步必须是唯一的副作用出口。

## 3. 这次要移植的概念，不是执行层

这里把 `THATREPO` 的概念拆成可以移植到 THISREPO 的版本。

### 3.1 Strategy-as-lens

策略不是步骤清单，而是观察问题的镜头。

在 `/plan` 里，每个分支都应该带着一个不同的镜头，例如：

- 直接交付镜头
- 架构重构镜头
- 风险优先镜头
- 约束优先镜头
- 简化优先镜头
- 反证优先镜头
- 交叉域迁移镜头

### 3.2 Build-and-break

不是只找能做的方案，也要找能打破方案的条件。

所以每个分支候选都要同时包含：

- 能成立的理由
- 会失败的地方
- 需要验证的点
- 如果失败了应该怎么改方向

### 3.3 Hypothesis Packet

不做真实执行时，也要保留假设侦察。

这意味着在 `/plan` 里，要先问清楚：

- 需求里最不确定的点是什么
- 哪些约束最可能冲突
- 哪些前提可能根本不成立
- 哪些边界条件最可能推翻某个方案

### 3.4 Structured Solution Pool

在这里它不是“完整解决方案池”，而是“计划候选池”。

每个候选都应该是可比较、可批判、可合并的计划草案。

### 3.5 Recursive Memory

轻量版里不做完整递归历史，但应该保留：

- 本次分支摘要
- judge 选择理由
- 被淘汰方案的关键原因

这样下一次 `/plan` 能形成最简 memory。

### 3.6 PQF

`PQF` 在这里退化成：

- 这条分支是否继续保留
- 这条分支是否只是漂亮但不可落地
- 这条分支是否重复了其他分支
- 这条分支是否命中真正问题

### 3.7 Final Judge Isolation

judge 只能看到候选，不看到实现细节、执行过程和隐藏状态。

这可以避免“谁写得长谁赢”的偏差。

## 5. `PlanEngine.fs` 需要定义的类型

下面是建议的核心类型。不是最终代码，只是实现者要照着做的结构。

```fsharp
type PlanRequest = {
    requestId: string
    rawRequirement: string
    normalizedRequirement: string
    branchCount: int
    branchModelName: string
    judgeModelName: string
    outputFileName: string
    workspaceRoot: string
    existingContext: string option
}

type PlanLens =
    | DirectDelivery
    | ArchitectureFirst
    | RiskFirst
    | SimplificationFirst
    | CounterexampleFirst
    | CrossDomainFirst
    | ConstraintFirst

type PlanHypothesis = {
    hypothesisId: string
    text: string
    targetBranchIds: string list
}

type PlanBranchCandidate = {
    branchId: string
    lens: PlanLens
    title: string
    candidatePlanMarkdown: string
    candidatePlanSummary: string
    keyAssumptions: string list
    keyRisks: string list
    validationChecks: string list
    selfCritique: string
    confidence: float
}

type PlanJudgeDecision = {
    winnerBranchId: string
    keptBranchIds: string list
    rejectedBranchIds: string list
    judgeReasoning: string
    mergeNotes: string list
}

type PlanRunResult = {
    request: PlanRequest
    hypotheses: PlanHypothesis list
    branches: PlanBranchCandidate list
    decision: PlanJudgeDecision
    finalMarkdown: string
    finalFileName: string
}
```

### 为什么这些字段够用

- `PlanLens` 保证分支不同，不只是改措辞。
- `candidatePlanMarkdown` 让分支真的产出可读草案。
- `keyAssumptions` / `keyRisks` / `validationChecks` 让它有 build-and-break 味道。
- `selfCritique` 让每个分支都能自己先打自己一遍。
- `confidence` 给 judge 一个排序维度，但不允许它独立决定胜负。
- `PlanJudgeDecision` 让第二模型做“评估 + 合并”而不是只报一个名字。

## 6. Kernel 需要提供的纯函数

### 6.1 需求标准化

```fsharp
normalizeRequirement : string -> string
```

用途：

- 去掉多余空白
- 统一换行
- 生成适合 prompt 的稳定输入

### 6.2 计划编号

```fsharp
formatPlanFileName : string -> string
```

输入是 4 位 hex token，例如 `7f3a`。

输出：

- `PLAN-7f3a.md`

### 6.3 分支镜头生成

```fsharp
buildPlanLenses : PlanRequest -> PlanLens list
```

建议默认返回 5 个镜头：

- `DirectDelivery`
- `ArchitectureFirst`
- `RiskFirst`
- `SimplificationFirst`
- `CrossDomainFirst`

如果需求很复杂，可以把第 5 个改成 `ConstraintFirst` 或 `CounterexampleFirst`。

### 6.4 分支 prompt 组装

```fsharp
buildPlanBranchPrompt : PlanRequest -> PlanLens -> PlanHypothesis list -> string
```

分支 prompt 要明确要求：

- 输出结构化 JSON
- 给出 plan draft
- 给出自我批判
- 给出风险与验证点
- 不能执行工具
- 不能写最终代码

### 6.5 假设 packet 组装

```fsharp
buildPlanHypotheses : PlanRequest -> PlanLens list -> PlanHypothesis list
```

用途：

- 先侦察需求里最重要的不确定性
- 给每个分支共享，但不完全相同的假设关注点

建议假设分三类：

- 需求真伪类
- 约束冲突类
- 落地风险类

### 6.6 分支候选解析

```fsharp
parsePlanBranchResponse : string -> Result<PlanBranchCandidate, string>
```

解析每个分支返回的 JSON。

要求实现者容忍：

- 外层包裹代码块
- 多余前后缀
- 字段顺序变化
- `confidence` 缺失时给默认值

### 6.7 judge prompt 组装

```fsharp
buildPlanJudgePrompt : PlanRequest -> PlanBranchCandidate list -> string
```

judge prompt 要强调：

- 只比较候选，不重做整题
- 只按候选内容本身判断
- 不要按文风评分
- 不要偏爱最长文本
- 不要偏爱最抽象文本

### 6.8 judge 解析

```fsharp
parsePlanJudgeResponse : string -> Result<PlanJudgeDecision, string>
```

建议 judge 输出至少包含：

- `winnerBranchId`
- `keptBranchIds`
- `rejectedBranchIds`
- `judgeReasoning`
- `mergeNotes`

### 6.9 最终 markdown 生成

```fsharp
renderPlanMarkdown : PlanRunResult -> string
```

这是最后落盘的唯一正文。

### 6.10 最终文件名生成

```fsharp
generatePlanHex4 : unit -> string
```

这不是 Kernel 纯函数的话也可以由 host 给，Kernel 只负责格式化。

更纯的做法是：

```fsharp
formatPlanFileName : string -> string
```

host 负责生成 hex4。

## 7. 分支候选的 JSON schema

为了让 judge 不被文风带偏，分支返回必须是结构化的。

建议 schema 如下：

```json
{
  "branchId": "b1",
  "lens": "ArchitectureFirst",
  "title": "先把系统骨架立住",
  "candidatePlanMarkdown": "# ...",
  "candidatePlanSummary": "用模块和边界先拆开问题",
  "keyAssumptions": ["需求可拆成三层", "落盘只发生一次"],
  "keyRisks": ["过度工程", "judge 误判抽象度"],
  "validationChecks": ["是否能单次执行", "是否只落盘一次"],
  "selfCritique": "这个方案可能太重，但边界清晰。",
  "confidence": 0.74
}
```

### 为什么要有 `candidatePlanMarkdown`

因为你要的是“计划”，不是“讨论提纲”。

每个分支都必须真的写出一个可以被人读懂的计划草案。

### 为什么要有 `selfCritique`

这能保留 `THATREPO` 的 build-and-break 气质。

分支不能只会夸自己，也要先指出自己最弱的点。

### 为什么要有 `validationChecks`

因为这是轻量版，不能执行真实代码，只能做“可验证的检查设计”。

它让后续人工实现或自动实现时，知道应该怎么验收。

## 8. judge 的评估标准

第二模型不是“谁最会写文档谁赢”，而是按以下标准筛：

### 8.1 贴题度

- 有没有真正回答用户需求
- 有没有抓到核心约束
- 有没有漏掉显著目标

### 8.2 可落地性

- 这份计划能不能被人接着实现
- 文件/模块/步骤是否清楚
- 是否把模糊地方显式标出来了

### 8.3 结构质量

- 计划层次是否清楚
- 是不是先目标后方案再风险
- 有没有重复啰嗦

### 8.4 风险意识

- 是否明确说出风险
- 是否指出会失败的条件
- 是否有回退/替代路径

### 8.5 分支独立性

- 每个分支是否真的不同
- 还是只是在改标题

### 8.6 约束遵守

- 是否遵守了用户指定格式
- 是否遵守“只消息、不运行、不落盘”的限制
- 是否遵守“不改 AgentRole”的限制

### 8.7 轻量但不轻浮

- 不能因为是 plan 就空泛
- 也不能因为要“原汁原味”就把系统做成大而全执行器

judge 最终要输出：

- 哪个分支是 winner
- 为什么 winner 最适合作为最终计划骨架
- 哪些分支的内容要合并进最终 markdown
- 哪些分支必须丢弃

## 9. `/plan` 的端到端流程

建议流程如下：

### 9.1 用户输入

用户输入：

```text
/plan 设计一个可支持多供应商的模型管理面板
```

### 9.2 预处理

主机先做：

- 取出需求文本
- 标准化空白
- 生成 plan id（hex4）
- 设定 branch count（默认 5）
- 准备 branch model 与 judge model

### 9.3 假设侦察

Kernel 先生成一组 `PlanHypothesis`。

这些假设不落盘，不执行，只作为分支共享输入。

### 9.4 多分支消息生成

每个分支拿到：

- 需求文本
- 自己的 lens
- 共享假设包
- 当前上下文摘要

每个分支返回：

- 一份结构化 JSON
- 一份具体的计划草案
- 自我批判
- 风险与验证点

### 9.5 第二模型评估

judge 收到全部分支候选，输出：

- winner
- kept branches
- rejected branches
- merge notes

### 9.6 最终计划合成

Kernel 把：

- 用户需求
- 假设侦察
- 分支对比
- judge 结论
- 选中的核心计划
- 被合并的辅助段落
- 风险/验收

渲染成一份完整 Markdown。

### 9.7 单次落盘

最后才调用写工具，生成：

```text
PLAN-7f3a.md
```

## 10. 计划文件 `PLAN-<hex4>.md` 的结构

建议最终文件固定包含这些章节：

```markdown
# PLAN-7f3a

## 需求

## 侦察到的不确定性

## 分支总览

## 分支候选对比

## Judge 结论

## 最终计划

## 实施步骤

## 验收标准

## 风险与回退

## 未决问题

## 附录：被淘汰分支摘要
```

### 10.1 需求

直接贴用户原话，保持原意。

### 10.2 侦察到的不确定性

列出这次 plan 最重要的几个未知点。

### 10.3 分支总览

列出每个分支的 lens、标题、confidence、简述。

### 10.4 分支候选对比

每个候选用小节写清：

- 适合解决什么
- 风险是什么
- 为什么被保留/淘汰

### 10.5 Judge 结论

说明赢家和原因。

### 10.6 最终计划

写成可以直接开工的计划。

### 10.7 实施步骤

按先后顺序写成可执行列表。

### 10.8 验收标准

写成明确可检查的标准。

### 10.9 风险与回退

写出失败时怎么退。

### 10.10 未决问题

写出仍然需要确认的点。

### 10.11 被淘汰分支摘要

保留一点 `THATREPO` 的 portfolio 透明度，但不要写成日志。

## 11. Branch lens 的建议设计

轻量版建议保留 5 个镜头，尽量贴 `THATREPO` 的“多世界线”味道。

### 11.1 DirectDelivery

- 目标：最快形成可实施计划
- 风险：太保守，容易漏掉结构性问题

### 11.2 ArchitectureFirst

- 目标：先把模块、边界、状态分清
- 风险：容易过度设计

### 11.3 RiskFirst

- 目标：先找失败点、冲突点、回退点
- 风险：可能太悲观

### 11.4 SimplificationFirst

- 目标：先砍到最小可验证子问题
- 风险：可能削弱全局视野

### 11.5 CrossDomainFirst

- 目标：用别的结构去看当前需求，例如状态机、流水线、分层协议、事件源
- 风险：可能变成修辞，不够落地

如果需求非常具体、非常工程化，也可以把第五个换成：

- ConstraintFirst

如果需求非常容易走偏，也可以把第五个换成：

- CounterexampleFirst

## 12. 不改 AgentRole 的兼容策略

这个方案不新增 AgentRole。

原因很简单：

- 你说要先纯函数 Kernel，再让 Opencode 和 Mux 兼容
- 现有角色系统已经足够复杂
- `/plan` 不是新的常驻角色，而是一个命令流

### 推荐做法

- 分支生成：直接调用模型，或调用现有 read-only reasoning 路径
- judge：再调用一次模型，作为第二模型评估
- 最终落盘：调用现有 write 工具

### 为什么这样最稳

- 不用改权限模型
- 不用改 host role 映射
- 不会影响 `editor / greper / reverie / browser / reviewer`
- Opencode 与 Mux 可以共享同一套纯内核

## 13. Opencode 侧怎么接

Opencode 侧建议做两件事：

### 13.1 Slash command 接入

在现有 `command.execute.before` / slash command 注册中增加 `/plan`。

### 13.2 主流程

接到 `/plan` 后：

- 解析需求文本
- 调 Kernel 生成 plan request / hypotheses / prompts
- 并行跑 5 个 branch messages
- 再跑 judge message
- 生成最终 markdown
- 用 write 工具落盘 `PLAN-<hex4>.md`

### 13.3 显示给用户的中间状态

建议只显示短状态，例如：

- 正在生成 5 个计划分支
- 正在第二模型评估
- 正在生成最终计划文件
- 已写入 `PLAN-7f3a.md`

### 13.4 Opencode 的关键点

- 不要让分支真实调用写工具
- 不要把分支做成会动手改仓库的 agent
- `/plan` 只是思考流，不是执行流

## 14. Mux 侧怎么接

Mux 侧建议做同样的三步：

### 14.1 slash command 接入

在 `src/MuxPlugin/MuxSlashCommands.fs` 增加 `/plan`。

### 14.2 host adapter

保留 Host 层兼容，Kernel 统一输出结构。

### 14.3 输出方式

Mux 最终也只做一件事：

- 由 LLM 调 `write` 工具落盘计划文件

### 14.4 Mux 的关键点

- 不要用工具结果驱动 branch 生成
- 不要把 `/plan` 做成“编辑器写计划”的变体
- 让 Mux 也只跑消息分支

## 15. 该方案最重要的落地点

这里是本次移植最关键的三个点。

### 15.1 核心不在执行，而在评估

`THATREPO` 真正可移植的是：

- 分支化思考
- 证据化评估
- judge 收敛

不是“能跑几个 agent”或者“能写几个文件”。

### 15.2 分支必须是正交的

如果 5 个分支只是换个说法，那就不值当。

每个分支必须真的有不同的：

- 目标
- 风险偏好
- 抽象层次
- 验证路径
- 取舍逻辑

### 15.3 最终文件只写一次

这是你提出的堵点，也是这版方案的硬约束。

所有分支都不写盘，只有最终 plan 写盘。

## 16. 运行时行为建议

### 输入

```text
/plan <需求>
```

### 过程中的状态消息

- 生成计划分支中…
- 第二模型评估中…
- 合成最终计划中…
- 已生成 `PLAN-7f3a.md`

### 最终输出

- 给用户一个简短确认
- 给出文件名
- 如有必要，给出一句总结

## 17. 冲突处理规则

### 17.1 分支冲突

如果分支之间冲突，先看谁更贴用户需求，而不是谁更漂亮。

### 17.2 judge 冲突

如果 judge 倾向于一个低 confidence 候选，但它解决了真正的核心约束，就选它。

### 17.3 模型幻觉

如果分支返回了不合理内容，直接判为低质量候选，不要给它“风格分”。

### 17.4 文件名冲突

如果 `PLAN-<hex4>.md` 已存在：

- 重新生成 hex4
- 再写一次
