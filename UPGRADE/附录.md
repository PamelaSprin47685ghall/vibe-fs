# 附录：`/plan` 轻量版多分支引擎的概念背景

## 1. 这份附录的目的

这份附录是给**没有接触过 `THATREPO`** 的实现者、评审者、协作者看的。

主方案 `方案.md` 已经给了具体实现方向，但里面引用了很多来自 `THATREPO` 的概念，例如：

- `Strategy-as-lens`
- `Hypothesis Packet`
- `EDFS`
- `Structured Solution Pool`
- `Memory Bank`
- `PQF`
- `Final Judge Isolation`

如果读者没有看过 `THATREPO`，这些词会显得像术语堆叠，不容易理解为什么我们要这么设计。

所以这份附录要做三件事：

1. 把 `THATREPO` 的核心思想讲清楚。
2. 解释这些思想为什么值得移植到本项目。
3. 解释为什么这次移植是“保留概念、轻量落地”，而不是把对方的实现细节整套照抄。

这份附录是**概念背景文档**，不是实现说明书。实现说明请看 `方案.md`。

这里还要加一条非常重要的实现原则：

> 第一版 `/plan` 也不能退化成 best-of-N。

`THATREPO` 的精髓不是“多几个候选然后选一个”，而是：

- DFS 式的 branch-local 深挖
- BFS 式的局部候选扩张
- 显式重组/移植/交叉吸收

所以第一版哪怕只做单次闭环，也必须保留这三个动作的最小形态。

---

## 2. 先讲清楚：`THATREPO` 到底是什么风格的系统

如果只用一句话总结，`THATREPO` 不是一个普通的“让 LLM 直接回答”的系统，而是一套：

- **把问题拆成多个认知分支去探索**
- **让分支互相接受批判和证据约束**
- **最后由更高一层的判断者收敛出最佳方向**

它的重点不在“某个神奇模型”，而在**搜索协议**。

这个协议背后有一个很重要的认知假设：

> 单个 LLM 非常容易被自己的第一直觉绑住。

一旦它写出了一个看起来像样的答案，后续就很容易进入一种危险模式：

- 不是重新想，而是继续“证明自己刚才是对的”
- 不是换问题表示，而是修修补补
- 不是离开局部最优，而是在局部最优里打蜡抛光

`THATREPO` 试图解决的正是这个问题。

它不相信“多想一会儿”就自然能突破局部最优，而是明确地构造：

- 不同策略分支
- 不同假设探针
- 不同候选池
- 不同批判视角
- 最终独立裁决

也就是说，`THATREPO` 的真正创新是：

> **把“思考”从一个线性过程，变成一个有分支、有筛选、有记忆、有收敛的系统过程。**

---

## 3. `THATREPO` 的核心概念总览

下面按实现者最需要知道的顺序来讲这些概念。

### 3.1 Strategy-as-lens

这是 `THATREPO` 最重要的起点。

它不把“策略”理解成：

- 第一步做什么
- 第二步做什么
- 第三步做什么

这种策略其实只是顺序计划。

它把策略理解成：

- **观察问题的镜头**
- **解释问题的方式**
- **决定后续工作优先级的框架**

比如面对同一个任务，不同策略可以是：

- 从架构边界看
- 从约束冲突看
- 从性能瓶颈看
- 从用户体验看
- 从反例和失败模式看
- 从另一学科的对应结构看

策略的作用不是直接给出答案，而是把后续工作**拉进不同的搜索空间**。

为什么这点这么重要？

因为如果所谓“多分支”只是把同一个想法换五种说法，那就没有任何意义。

真正的多分支必须做到：

- 分支 A 和分支 B 会产出实质不同的后续工作
- 它们关注的证据、风险和成功标准不同
- 它们可能通向不同的最终结论

所以：

> `Strategy-as-lens` 的本质，是强制把问题投影到不同的认知坐标系里。

---

### 3.2 Hypothesis Packet

这是第二个非常关键的概念。

`THATREPO` 不是直接让策略分支开始做事，而是先生成一组**可测试的假设**。

这些假设本质上是在问：

- 哪些前提可能根本不成立？
- 哪些约束可能互相冲突？
- 哪些边界条件最容易击穿当前方案？
- 哪些看上去合理的解释其实只是幻觉？

然后系统会让独立的 hypothesis tester 去测试这些假设。

测试结果形成一个信息包，也就是 `Hypothesis Packet`。

这个信息包不是答案，它是：

- 已验证的风险
- 已排除的路径
- 仍未定论的不确定性
- 可以指导分支工作的证据

为什么这比直接开做更强？

因为很多时候，分支失败不是因为“执行差”，而是因为它建立在错误前提上。

如果在进入分支之前先把高风险假设打掉，可以明显减少后续无效探索。

所以：

> `Hypothesis Packet` 的本质，是在大规模展开方案之前，先做一轮低成本侦察。

---

### 3.3 Selective Injection

`THATREPO` 里，假设测试的结果不是一股脑喂给所有分支。

它有一个更细的机制：`Selective Injection`。

意思是：

- 有些证据对所有分支都重要
- 有些证据只对某些分支有意义

例如：

- 某个性能瓶颈的证据，对性能策略分支很关键
- 某个用户心智模型的证据，对 UX 分支更关键
- 某个结构性反例，对数学/证明分支更关键

如果把所有证据全量塞给所有分支，会出现两个问题：

1. 上下文污染，分支之间越来越像
2. 分支被无关信息淹没，失去自己的镜头

所以 `Selective Injection` 的价值在于：

- 保留共享证据
- 但避免分支收敛成一锅粥

轻量版 `/plan` 不一定要做非常复杂的 selective routing，但至少要保留这个思想：

> 并不是所有分支都应该看到完全一样的附加信息。

---

### 3.4 EDFS：Evolving Depth First Search

这是 `THATREPO` 的标志性概念之一。

传统 DFS 是：

- 选一条路径
- 一直往下走

而 `EDFS` 多了两个字：`Evolving`

也就是：

- 每条分支不是死的
- 它会带着历史持续演化
- 当证据表明这条分支不值得继续时，可以被替换

在 `THATREPO` 里，每个分支不是一个简单字符串，而是有：

- 分支版本号
- 分支历史
- 最近 correction / critique
- memory bank
- solution pool
- PQF 判断结果

这让一条分支看起来更像一条“世界线”，而不是一次性答案。

为什么这很强？

因为很多难题不是一轮 critique 就能解决的。你需要：

- 在同一个问题框架里多轮深入
- 但又不能无限沉没成本

`EDFS` 试图平衡的就是：

- 深挖
- 纠错
- 换道

因此在 `/plan` 的第一版里，哪怕不做完整多轮 EDFS，也必须保留它的最小骨架：

- 初始 branch draft
- critique
- revision

如果没有这一轮“被批判后再往下走”，那它不是 EDFS-lite，只是多初稿选择。

---

### 3.5 Structured Solution Pool

这是最容易被误解的概念之一。

“solution pool” 不是简单的“再生成几个答案”。

它更像是：

- 在当前主分支之外，再向周边做一圈 BFS
- 生成多个候选工件
- 给主分支提供可借鉴、可对比、可替换的材料

它的作用不是直接替代主分支，而是：

- 防止主分支钻牛角尖
- 给 correction agent 提供替代思路
- 给 judge 更多比较面

在 `THATREPO` 里，solution pool 不是完全游离的，它和当前 critique、当前 memory、当前 branch 状态强耦合。

也就是说，它不是“顺便想几个点子”，而是：

> 针对当前分支最容易卡住的位置，构造一组高价值候选。

第一版 `/plan` 里，这个概念不应该只退化成“全部候选的集合”。

它至少应该变成：

- 每个 branch 在主 draft 之外，再生成一小组 branch-local pool entries
- 这些局部候选用来给该 branch 的 revision 做 BFS 邻域扩张
- 这些候选也能为最终 judge 提供合并素材

也就是说，第一版必须保留**局部 BFS**。

---

### 3.6 Recursive Memory Bank

这是另一个非常值得移植的概念。

`THATREPO` 的 memory bank 不是“总结答案”，而是总结：

- 哪些探索方向有效
- 哪些假设已经被打掉
- 哪些失败模式反复出现
- 哪些局部修复其实没用
- 哪些约束是必须记住的

也就是，它记的是：

> **探索史，而不是解答史。**

这点极其重要。

因为如果 memory 只压缩“最后写了什么”，那它会让系统越来越短视。

真正应该压缩的是：

- 已知死路
- 已知不变量
- 已知风险
- 已知不能再重复犯的错

这就是为什么在轻量版 `/plan` 中，即使不实现完整递归 memory，也应该至少保留：

- 被淘汰方案的原因
- judge 选择原因
- 本轮最高价值不确定性

这样下一轮 `/plan` 就不是完全失忆。

---

### 3.7 PQF：Post Quality Filter

`PQF` 的作用不是“挑最好答案”，而是更粗粒度地判断：

- 这条分支还值不值得继续长？

也就是说，PQF 面向的是**分支级生死**，不是候选级打分。

它判断的是：

- 这条策略是否已经结构性失效
- 这条分支是不是一直在重复同一种错误
- 这条分支是不是虽然努力，但其实优化错了目标
- 这条分支是不是和别的分支重复得太厉害

这跟 final judge 完全不同。

轻量版 `/plan` 里没有真正的多轮 branch life-cycle，但 judge 仍然应该保留一点 PQF 精神：

- 淘汰明显空泛、重复、不可落地的分支
- 保留少量真正有价值的分支

所以你可以把轻量版 judge 看成：

- `Final Judge`
- 加一点 `PQF`

---

### 3.8 Final Judge Isolation

`THATREPO` 特别强调一件事：

- 最终裁决者不能被整个搜索过程污染

它最好只看：

- 当前候选
- 或候选的必要摘要

而不是看见：

- 某个候选在前面被夸了多少
- 哪条分支用了多少轮
- 哪个 agent 自信心多高
- 哪条分支写得特别长

因为这些都会带来偏差。

一个好的 judge 应该做的是：

- 基于候选内容本身比较
- 基于要求和约束做选择
- 尽量少受“生成过程戏剧性”的影响

这也是为什么这次 `/plan` 设计里，judge 只应该看到：

- 结构化候选
- 候选摘要
- 风险与验证点

而不应该看见隐藏的实现过程噪声。

---

### 3.9 Build-and-Break Portfolio

这一点在 `THATREPO` 里非常有味道。

它不是只要求系统生成“可行方案”，而是要求方案组合同时具备：

- 构造性
- 破坏性

也就是：

- 有些分支负责提出建设性方案
- 有些分支负责提出反例、边界、失败条件、风险点

这样做的意义在于：

- 避免整个系统只会乐观推进
- 避免系统只会顺着最早的成功路径一路自我强化

在轻量版 `/plan` 里，这意味着分支候选不能只写：

- 我建议这么做

还必须写：

- 为什么它可能不行
- 哪些条件下会失败
- 如果失败，说明什么

否则它只是“建议池”，不是“搜索池”。

并且在实现上，build-and-break 也不应只表现为一个 `selfCritique` 字段，而应同时体现在：

- critique 对 draft 的攻击
- pool 对 branch 的替代展开
- revision 对 pool 和 critique 的吸收

这样第一版才真正具备搜索味道，而不是选美味道。

## 4. `THATREPO` 为什么不适合直接照搬执行层

这是本次方案最关键的工程取舍。

如果只看概念，`THATREPO` 很迷人。

但如果把它的“多分支演化”直接搬到本项目，有一个马上爆炸的现实问题：

> 不同分支如果同时改同一个真实工作区，文件夹会被写坏。

具体会坏在这些地方：

### 4.1 分支不再独立

一条分支的写入，会改变另一条分支看到的世界。

于是本来应该是：

- branch A 探索 A
- branch B 探索 B

会变成：

- branch B 实际上探索的是“被 A 改过之后的 B”

这会让整个搜索协议失真。

### 4.2 critique 会被污染

如果 critique 看到的工件已经被别的分支改过，它的批判对象就不再清晰。

### 4.3 memory 会不可回放

共享写盘让历史变成“执行顺序偶然产物”，而不是可重建的分支轨迹。

### 4.4 PQF 会判断失真

它以为自己在评估一条策略分支，实际上看到的是多条分支混写后的结果。

### 4.5 final judge 会依赖竞态顺序

谁先写、谁后写、谁覆盖谁，都会影响最终结果。

所以本次实现的核心取舍非常明确：

> 我们移植 `THATREPO` 的**认知协议**，不移植它在这里会变危险的**执行并发模型**。

---

## 5. 轻量版 `/plan` 是怎么对应这些概念的

为了让实现者一眼看懂，下面给一个对应表。

| `THATREPO` 概念 | 轻量版 `/plan` 中的对应 |
|---|---|
| Strategy-as-lens | 多个 `PlanLens` 分支 |
| Hypothesis Packet | 计划前的需求不确定性侦察 |
| Selective Injection | 不同分支拿到不同重点假设/附加上下文 |
| EDFS | 第一版做单次闭环版：draft -> critique -> revision |
| Structured Solution Pool | 每个 branch 的局部 BFS 候选池 |
| Recursive Memory Bank | 先保留为计划摘要与淘汰理由 |
| PQF | judge 的淘汰/保留逻辑 |
| Final Judge Isolation | 第二模型只看候选，不看隐藏过程 |
| Build-and-Break | 每个候选都带 `selfCritique` / 风险 / 验证点 |

换句话说，这次 `/plan` 不是“缩水版聊天命令”，而是：

- 保留 `THATREPO` 的认知结构
- 删掉高风险执行并发
- 做成一个可控、可落地、可复用的规划引擎

## 6. 为什么这次要先做“纯消息分支 + 第二模型评估”

这是有意为之，不是偷懒。

### 6.1 它先验证的是“协议有没有价值”

很多时候最容易犯的错误是：

- 还没确认概念值不值，就先把执行层做复杂

这样会导致：

- 工程成本高
- 调试困难
- 一旦效果不明显，很难判断是协议不行还是执行层太乱

纯消息分支的好处是：

- 先验证 strategy / hypothesis / PQF / judge 这套认知结构
- 先不引入共享盘问题
- 先把核心价值跑出来

### 6.2 它是最自然的 Kernel-first 设计

既然你要求：

- 先纯函数实现在 Kernel
- Opencode 和 Mux 两边兼容

那最自然的实现就是：

- Kernel 定义协议和文档生成
- Host 只负责调模型和写最终文件

消息级分支非常适合这件事。

### 6.3 它保留未来升级空间

以后如果真的要升级，也可以往两条路走：

1. 分支返回更细粒度工件，例如 patch 候选
2. 分支进入内存快照工作区，而不是共享真实文件夹

但这些都应该建立在：

- 先确认概念层确实有增益

之上。

---

## 7. 这次不做什么

为了防止实现者误解，这里明确写出**这次不做的东西**。

### 7.1 不做真实多分支执行

分支不会：

- 执行 shell
- 跑测试
- 改代码
- 写文件

### 7.2 不做多分支共享目录

不会出现：

- branch A 改了文件再让 branch B 继续看

### 7.3 不做完整 EDFS 生命周期

第一版不实现：

- branch version 增长
- 多轮 correction / critique loop
- 真正的 branch replace
- 真正的 recursive memory

### 7.4 不改 AgentRole

这次不新增：

- planner
- judge
- branch-explorer

这些职责先作为命令流程中的逻辑角色存在，而不是系统角色枚举中的新 case。

---

## 8. 实现者最需要记住的一句话

如果让一个没看过 `THATREPO` 的实现者只记住一句话，那应该是：

> `/plan` 要做的是“把一个需求扔进多个认知镜头里，先侦察关键不确定性，再生成结构化计划候选，由第二模型比较和筛选，最后一次性把最佳计划写成 Markdown”，而不是“并发改代码”。

---

## 9. 和 `方案.md` 的关系

- `附录.md` 负责解释：这些概念为什么存在、为什么值得移植、为什么轻量化。
- `方案.md` 负责解释：具体要建哪些类型、哪些函数、哪些文件、哪些接线点。

如果你是要找人实现：

1. 先让他看 `附录.md`
2. 再让他看 `方案.md`

这样他先明白“为什么”，再去做“怎么做”。

把这两份文档一起给实现者时，建议附一句交接说明：
