## 一、宝典符合度评估

以六条第一性原理为度量尺：

| 原理 | 符合度 | 证据 |
|---|---|---|
| 1. 稳定资产是领域规则非宿主API | B+ | Kernel 状态机（Review/Nudge/KG/Todo）纯函数且无 Dyn 泄漏。但 KG Runtime 三处等价拷贝，领域逻辑泄漏到宿主层 |
| 2. 历史是事实内存是投影 | A- | BacklogProjection/WorkBacklog/replay 机制已落地。NudgeEffect 的 StateHolder 需确认是缓存还是不可恢复真理 |
| 3. 副作用压到边界 | B | Dyn/obj 边界严格经 Shell codec。但 `Domain.fs:100` 读 `DateTimeOffset.UtcNow`——Kernel 内直接读时钟，是真侧效泄漏 |
| 4. LLM边界尽快强类型化 | A | 架构测试 126+ 条目强制全工具链经 Decode+Execute，Hook output 禁裸写 Dyn。执行最彻底 |
| 5. 长会话折叠进度可压缩 | A- | completedWorkReport + BacklogSessionCodec 已落地 |
| 6. 并发本质是共享可变状态 | A | PromiseQueue 串行域 + RuntimeScope 派生实例，SessionExecutor 无模块级队列 |

文件体量铁律已守住（0 个超 300 行），非主要矛盾。

## 二、偏离点清单（按优先级降序）

### P0：Kernel 纯度缺口——时钟直读

位置：`src/Kernel/Domain.fs:100`

```
System.DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()
```

这是唯一真正的 Kernel 侧效泄漏。违反原理 3"不读时钟"。后果：Kernel 函数不再同入同出，重放依赖墙钟运气。

架构测试 `kernelBoundary` 只拦 `Dyn.` 和 `open Shell`，未守 `DateTimeOffset`/`UtcNow`，是覆盖盲区。

注意区分：
- `KnowledgeGraph/Id.fs:17` 的 `randomInt` 是注入式参数（签名 `randomInt: unit -> int`），合规
- `Yaml.fs`/`PromptFrontMatter.fs` 的 `jsNative`/`Emit` 是纯值 interop（无 IO），可接受

### P1：三宿主知识重复——同一事实多处拷贝

| 重复项 | 位置 | 行数 | 判定 | 合并风险 |
|---|---|---|---|---|
| `StateHolder<'state>` | Opencode/NudgeEffect.fs:22-28 + Shell/NudgeRuntime.fs:18-24 | 6×2 | 真DRY违反 | 零 |
| KGTestHooks | Opencode(33行) + Mux(46行) | 33+46 | 真DRY违反 | 低 |
| collectSnapshot | Opencode + Shell 两处 | 22×2 | 骨架等价，数据源异 | 中 |
| ReviewLoop 循环 | Opencode + Mux + Omp 三处 | ~30×3 | 骨架等价，IO异 | 中 |
| KG Runtime | 三处各 190+190+159 行 | 539 | 字段同构，构造异 | 高 |

判定标准（宝典）："两段像只说明此刻长得像，不说明同一份知识。"逻辑是否依赖宿主API：
- `StateHolder`/`KGTestHooks`：零宿主耦合，纯泛型容器+测试接口。同一知识，必合并
- `collectSnapshot`/`ReviewLoop`/`KG Runtime`：骨架不依赖宿主，差异仅在 IO 调用点（`session.prompt` vs `pi?sendMessage` vs `delegate`）。骨架是同一知识，IO 是宿主领土。应提取高阶函数，非抽象基类

## 三、拒绝清单（不做什么）

| 拒绝项 | 理由 |
|---|---|
| 给三宿主加抽象基类 | 基类内塞 `if host==X` = 偶然复杂度。用高阶函数组合 |
| 把所有 FFI 移出 Kernel | Yaml 解析是纯计算，移出增 codec 往返。只移真侧效（nowMs） |
| 追求零重复 | 若三宿主差异确实需要分居，强行合并制造耦合 |
| 先拆文件再修架构 | 文件体量非主要矛盾。先修 P0/P1 知识问题，黄牌文件自然瘦身 |
| 大批量自动化改代码 | 手工编辑步步为营。每步编译+测试验证 |

## 四、安全网（不可破不变量）

重构过程中，以下不变量由三层机制守卫：

| 守卫层 | 机制 | 破坏症状 |
|---|---|---|
| 物理边界 | F#类型系统 + Fable编译 | 编译断：类型不匹配 / 引用未找到 |
| 架构探针 | 126+ 架构测试条目 | 测试红：边界违规被拦截 |
| 行为基线 | 231+ 端到端测试 | 运行崩 / 逻辑漂移 |

最危险 = 逻辑漂移：测试绿但行为变（Nudge 不触发 / Review 状态错 / KG 数据丢）。防范：每个迁移切片后跑 `npm run build-and-test`，确认全绿。

现有 8 个不变量域：

| 不变量域 | 守卫测试 | 重构期风险 |
|---|---|---|
| Kernel 禁引 Dyn/Shell | `kernelBoundary` | 注入参数时若误传 Shell 类型则破坏纯度 |
| Omp 禁引 Opencode/Mux | `ompBoundary` | 合并代码至 Shell 时易引入反向依赖 |
| Opencode↔Mux 禁互引 | `opencodeNoMuxRef` / `muxNoOpencodeRef` | 同上 |
| Hook output 经 codec | `ArchitectureTestsWireHook*` | 新增回调接口时若绕过 Codec 则破坏 |
| 全工具链经 Decode+Execute | `ArchitectureTestsWireToolExec` | 同上 |
| KG 维护经 Shell | `ArchitectureTestsRuntimeKg` | 移动 StateHolder 时未绑 RuntimeScope 则破坏并发 |
| 单文件 ≤300 行 | `fileBodyUnder300` | 拆分文件时需同步更新路径映射 |
| 测试全绿 | 231+ 测试文件 | 任何操作后必须保持 |

重构期新增纪律：
1. 修改 Kernel 注入参数必守纯函数不碰环境
2. 迁移 StateHolder 至 Shell 必绑定 RuntimeScope 派生实例
3. 新增 Shell 抽象与回调接口必同步补全架构测试

## 五、执行方案（六 Phase 迁移）

核心原则：用高阶函数 + record 组合，不造类层级，不造 DI 容器。宝典："GoF 翻到末页只剩数据函数类型组合。"

每个 Phase 的标准流程：
```
Step A：读取涉及文件，确认当前结构
Step B：手工修改代码
Step C：npm run build-and-test（编译+全测）
Step D：确认全绿 → 进入下一 Phase
Step E：若红 → 回滚本步修改 → 重新分析
```

### Phase 1：Domain.fs 时钟注入化

目标：消除 Kernel 内唯一真侧效泄漏，还 Kernel 纯度。

涉及文件：
- `src/Kernel/Domain.fs`（修改）
- `tests/ArchitectureTestsFoundation.fs`（修改：补扫描规则）
- 所有调用 `nowMs` 的上游文件（修改：传入注入参数）

操作步骤：

Step 1：在 `src/Kernel/Domain.fs` 找到 `nowMs` 的定义和使用点（约第 100 行）。确认所有调用链。

Step 2：将 `nowMs` 从直接读取改为函数参数注入：
```
原：let nowMs () = System.DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()
改：从签名中移除，改为调用方传入 getTimestamp: unit -> int64 参数
```

Step 3：沿调用链向上推：每个调用 `nowMs` 的 Kernel 函数，签名增加 `getTimestamp: unit -> int64` 参数。直到到达 Shell 边界。

Step 4：Shell 边界函数创建实际的时间戳获取器：
```
let getTimestamp () = System.DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()
```
在调用 Kernel 函数时传入。

Step 5：在 `tests/ArchitectureTestsFoundation.fs` 的 `kernelBoundary` 测试中，增加对 `DateTimeOffset`/`UtcNow`/`DateTime.Now`/`Date.now` 的扫描，断言 `src/Kernel/*.fs` 不含这些字符串（白名单允许 `Fable.Core` interop）。

Step 6：`npm run build-and-test`，确认全绿。

验收标准：
- `src/Kernel/` 下全局搜索 `UtcNow`/`DateTime.Now`/`Date.now` 零匹配（排除 Fable.Core interop 注释）
- 架构测试 `kernelBoundary` 包含新的时钟扫描断言
- 全测绿

回滚策略：`git checkout` 涉及文件，重新分析调用链。

风险提示：
- nowMs 可能被多个 Kernel 模块间接调用，需完整追踪调用链
- 若调用链过深（>5层），考虑在 Kernel 定义 `Clock` 抽象类型，Shell 创建实例传入，减少参数穿透层数

### Phase 4：ReviewLoop 骨架收敛（Slice 3）— **DEFERRED**

> **结论：骨架不等价，强制收敛引入偶然复杂度。**

逐行对比三宿主 ReviewLoop 实现（2025-01）：

| 宿主 | 循环机制 | decideAfterRound | promptParts | maxNudges |
|------|---------|-----------------|-------------|-----------|
| Opopen/ReviewerLoop.fs | 显式递归 `loop` | ✓ L79 | ✓ L77 | ✓ L40 |
| Mux/ReviewToolsMux.fs | **无循环**，两次顺序 `runReviewRound` | ✗ | ✗ | ✗ |
| Omp/ReviewLoop.fs | **轮询 + graceMs** | ✗ | ✗ | ✗ |

**判定：DEFERRED 保持分居**
- Mux `submitReviewTool`：仅两次 `runReviewRound` 顺序调用，无 `decideAfterRound`/`promptParts`/`maxNudges` 循环控制
- OMP `runReviewLoop`：`runNudgeLoop` 轮询 + `graceMs` 等待机制，无 `decideAfterRound` 循环
- 强行提取公共高阶函数需要 `if host==X` 分支或抽象层 → 违反宝典"不造类层级""压缩不是合并"原则

**已留存的瘦身空间（不急于做）：**
- `src/Opencode/ReviewerLoop.fs` L75-82 的 `loop` 递归与 `src/Kernel/ReviewSession/StateMachine.fs` 的 `decideAfterRound`/`promptParts` 调用 → 已是最小化（仅 Opopen 一处）
- `src/Mux/ReviewToolsMux.fs` L107-111 两次 `runReviewRound` → 可考虑提取为 `runDoubleCheck` 工具函数，但收益有限
- `src/Omp/ReviewLoop.fs` L58-76 轮询机制 → 独立实现，无收敛对象

**量化目标调整：**
- ReviewLoop 骨架数：3 → 3（原目标 1，已调整为不适用）
- 架构测试 `noDuplicateReviewLoopSkeleton`：不注册（不存在可收敛的重复骨架）

> **方法论论文：** abduction 分析记录见方法论笔记 `abduction_Phase4_ReviewLoop_不等价`。working_backwards/invariance/test_driven_reasoning/decomposition_recombination 四篇论文已建立，结论为 deferred。

### Phase 5：KG Runtime 状态归一（Slice 4）— **CANCELLED**

目标：消除 KG Runtime 三处等价字段定义（state+commandQueue+registeredJobs+writeQueues），用 F# record 组合统一。

高风险原因：三处构造函数签名差异大（Opencode 6参 / Mux `?deps:obj` / Omp `pi:obj`），外部 API 紧耦合。可能引入隐式依赖。

宝典指导："边界先于抽象成熟。"若收益不足可只做部分合并，标记 deferred 保持分居。

**判定：CANCELLED。** 纯状态操作已收敛至 `Kernel/KnowledgeGraph/RuntimeState.fs`；`registeredJobs`/`writeQueues` 字段与三宿主构造参数/IO 注入点属于宿主领土，合并收益有限、风险高。按宝典"边界先于抽象成熟"原则，本 Phase 取消。

涉及文件：
- `src/Shell/KnowledgeGraphRuntimeState.fs`（新建）
- `src/Opencode/KnowledgeGraphRuntime.fs`（修改：持有 record，保留 IO 注入）
- `src/Mux/KnowledgeGraphRuntimeMux.fs`（修改：同上）

## 附：宝典原则速查

重构过程中随时对照：

1. 压缩不是合并，复用不是提前抽象。两段像只说明此刻长得像，不说明同一份知识
2. 唯一表示是同一事实多处重复并开始不一致。独立生命周期概念逐字相同也该分居
3. 边界先于抽象成熟。规则网络协议持久化权限语境视图各有领土
4. 类型系统是最便宜边防。概念独立命名在运行时零成本
5. 纯函数是内核：不读时钟不掷骰子不查库不发网不写盘不改入参
6. 命令和事件必分：意图可拒事实不可驳
7. GoF 翻到末页只剩数据函数类型组合
8. 文件体量是梯度刑场：200行亮黄牌，300行亮红牌，触发即新建文件
