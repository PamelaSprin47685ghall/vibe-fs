# 保姆级重构指南

> 目标：按 `TASK.md` 的问题清单，把 `vibe-fs` 从“能跑但语义发散”收敛成“命名准确、边界清晰、类型封闭、可重放、可测试”的 F# / Fable 工程。

## 0. 先定总原则

这份重构不是“顺手优化”，而是一次带约束的收口：

- 先保语义，再谈形态。
- 先收边界，再收实现。
- 先去 `obj` / 字符串魔法，再去重复代码。
- 先拆纯逻辑，再搬 IO。
- 先让编译器站岗，再让人肉 review。

仓库的第一性原理已经在 `README.md` 里讲得很清楚：

- `Kernel/` 放稳定领域语义。
- `Shell/` 放 Node、文件系统、网络、子进程等副作用。
- `Opencode/` 与 `Mux/` 只做宿主适配，不承载核心规则。
- 历史记录是事实来源，进程内状态只是投影。
- LLM 边界要尽快强类型化。
- 长会话一定会折叠，所以进度必须可压缩。

所有后续动作，都要服从这六条。

## 1. 这次重构要解决什么

按 `TASK.md`，问题分两大类：

### 1.1 名实不符

重点不是“名字不好看”，而是名字在误导维护者：

- `Magic*` 误导为“黑盒技巧”，实际是在做 backlog、projection、history compaction。
- `OllamaClient` 误导为 Ollama SDK，实际在打 web search / web fetch。
- `HostReadExec` 误导为执行器，实际只是宿主函数捕获容器。
- `TreeSitterKernel` 混入正则和工具分类，名字掩盖了职责污染。
- `NudgeHook` 只像一个小 hook，实际承担全局生命周期观测。

### 1.2 结构不当

重点是架构边界被打穿：

- `obj` / 动态反射渗透到业务核心。
- `Mux` 与 `Opencode` 重复实现同一套逻辑。
- 错误在字符串、异常、`Result` 之间乱飞。
- 全局可变状态横跨 session / workspace。
- `KnowledgeGraphRuntime` 责任过载。
- 工具签名太原始，DI 缺位，副作用直接缠进执行器。

## 2. 重构总路线

不要并行乱改。按下面顺序走，风险最小：

### Phase A：命名纠偏

先把“误导维护者”的词换掉。只改语义标签，不改行为。

### Phase B：边界强类型化

把 `obj`、字符串属性读取、运行时猜类型，尽量收敛到宿主边界。

### Phase C：抽公共内核

把 `Mux` / `Opencode` 的重复逻辑下沉到 `Kernel`，宿主层只保留差异。

### Phase D：统一错误模型

把工具链从“字符串错误 / 异常 / 半吊子解析”收敛到 `Result<'T, DomainError>`。

### Phase E：收拢可变状态

把全局 mutable 字典、单例 registry、session 状态改成显式上下文或 actor。

### Phase F：拆 `KnowledgeGraphRuntime`

把状态、存储、编排、测试门洞拆开。

### Phase G：重塑工具 DI

工具执行签名改成强类型输入 + 强类型上下文 + 纯返回值。

## 3. 命名重构：先改哪些词

命名重构的目标不是“统一风格”，而是“把真实职责说出来”。

### 3.1 `Magic*` 家族

问题：`MagicTodo`、`MagicCore`、`MagicSessionStore` 这些名字没有领域信息，只有情绪色彩。

建议方向：

- `MagicTodo` → `WorkBacklog` / `HistoryCompaction` / `SessionProjection`
- `MagicCore` → `BacklogProjectionCore` / `SessionProjectionCore`
- `MagicSessionStore` → `SessionProjectionStore` / `CompactionStore`

改名原则：

- 名字里必须出现对象是什么。
- 名字里最好出现它解决的业务问题。
- 不允许 `Magic`、`Helper`、`Util` 这种遮蔽语义的前缀继续扩散。

### 3.2 `OllamaClient`

问题：它不是 Ollama 客户端，而是某种代理化 web API 入口。

建议方向：

- 如果它主要封装网页搜索/抓取：`WebSearchApi`
- 如果它是更宽的模型后端网关：`AgentBackendClient`

改名时必须同步检查：

- 文件名
- 模块名
- 调用点变量名
- 测试名
- 文档里的 API 名称

### 3.3 `HostReadExec`

问题：这个名字暗示“执行器”，实际是“宿主原始函数捕获器”。

建议方向：

- `HostFunctionCapture`
- `NativeReadCallbackRegistry`
- `HostCallbackRegistry`

判断标准：

- 如果它不负责执行，只负责保存 / 暴露回调指针，就不要叫 `Exec`。

### 3.4 `TreeSitterKernel`

问题：文件名说的是 AST 解析内核，内容却混进了正则补丁解析、工具分类表。

建议拆分：

- `TreeSitterKernel` 保留纯 tree-sitter / AST 能力。
- `PatchParser` 放 `pathsFromPatchText` 这类文本解析。
- `ToolCatalog` 放 `isFileEditTool` 这类工具分类。

### 3.5 `NudgeHook`

问题：名字太小，职责太大。

建议方向：

- `SessionLifecycleObserver`
- `AgentEventTracker`
- `NudgeLifecycleHook`

如果它同时看 `chatMessage`、`toolExecuteAfter`、`commandExecute`，就不能再伪装成“只管 nudge 的小 hook”。

## 4. 边界类型丢失：怎么把 `obj` 赶出去

这是最关键的一类问题。`obj` 不是罪，但它只能活在入口边界。

### 4.1 先画边界

先明确三层：

- 宿主输入：`obj` / 动态消息 / 原生回调
- 边界适配：一次性 decode / normalize / validate
- 内核逻辑：只收强类型 record / DU / `Result`

### 4.2 目标状态

内部代码里尽量不要出现：

- `Dyn.get`
- `Dyn.str`
- 任意属性名字符串常量散落各处
- `obj` 直接穿透到核心函数

### 4.3 推荐做法

#### 做法 1：入口一次性解码

在 `Opencode/`、`Mux/` 的边界层，把宿主对象转成记录或 DU。

#### 做法 2：定义强类型边界契约

对能稳定建模的输入，优先用 record / DU。

#### 做法 3：让动态只活在胶水层

动态反射只允许存在于：

- `Codec`
- `Wrapper`
- `Transform`
- `HostAdapter`

#### 做法 4：把失败显式化

解码失败不要一路 `null` 下传，直接回 `Result`。

### 4.4 验收标准

- 内核模块不再直接碰宿主 `obj`。
- 动态属性读取集中到少数 codec / adapter 文件。
- 新工具 / 新 hook 不再复制一套字符串反射模板。

## 5. `Mux` / `Opencode` 重复代码：怎么抽而不抽死

这里最怕两种错误：

- 复制粘贴继续繁殖。
- 过早抽象成大而空的通用层。

### 5.1 先判断“真重复”还是“假相似”

看三件事：

- 逻辑是否相同。
- 差异是否只是宿主 API 签名不同。
- 差异是否只是对象形态不同。

如果答案都是“是”，就可以下沉公共逻辑。

### 5.2 下沉原则

公共逻辑下沉到 `Kernel`：

- 纯状态机
- 消息折叠
- review / nudge / knowledge graph 的决策逻辑
- 工具目录与权限判定
- prompt 拼装中的纯部分

宿主层只保留：

- tool registration
- wrapper 适配
- callback 转换
- message 读写
- session / workspace API 对接

### 5.3 抽象形态

优先顺序：

1. 纯函数公共模块
2. record / interface 适配层
3. 小型 adapter / functor

不要一开始就上巨型 `HostAdapter`，除非两个宿主的差异已经稳定且边界清晰。

### 5.4 典型迁移法

以 `SubagentTools.fs`、`MessageTransform.fs` 这类双生文件为例：

1. 先把完全一致的纯逻辑提到 `Kernel`。
2. 再把宿主差异封成最小的输入输出 record。
3. 最后让 `Mux` / `Opencode` 各自提供自己的 adapter。

### 5.5 验收标准

- 公共逻辑只在一处实现。
- 宿主文件只剩少量 glue。
- 任何业务规则变更，不需要同步改两份核心逻辑。

## 6. 错误模型：统一成 `Result<'T, DomainError>`

这是从“能跑”变成“可维护”的分水岭。

### 6.1 现状问题

现在常见的坏味道是：

- 工具函数直接回字符串。
- JS 异常靠正则倒猜。
- 错误语义在多个层次里来回翻译。
- 调用方不知道失败是业务失败还是系统事故。

### 6.2 目标语义

- 业务失败：显式 `Result.Error`。
- 系统事故：显式 `DomainError.SystemPanic` 或同类分支。
- 最终展示：只在编码到模型输出时格式化成字符串。

### 6.3 迁移步骤

1. 先让内核函数返回 `Result`。
2. 再让工具执行层接收并透传 `Result`。
3. 最后在宿主输出层统一做 stringify。

### 6.4 不要做什么

- 不要把错误拼进正常返回值字符串里。
- 不要让异常承担业务分支职责。
- 不要让正则分类器成为错误模型的主来源。

### 6.5 验收标准

- 调用链上能明确区分“业务失败”与“程序崩溃”。
- 失败路径可测试、可匹配、可枚举。
- 最终面向模型的文本只是展示层，不是事实层。

## 7. 全局可变状态：怎么收口

全局 mutable 最大的问题不是“脏”，而是“串”。

### 7.1 高风险点

按 `TASK.md`，典型对象包括：

- `MagicSessionStore`
- `FuzzyIteratorStore`
- `ChildAgentRegistry`
- `SessionExecutor`

### 7.2 目标状态

状态必须显式归属：

- session 级
- workspace 级
- process 级

不要让隐式单例横跨不同请求域。

### 7.3 两条可行路

#### 路线 A：显式上下文传递

把状态塞进 `SessionContext` / `WorkspaceContext`，沿调用链传递。

适合：

- 作用域清晰
- 状态量不大
- 需要测试可控

#### 路线 B：actor / 串行队列

把可变状态封在单线程串行执行体里。

适合：

- 高并发异步交错
- 顺序敏感
- 需要防重入 / 防串写

### 7.4 重构顺序

1. 找出顶层 `mutable`。
2. 标记它所属的最小作用域。
3. 先把作用域外的访问全部改成显式传参。
4. 再决定是 context 还是 actor。

### 7.5 验收标准

- 不同 session 不会共享同一份隐式状态。
- 关闭进程后可重建必要状态。
- 并发交错不会导致串数据。

## 8. `KnowledgeGraphRuntime`：必须拆

这是结构问题里最典型的 SRP 失守点。

### 8.1 它现在混了什么

- 内存状态 reducer
- IO 队列 / `SerialQueue`
- prompt 拼接
- 测试后门

这几件事本质不同，不能放一个对象里硬扛。

### 8.2 建议拆法

#### 8.2.1 `KnowledgeGraphState`

只放纯状态与转移。

职责：

- 事件如何更新状态
- daily rewrite 何时触发
- job marker 如何解释

#### 8.2.2 `KnowledgeGraphStorage`

只负责持久化。

职责：

- NDJSON 读写
- 文件锁
- 原子追加
- 恢复校验

#### 8.2.3 `KnowledgeGraphWorkflow`

只负责编排。

职责：

- 拉起 bookkeeper
- 安排 append / rewrite
- 连接 prompt 与 state

### 8.3 测试后门怎么处理

不要把测试专用方法继续挂在生产 runtime 上。

改法：

- 通过依赖注入给测试 stub。
- 或者给 workflow 单独暴露测试钩子接口。
- 不要让生产对象长出“只给测试用”的公开逃生门。

### 8.4 验收标准

- 状态、存储、编排三层彼此解耦。
- 纯逻辑能单测，不依赖 IO。
- 维护策略改动不需要碰存储实现。

## 9. 工具定义与 DI：从“obj 乱摸”改成“强类型输入”

工具是本项目最容易失控的入口。

### 9.1 现状问题

- `execute` 闭包里硬读巨型 `config` / `context`。
- 参数靠字符串 key 到处捞。
- 子代理创建、网络请求等副作用直接贴在工具内部。

### 9.2 目标签名

建议朝这个方向收：

`ToolArgs -> IToolContext -> Async<ToolResult>`

更准确地说：

- `ToolArgs`：强类型工具参数
- `IToolContext`：显式能力集合
- `Async<ToolResult>`：统一返回

### 9.3 上下文应直接提供什么

至少应该显式提供：

- `CurrentDirectory`
- `AbortToken`
- `ILogger`
- session / workspace 标识
- 必要的宿主能力接口

### 9.4 迁移步骤

1. 先为每个工具定义专属参数 record。
2. 再为工具创建窄接口 context。
3. 把 `Dyn.str ctx "directory"` 这类读法收进边界层。
4. 最后把工具内部副作用改为调用 context 能力。

### 9.5 验收标准

- 工具函数签名一眼能看懂依赖。
- 工具内部不再自行挖掘巨型 `obj`。
- 新工具能直接复用上下文契约。

### 9.6 文案归类与权限语义化

这是本次新增的横切收口项，和前面的 `obj` 收敛同级，但目标不同：它收的是“文本”和“权限键”。

#### 9.6.1 文案去重

先把所有可见文本拆成固定类型：

- tool description
- field description
- refusal text
- error prefix
- success summary
- prompt snippet

然后建立统一文案目录，工具实现只取语义键，不再直接拼长句。

目标不是“把文案集中到一个文件”，而是把同义文本收成同一事实源，避免 `Opencode`、`Mux`、`Mimo-Code` 三边各写一版。

#### 9.6.2 权限映射

把工具权限先映射成有限语义类型，再做 pattern matching：

- `read`
- `write`
- `search`
- `edit`
- `submit`
- `notify`
- `capture`
- `inspect`

禁止直接拿工具名字符串做核心权限判断。字符串只能作为最外层输入，进入核心前必须转成语义类型。

参考方向：`opencode/packages/core/src/tool/tool.ts` 的 canonical tool + permission action 模型，说明权限应该跟工具语义绑定，而不是跟宿主字符串绑定；`mux/src/node/services/tools/*` 里的 description / confirm / error 文案，说明文案也应该从执行器里退出。

#### 9.6.3 落地步骤

1. 盘点三处宿主的全部工具名称、description、error 文本、confirm 文本、参数说明。
2. 抽出统一文案表，按语义键归档。
3. 抽出工具语义映射函数，把字符串工具名折叠到有限 DU / enum。
4. 所有权限判断改成对语义类型的 pattern matching。
5. 所有工具执行器只保留业务逻辑，不再负责文案拼装和权限猜测。

#### 9.6.4 验收标准

- 同一语义的文案只保留一处事实源。
- 新增工具时只需要补语义映射，不需要复制权限分支。
- 权限规则能被穷尽匹配，新增语义会在编译期或测试期暴露遗漏。
- 工具执行器里不再出现大段重复英文提示词和错误前缀。

## 10. 推荐重构顺序

如果要真改，按这个顺序最稳：

1. `Magic*` 改名。
2. `OllamaClient` / `HostReadExec` 这类误导名纠正。
3. 把 `obj` 解析集中到 codec / wrapper。
4. 抽 `Mux` / `Opencode` 真重复逻辑。
5. 统一错误到 `Result<'T, DomainError>`。
6. 收拢全局 mutable。
7. 拆 `KnowledgeGraphRuntime`。
8. 重塑工具 DI。

这个顺序的好处：每一步都在减噪，不在加新噪音。

## 11. 每一步都怎么验收

### 11.1 命名验收

- 新名字能直接说出职责。
- 读文件名不需要猜。
- 搜索调用点能定位到同一概念。

### 11.2 边界验收

- `obj` 不再跨核心层传播。
- 宿主差异只在 adapter 层。
- 解析失败能被显式处理。

### 11.3 重复代码验收

- 同一业务规则只实现一次。
- `Mux` / `Opencode` 只保留差异。
- 修改一个规则不用追两份。

### 11.4 错误验收

- 业务失败可枚举。
- 系统故障可区分。
- 展示层不反向污染内核。

### 11.5 状态验收

- 不出现跨 session 串状态。
- 重启后能恢复必要状态。
- 并发不会把数据写乱。

### 11.6 KnowledgeGraph 验收

- 纯状态可单测。
- 持久化可替换。
- 编排层不混存储细节。

### 11.7 工具 DI 验收

- 工具签名不再依赖巨型 `obj`。
- 工具能力来自显式上下文。
- 副作用在边界层被说明白。

## 12. 回滚策略

重构不是一次性赌博，必须可退。

### 12.1 先改可逆层

- 先改命名。
- 再改 codec / adapter。
- 后改核心逻辑。

### 12.2 每次只碰一个语义块

不要同时改：

- 命名
- 文件移动
- 签名
- 错误类型

同时改四样，回滚会很痛。

### 12.3 保留过渡适配层

如果下游很多，先保留薄 wrapper，再逐步削掉。

### 12.4 回滚判断

出现以下任一情况就先停：

- 编译不稳定。
- 旧测试大面积失真。
- 边界层被继续侵蚀。
- 新抽象开始比旧代码更难读。

## 13. 最后一句

这次重构的判定标准很简单：

- 名字不再骗人。
- 边界不再穿透。
- 错误不再混血。
- 状态不再串线。
- 重复不再扩散。
- 代码读起来像系统本身，而不是像一堆救火记录。

如果某一步不能让这些事实更明显，就别做。
