# message transform 钩子重构计划

## 0. 背景：当前 bug 面

当前排查的三类问题共享同一个根因——**transform 钩子每轮都做"整体重算"，而不是"增量维护"**：

1. `MessageTransformHostEntry.fs` 里的 `SessionTransformCache` 用深比对（逐元素 `obj.Equals`）+ 6 个字段的 `FingerprintMetadata`（Model/ContextLimit/TokenUsage/BacklogRevision/EventlogRevision/ReviewRound/ReviewFeedback/CapsVersion/...）做缓存判定。任何一个字段抖动（比如 token 估算数值波动、EventLog revision 递增）都会让整条缓存失效，触发全量 pipeline 重跑，产生全新的消息对象，破坏 provider 端的 prompt-cache 前缀命中。
2. `Messaging.stripSyntheticBySource` 在每次 transform 开头无条件扫描并剔除**所有** `Synthetic` 消息（caps、backlog projection、context-budget-nudge、parallel-tool-synth），然后管线重新计算、重新插入。哪怖只有栈顶的 nudge 需要换一条，也要把栈底稳定了几十轮的 caps 前缀一起摘掉再蒂回去。
3. `messagesTransform` 钩子里通过 `extractTextsFromEncodedMessages` + `inferReviewTaskFromTexts` 从消息文本里正则解析 `task:` frontmatter 来推断 review 是否激活——这不是 transform 钩子该做的事，因为消息切片在 compaction 后可能看不到激活锚点，且真相源应该只有一个：`.wanxiangshu.ndjson`。

## 1. 设计纠偏（用户指正后的最终版本）

**关键认识**：opencode 主机传给 transform 钩子的 `messages` 数组，其**前缀本身就是 host 侧维护好的、跨轮次稳定的引用**（这是 provider prompt-cache 依赖的基础事实，不需要我们重新发明）。我们的插件只是在这个数组的**末尾**（有时中段，见 backlog）追加/替换少量 synthetic 消息。

所以整个"缓存"根本不需要指纹、哈希、深比对。它只需要退化成一个**真栈**：

- 记住"上一轮我在栈顶推了几个 synthetic 元素"（一个整数）。
- 本轮开始时，从**原始 encoded 数组**弹出这么多个末尾元素（`array.length -= N`），得到"稳定前缀"。
- 在稳定前缀之后，重新计算并 `push` 新的 synthetic 元素。
- 记录新的推入数量，供下一轮弹出。

不比较内容、不算指纹、不存 fingerprint。稳定前缀的 JS 对象引用完全不变 → provider 端 JSON 序列化字节完全一致 → 前缀缓存 100% 命中。

## 2. 三类 synthetic 注入各自的栈语义

不同 synthetic 注入在数组中的位置不同，处理方式也不同：

| 注入类型 | 位置 | 触发重算条件 | 栈操作 |
| :--- | :--- | :--- | :--- |
| **CAPS 前缀**（caps-synth-user/assistant/ack + caps-tool 读取） | 数组**最前面**（下标 0 起 N 条） | 本 session 内存中"上次是否已经建过"——建过就直接复用引用，**永不因内容变化重建**（除非进程重启，内存状态本来就丢了） | 只在 `None`（从未建过）时 `unshift` 一次；后续所有轮次原样保留同一份引用 |
| **Backlog 折叠前缀**（backlog-prefix-N） + **backlog 折叠区间本身** | 数组**中段**（todo 锚点之间的一大段原始消息被替换为几条折叠文本） | `foldedBacklog` 条目数变化（即新 todowrite 完成） | 记录"上一轮折叠到了第几个 backlog 条目"，若条目数增加：把上一轮产生的折叠前缀+折叠区间**整体弹出**，重新用新的 `foldedBacklog` 生成一遍再 append；若条目数不变，直接复用上一轮产出的那一段引用 |
| **context-budget-nudge** | 数组**最后 1 条**（如果需要） | 每轮都可能变 | 若本轮判定不需要nudge，且上一轮有：弹出 1 条；若需要且上一轮没有：push 1 条；若两轮都需要且是"同一 episode"：复用同一个 id/引用，不弹出不重建 |
| **parallel-tool-synth** | 数组**最后 1 条**（如果需要） | 每轮都可能变 | 同 nudge：pop-if-present + push-if-needed，二者互斥不会同时出现在栈顶 |

**核心纪律**：三类互不干扰，栈顶只有 nudge 和 parallel-hint 会竞争"最后一条"的位置（两者本来就有先后依赖关系，pipeline 顺序已经决定了谁在下面谁在上面）。CAPS 在最前面，backlog 在中段，二者与栈顶操作在数组的不同区域，互不覆盖。

## 3. 新增/删除的模块

### 删除
- `Shell/MessageTransformHostEntry.fs` 里的 `TransformFingerprint`、`FingerprintMetadata`、`SessionTransformCache`、`computeFingerprint`、`fingerprintEqual`、`computeMetadata`、`getSessionCache`——整套深比对缓存机制全部删除。
- `Opencode/MessageTransform.fs`、`Mux/MessageTransform.fs`、`Omp/MessageTransform.fs` 三处 `messagesTransform` 里对 `stripSyntheticBySource` 的**无条件调用**——改为只在真正需要重算的分支里调用，而不是每轮开头就调用一次。
- `runHostMessagesTransform` 的 `_reviewReplayMode: ReviewReplayMode` 和 `_replayTexts: unit -> JS.Promise<string seq>` 两个已经是死参数（下划线前缀说明当前实现根本没用）——直接从函数签名里删掉，三个宿主调用点同步删除对应实参。
- `Kernel/ReviewReplayPolicy.fs` 的 `reviewTaskFromTexts`、`Shell/ReviewReplaySync.fs` 的 `syncReviewFromTexts`——不再在 messagesTransform 路径里调用（`inferReviewTaskFromTexts` 本身在 `Kernel/LoopMessages.fs` 保留，因为它有独立单测且不违反纯函数原则，只是不再被 transform 钩子调用）。

### 新增
- `Shell/MessageTransformStack.fs`：三个真栈的状态与操作，每个都只有"数量"或"引用"两个字段，不做指纹比对：

```fsharp
module Wanxiangshu.Shell.MessageTransformStack

// CAPS：建过一次就是 Some，永不重建。存的是"已经拼好的前缀 encoded 片段"引用。
type CapsSlot = { Prefix: obj array }  // 长度 = 已注入的 caps 消息条数

// Backlog：栈顶几条 + 折叠了多少个 backlog 条目。
type BacklogSlot =
    { FoldedCount: int          // 上一轮 foldedBacklog.Length
      PrefixAndProjection: obj array }  // 上一轮产出的"backlog-prefix-* 消息 + 折叠后的锚点消息本身"这一段的引用

// 栈顶单条：nudge 或 parallel-hint，各自独立，只记 Some/None + 引用。
type TopSlot = { Item: obj option }
```

每个 slot 都存在 `RuntimeScope` 里，key 用 `sessionID`，和 `ContextBudgetStore` 一样的存法（`RuntimeScope.TryFindKey`/`Add`），跟进程生命周期，不落盘——完全符合"不重启就一直用旧的，重启即重建"的既定语义。

### 修改
- `Shell/MessageTransformPipeline.fs::runMessageTransformPipeline`：把"每轮全量跑 5 个阶段"改成"先问每个栈是否需要动，只动需要动的那一段"。
- `Shell/MessageTransformCore.fs::checkAndInjectNudge`：不再依赖上游已经把 nudge 从数组里剥掉这一前提，改成自己精确操作 `TopSlot`。
- `Shell/MessageTransformHostHooks.fs::loadCapsForScope`：不变（caps 文件读取本来就有 `RuntimeScope.CapsFiles` 级缓存），但调用方（`Opencode/MessageTransform.fs`）里"要不要重建 caps 消息"的判断从"fingerprint 变了就重建"改成"`CapsSlot` 是 `None` 才建"。

## 4. Review 状态：messagesTransform 钩子里做什么（用户澄清后的结论）

用户问："完全不参与，但为何要渲染提示？渲染什么提示？"——这提醒了一个关键点：**目前 messagesTransform 钩子里其实没有独立的"review 提示渲染"步骤**。查证源码：

- `/loop` 命令的提示文本（`buildLoopMessage`）由 `Opencode/CommandHooks.fs::commandExecuteBefore` 在**命令执行时**直接写入 `output` parts，不经过 `messagesTransform`。
- `submit_review`/`return_reviewer` 的反馈文本由 `Opencode/ReviewTools.fs` 在**工具执行时**直接写入工具返回值。
- `messagesTransform` 钩子目前对 review 状态的**唯一**触碰点，就是 `Opencode/MessageTransform.fs` 里从未被调用的 `replayTexts` + 从未被传给 `runHostMessagesTransform` 内部逻辑使用的 `_replayTexts` 参数——它是历史遗留的死代码路径（签名里带下划线前缀已经说明了这一点），**当前根本没有生效的"文本解析恢复 review"逻辑在跑**。

**结论**：messagesTransform 钩子对 review 不需要做任何事——不渲染、不推断、不同步。删除 `replayTexts`/`_replayTexts` 这个死参数就是全部动作。真正的 review 状态同步路径（`syncReviewFromEventLogDedicated`，在 `/loop` 命令执行、reviewer 会话结束时调用）保持不变，它已经是从 `.wanxiangshu.ndjson` 单向恢复到 `ReviewStore` 内存投影，符合"真相源唯一"的原则，不需要改。

## 5. 分步执行顺序

1. **Kernel 层**：无新增（栈逻辑是纯 Shell 层的引用管理，不涉及需要 Kernel 侧纯函数改动的部分——backlog 折叠的纯计算逻辑本身`BacklogProjection.fs::projectBacklogFor`不变，只是调用方决定"要不要调它"的判断变了）。
2. **新增 `Shell/MessageTransformStack.fs`**，写三个 slot 类型 + get/set/pop/push 的 `RuntimeScope` 存取函数，每个函数体不超过 5 行。
3. **改 `Shell/MessageTransformHostEntry.fs`**：删除整套 Fingerprint 机制，`runHostMessagesTransform` 签名去掉 `_reviewReplayMode`/`_replayTexts` 两个参数，函数体改为直接调用新 pipeline（不再有"整体命中/整体重算"两分支，因为不再有整体缓存）。
4. **改 `Shell/MessageTransformPipeline.fs`**：`runMessageTransformPipeline` 内部按栈拆分成三段独立判断，CAPS 段查 `CapsSlot`，backlog 段查 `BacklogSlot`，栈顶段（nudge + parallel-hint）用 `TopSlot`。
5. **改 `Shell/MessageTransformCore.fs::checkAndInjectNudge`**：改造为读写 `TopSlot`，不再假设"消息列表已经不含旧 nudge"。
6. **改三个宿主 `MessageTransform.fs`**（Opencode/Mux/Omp）：删除每轮开头的 `stripSyntheticBySource messagesList` 无条件调用；删除 `replayTexts`/`extractTextsFromEncodedMessages` 相关死代码及其向 `runHostMessagesTransform` 的传参。
7. **删除死代码**：`Kernel/ReviewReplayPolicy.fs::reviewTaskFromTexts`、`Shell/ReviewReplaySync.fs::syncReviewFromTexts`（连同它们唯一的调用点，即 messagesTransform 里那条从未生效的死路径）。保留 `Kernel/LoopMessages.fs::inferReviewTaskFromTexts` 本体和它的独立单测（`tests/LoopMessagesTests.fs`），因为它是纯函数且有自己的测试价值，只是不再被 transform 钩子引用。
8. **调整测试**：`MessageTransformPolicyTests.fs`、`MessageSanitizationTests.fs`、`ParallelToolPromptTests.fs`、`CompactionTransformTests.fs`、`ContextBudget*Tests.fs` 里所有调用 `runHostMessagesTransform` 的地方去掉两个死参数；新增 `MessageTransformStackTests.fs` 验证"弹出数量精确匹配上一轮推入数量"“"caps 建过一次后引用不变"两个核心不变式；`ReviewReplaySyncTests.fs` 里依赖 `syncReviewFromTexts` 的用例删除或迁移到纯 `inferReviewTaskFromTexts` 单测（已存在，不用动）。
9. **验证**：`npm run build-and-test`，重点看 `IntegrationCapsSpecs.fs`（epoch 稳定性）、`tests/MessageTransformPolicyTests.fs::testTransformO1Cache`（O(1) 缓存语义要改成"栈操作次数为 0"而不是"pipelineRunCount 不增加"）不回归。

## 6. 不变式（验收标准）

1. 同一 session 连续两轮，若无任何 synthetic 状态变化，两轮的 encoded 数组里**前 N-0 条**（N = 数组长度）对象引用逐一 `ReferenceEquals` 成立（除了最后可能的 nudge/hint 位置，若两轮都不需要则那也不存在差异）。
2. CAPS 前缀一旦建立，进程存活期间任何后续轮次里，数组前 K 条（K = caps 消息条数）永远是同一份引用，不因 caps 文件内容变化重建（重启后自然重建一次）。
3. Backlog 折叠区间只有在 `foldedBacklog.Length` 真正增加时才会整体替换一次；不增加则复用上一轮引用。
4. context-budget-nudge / parallel-tool-synth 只操作数组最后 0~1 个元素，不触碰其余位置。
5. messagesTransform 钩子对 review 激活/关闭状态零读取、零写入；`.wanxiangshu.ndjson` 是唯一真相源，钩子完全不参与判定。
