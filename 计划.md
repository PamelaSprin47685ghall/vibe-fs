# message transform 钩子重构计划

## 0. 背景：当前 bug 面

当前排查的三类问题共享同一个根因——**transform 钩子每轮都做"整体重算"，而不是"增量维护"**：

1. `MessageTransformHostEntry.fs` 里的 `SessionTransformCache` 用深比对（逐元素 `obj.Equals`）+ 6 个字段的 `FingerprintMetadata`（Model/ContextLimit/TokenUsage/BacklogRevision/EventlogRevision/ReviewRound/ReviewFeedback/CapsVersion/...）做缓存判定。任何一个字段抖动（比如 token 估算数值波动、EventLog revision 递增）都会让整条缓存失效，触发全量 pipeline 重跑，产生全新的消息对象，破坏 provider 端的 prompt-cache 前缀命中。
2. `Messaging.stripSyntheticBySource` 在每次 transform 开头无条件扫描并剔除**所有** `Synthetic` 消息（caps、backlog projection、context-budget-nudge、parallel-tool-synth），然后管线重新计算、重新插入。哪怕只有栈顶的 nudge 需要换一条，也要把栈底稳定了几十轮的 caps 前缀一起摘掉再贴回去。
3. `messagesTransform` 钩子里通过 `extractTextsFromEncodedMessages` + `inferReviewTaskFromTexts` 从消息文本里正则解析 `task:` frontmatter 来推断 review 是否激活——这不是 transform 钩子该做的事，因为消息切片在 compaction 后可能看不到激活锚点，且真相源应该只有一个：`.wanxiangshu.ndjson`。

## 1. 关键事实修正：transform 钩子修改不持久

### 1.1 原文假设（已推翻）

原文假设："opencode 主机传给 transform 钩子的 `messages` 数组，其前缀本身就是 host 侧维护好的、跨轮次稳定的引用"，并据此设计了"真栈"方案——记住上一轮推了几个 synthetic 元素，本轮弹出那么多个，再 push 新的。

### 1.2 实际行为（源码验证）

OpenCode `prompt.ts` 的 `runLoop` 每次循环迭代顶部：

```typescript
// prompt.ts:1092
let msgs = yield* MessageV2.filterCompactedEffect(sessionID)
```

`filterCompactedEffect` → `stream(sessionID)` → `page()` 从**数据库**分页读取消息，每次产生全新 JS 对象。然后才调用 transform 钩子：

```typescript
// prompt.ts:1254
yield* plugin.trigger("experimental.chat.messages.transform", {}, { messages: msgs })
```

插件的 `messagesTransform` 用 `replaceArrayInPlace` 原地修改 `msgs`（清空 + 重填）。修改后的 `msgs` 直接传给 `MessageV2.toModelMessagesEffect` 发往 provider。**但修改从未写回数据库。** 下一轮迭代 `msgs` 重新从 DB 读取，上一轮注入的 synthetic 消息全部消失。

`ContextBudget.fs:91` 注释已承认此事实：`"Host strips synthetic nudge each transform round; reinject whenever pressure still holds."`

**结论**：transform 钩子的修改不持久。"真栈"的 pop 操作是一个空操作——上一轮推入的 synthetic 元素在本轮数组里根本不存在，无从弹出。

### 1.3 正确认识

- **无钩子时**，OpenCode 前缀天然稳定：同一 human turn 内 auto-continue 步骤间 DB 行不变（只有新 tool result 追加到末尾），前缀内容字节一致 → provider prompt-cache 命中。
- **有钩子时**，钩子每轮收到 fresh host 数组（DB 直读，无上一轮 synthetic），必须重新注入全部 synthetic 消息。
- **问题**：如果每轮重新创建 synthetic JS 对象（即使内容相同），对象引用不同 → 下游 `replaceArrayInPlace` 产出全新数组 → 无法保证 JSON 序列化字节一致。
- **解法**：在 RuntimeScope 中维护 synthetic 注入段的**状态**。状态不变时复用同一份 JS 对象引用；状态变化时重建。host 前缀天然稳定 + synthetic 段引用不变 → 整体 JSON 字节稳定 → prompt-cache 命中。

## 2. TransformState：synthetic 注入段的权威状态

### 2.1 状态层次与 SSOT

| 层级 | 持久性 | 角色 |
| :--- | :--- | :--- |
| NDJSON 事件日志 | 跨重启 | **durable SSOT**：backlog 内容、review 状态、续命租约、nudge 去重等（docs/05、docs/18） |
| `FallbackRuntimeState` / `ContextBudgetStore` / `TransformState` | 进程生命周期 | **进程状态**：从 NDJSON fold 重建（重启时），运行时更新 |
| 每轮 host 消息数组 | 瞬时（不持久） | **非状态**：每轮从 DB 重读，上一轮修改消失 |

`TransformState` 不是任何东西的"缓存"——它**就是** synthetic 注入段的进程状态。其内容 SSOT 关系：

| 段 | 内容 SSOT | TransformState 中的字段 | 何时更新 |
| :--- | :--- | :--- | :--- |
| CAPS | `CapsFormat` / `CapsPrelude`（docs/03） | `Caps: obj array option` | `None` → 构建一次 → `Some`（永不重建，重启自然丢失） |
| Backlog | NDJSON `work_backlog_committed`（docs/07） | `Backlog: { EventCount: int; Segment: obj array }` | `EventCount` 递增 → 重建投影段 |
| Top slot | `PromptFragments`（docs/10） | `Top: { Kind: TopSlotKind; Item: obj option }` | `Kind` 变化 → 重建或清除 |

### 2.2 设计原则

1. **不弹不改 host 前缀**：每轮收到 fresh host 数组，原样保留。
2. **纯计数判断是否重建**：整数/ADT 比较，不做内容比对、不算指纹、不做深比对。
3. **状态不变 → 同一份引用**：`ReferenceEquals` 成立 → JSON 字节一致。
4. **字符串状态全部升级为 ADT**：F# union case 的相等比较编译为 tag 整数比较，零字符串比较。
5. **保留文档记载的行为**：docs 描述的管线行为（7 阶段、同 source 替换不重复追加等）是正确的；本计划只改实现方法，不改行为契约。

### 2.3 与文档管线（docs/10）的对应

文档记载的 7 阶段管线：

| 阶段 | 文档描述 | 本计划调整 |
| :--- | :--- | :--- |
| 1. 剥离 synthetic | `stripSyntheticBySource` 无条件调用 | **删除**：修改不持久，host 数组中不存在上一轮 synthetic，strip 是空操作 |
| 2. Caps / 清理 | caps 前缀注入 + 消息净化 | `TransformState.Caps`（Option），建过一次即同一份引用 |
| 3. Backlog 投影 | 从事件 fold 投影，非历史 tool SSOT | `TransformState.Backlog`（`EventCount` 整数驱动） |
| 4. Review replay | 从消息文本推断 review 状态 | **删除死代码**：`_replayTexts` 参数从未被使用，review 状态走 NDJSON 事件 fold（docs/06） |
| 5. applyContextBudget | `classifyPressure` → 紧急 nudge 注入 | `TransformState.Top`，ADT 驱动。扩展现有 `ContextBudgetStore.StableSyntheticNudgeID` + `BudgetNudgeTrack`（ADT）到完整 JS 对象 |
| 6. tryInjectParallelToolPrompt | 单工具伪装并行提示 | `TransformState.Top`，与 budget nudge 互斥 |
| 7. Semble | investigator 断点注入 | **不在本计划范围**：Semble 操作 last assistant 的 parts，与三段状态的数组级操作正交 |

### 2.4 TransformState 类型

```fsharp
type TransformState = {
    Caps: obj array option
    Backlog: { EventCount: int; Segment: obj array }
    Top: { Kind: TopSlotKind; Item: obj option }
}
```

存在 `RuntimeScope` 里，key 用 `sessionID`，跟进程生命周期，不落盘。重启后自然为空 → 从 NDJSON 重建。

### 2.5 CAPS 状态

- `None` → 构建 caps 前缀，存为 `Some(prefix)`，`unshift` 一次。
- `Some` → 直接 `unshift` 同一份引用。
- 进程重启 → RuntimeScope 清空 → `None` → 自然重建一次。
- Compaction **不触发重建**：caps 内容是文件哈希/标签（`CapsFormat`），与对话内容无关。
- **不需要任何计数器**，`Option` 即可。与 docs/10 "Shell caps cache" 一致。

### 2.6 Backlog 状态

- 从 `EventLogStore.GetSessionStateSync(sessionID).EventCount` 读取当前事件计数（`Fold.fs:864`，每次 `applyEvent` 递增）。
- `EventCount = state.Backlog.EventCount` → 同一份 `Segment` 引用。
- `EventCount > state.Backlog.EventCount` → 用新 `foldedBacklog` 重建投影段，更新状态。
- Compaction 不写 NDJSON → `EventCount` 不变 → 同一份引用（与 docs/13 phase reset 语义一致）。
- Auto-continue 期间若有 `todowrite` → `work_backlog_committed` 事件 append → `EventCount` 递增 → 只重建 backlog 段。

### 2.7 Top slot 状态

**关键区分**（docs/06 + docs/13）：
- **context-budget-nudge**（docs/13）：transform 管线内的**同步注入**，由 `classifyPressure` 返回 `RequireTodoWriteEmergency` 驱动。消息内容固定（`PromptFragments` SSOT）。`ContextBudgetStore` 已有 `BudgetNudgeTrack` ADT（`Idle` / `EmergencySignaled`）。
- **async nudge**（docs/06）：`SessionIdle` 后的**异步** `session.prompt`，由 `NudgeRuntime` 编排。**不经 transform 钩子**，不在 Top slot 范围内。
- **parallel-tool-synth**（docs/10）：`tryInjectParallelToolPrompt` 条件注入，消息内容固定。

```fsharp
type TopSlotKind =
    | NoTop
    | BudgetNudgeTop    // BudgetNudgeTrack = EmergencySignaled 且未达 NudgeCount 上限
    | ParallelHintTop   // 最后一条 assistant 有且仅有 1 个 tool part
```

无 payload：budget nudge 文本固定，parallel hint 文本固定。ADT tag 即状态 key。

`computeTopSlotKind` 从 `ContextBudgetStore` + 当前消息列表读取决策，返回 ADT 值。状态比对用 F# 结构相等（tag 整数比较）。

Budget nudge 和 parallel-hint 互斥（pipeline 顺序已决定优先级，docs/10 阶段 5 在阶段 6 之前）。

**与现有代码的关系**：`ContextBudgetStore.StableSyntheticNudgeID` 已存 nudge ID 用于 `isSameEpisode` 判断（`MessageTransformCore.fs:198-210`）。本计划将状态从 ID 扩展到完整 JS 对象引用，避免每轮 `buildContextBudgetNudgeMessage` 重构。

### 2.8 transform 函数

```
transform(hostArray, state) → (newArray, newState)

1. CAPS（阶段 2）：
   match state.Caps with
   | Some prefix → unshift prefix         // 同一份引用
   | None → build, state.Caps ← Some, unshift

2. Backlog（阶段 3）：
   let count = EventLogStore.GetSessionStateSync(sid).EventCount
   if count = state.Backlog.EventCount → append state.Backlog.Segment
   else → rebuild, state.Backlog ← { count, newSegment }, append

3. Top slot（阶段 5+6）：
   let kind = computeTopSlotKind(contextBudgetStore, messages)
   if kind = state.Top.Kind →
       match state.Top.Item with
       | Some item → append item          // 同一份引用
       | None → ()                         // 不追加
   else →
       rebuild or clear, state.Top ← { kind, newItem }, append

4. replaceArrayInPlace(hostArray, finalArray)
5. return (finalArray, state)
```

**核心不变式**：当所有状态字段不变时，`finalArray` 中 synthetic 段与上一轮 `ReferenceEquals`。host 段内容天然稳定（DB 不变）。整体 JSON 字节一致 → prompt-cache 命中。

**行为保持**：docs/10 记载的"同 source 已存在则替换而非无限追加"行为不变——由于 host 数组不含上一轮 synthetic，"替换"自然退化为"追加"，效果等价。

## 3. 字符串状态升级为 ADT

现有代码中大量使用字符串做同步判断（`sessionOwner = "Fallback"`、`lease.Status = "requested"` 等）。全部替换为 ADT，消除字符串比较。

### 3.1 新增 ADT 类型

```fsharp
// 替代 sessionOwner: string（"None"/"Human"/"Fallback"/"Nudge"/"Compaction"）
type SessionOwner =
    | NoOwner
    | Human
    | Fallback
    | Nudge
    | Compaction

// 替代 PendingLease.Status: string 和 NudgeLease.Status: string
// （"requested"/"dispatch_started"/"dispatched"/"running"/"cancelled"）
type LeaseStatus =
    | Requested
    | DispatchStarted
    | Dispatched
    | Running
    | Cancelled

// 替代 finishContinuation 的 outcome: string（"failed"/"cancelled"/"settled"）
type ContinuationOutcome =
    | Failed
    | Cancelled
    | Settled
```

F# 对无数据 union case 的相等比较编译为 **tag 整数比较**。零字符串、零 hash、零内容比对。

### 3.2 ADT 相比 enum 的优势

| 方面 | enum | ADT |
| :--- | :--- | :--- |
| 比较成本 | 整数 | tag 整数（无数据 case）或 tag+payload |
| 携带数据 | 否 | 是（`BudgetNudgeTop` 未来可携带 ordinal） |
| 穷尽匹配 | 需手动 | 编译器强制 |
| 新增 case | 全部 `if/match` 静默漏过 | 编译器报错所有未处理位置 |
| F# 惯用法 | 不推荐 | 推荐 |

### 3.3 受影响的位置

| 现有（string） | 替换为（ADT） | 影响文件 |
| :--- | :--- | :--- |
| `sessionOwner: string` | `SessionOwner` | `FallbackRuntimeState.fs` 全部 owner 读写；`FallbackEventBridge.fs` 全部 owner 比较 |
| `PendingLease.Status: string` | `LeaseStatus` | `verifyLeaseWithStatus`、`TryTransitionPendingLease`、`executeContinuationIntent` |
| `NudgeLease.Status: string` | `LeaseStatus`（复用） | `TryTransitionPendingNudgeLease` |
| `finishContinuation` 的 `outcome: string` | `ContinuationOutcome` | `finishContinuation`、所有调用点 |
| `PendingLease.Owner: string` | `SessionOwner`（复用） | lease 构造与验证 |
| `NudgeLease.Owner: string` | `SessionOwner`（复用） | nudge lease 构造与验证 |

替换后，`verifyLeaseWithStatus` 中：

```fsharp
// 现在：
currentOwner = "Fallback"  && pending.Status = "requested"  // 字符串比较

// ADT 后：
currentOwner = Fallback    && pending.Status = Requested    // tag 整数比较
```

## 4. Auto-continue 与复杂时序

### 4.1 Auto-continue（OpenCode tool-call 循环）

OpenCode `runLoop` 每步：`status.busy` → `filterCompactedEffect`（DB 重读）→ transform 钩子 → provider → tool result 入 DB → 循环。只有循环退出才 `session.idle`（docs/11）。

transform 钩子被调用多次（每步一次），每次收到 fresh host 数组。

| 计数器/状态 | auto-continue 步骤间 | TransformState 行为 |
| :--- | :--- | :--- |
| `sessionGeneration` | 不变（同一 human turn） | — |
| `EventCount` | 可能变（tool 里有 todowrite） | 变 → backlog 重建；不变 → 同一份引用 |
| `BudgetNudgeTrack` | 可能变（token 增长触发 `EmergencySignaled`） | 变 → top slot 重建；不变 → 同一份引用 |
| `Caps` | 不变 | 同一份引用 |

**结果**：auto-continue 步骤间若无 todowrite 且未触发 budget nudge，所有 synthetic 段 `ReferenceEquals` 上一轮 → JSON 字节一致 → prompt-cache 跨步骤命中。有 todowrite → 只重建 backlog 段。触发 budget nudge → 只重建 top slot 段。caps 始终不变。

### 4.2 Compaction

compaction 发生时 `compactionOrdinal` 递增，`SessionOwner` 短暂为 `Compaction`，完成后回到 `NoOwner`。`sessionGeneration` 递增（docs/05）。host 消息被摘要替换。

| 段 | compaction 后 | 原因 |
| :--- | :--- | :--- |
| CAPS | **不变** | caps 内容是文件哈希/标签（`CapsFormat`），与对话无关 |
| Backlog | **不变**（`EventCount` 不变） | backlog 投影从 NDJSON 事件 fold 计算（docs/07），不从 host 消息计算；compaction 不写 NDJSON |
| Top slot | **可能重建** | `ContextBudgetStore` phase reset（docs/13），`BudgetNudgeTrack` 回到 `Idle` → `TopSlotKind` 从 `BudgetNudgeTop` 变为 `NoTop` → 重建 |

compaction 后 host 消息变了（摘要），但 synthetic 段状态不变 → 重新插入同一份引用。

### 4.3 Async nudge 派发（docs/06）

async nudge 是 `SessionIdle` 后的异步 `session.prompt`，**不经 transform 钩子**。nudge 派发时 `nudgeOrdinal` 递增，`SessionOwner` 变为 `Nudge`。

- **对 TransformState 无直接影响**：async nudge 创建的是真实 user 消息（DB 持久化），不是 synthetic 消息
- 下一轮 transform 收到的 host 数组包含这条 nudge 消息（作为 Native 消息），不影响 synthetic 段状态
- CAPS、Backlog、Top slot 均不受影响

### 4.4 Fallback continuation 派发（docs/12）

continuation 派发时 `continuationOrdinal` 递增，`SessionOwner` 变为 `Fallback`。ZWS prompt 作为真实 user 消息入 DB。

- **对 TransformState 无直接影响**：与 async nudge 类似，continuation 创建真实 user 消息
- CAPS、Backlog 不受影响
- Top slot：`BudgetNudgeTrack` 可能因 token 增长而变化 → 独立判断

### 4.5 新 human turn

`sessionGeneration` 递增，`SessionOwner` 变为 `Human`，pending lease/nudge lease 被取消。

- `ContextBudgetStore` phase reset（docs/13）：`BudgetNudgeTrack` 回到 `Idle` → `TopSlotKind` 变为 `NoTop` → 重建（清除）
- CAPS：不变
- Backlog：`EventCount` 不变 → 同一份引用；有新 todowrite → 重建

### 4.6 状态变化总览

```
                        auto-continue    compaction     async nudge   cont.派发     新human turn
──────────────────────────────────────────────────────────────────────────────────────────────────
sessionGeneration       不变             递增           不变          不变          递增
EventCount              可能递增         不变           不变          不变          可能递增
BudgetNudgeTrack        可能变           Idle(复位)     不变          不变          Idle(复位)
SessionOwner(ADT)       不变             Compaction    NoOwner→Nudge NoOwner→Fallback NoOwner→Human
                                        →NoOwner(完成后)

Caps                    不变             不变           不变          不变          不变
Backlog                 count不变→不变   不变           不变          不变          count不变→不变
Top                     kind不变→不变    可能重建       不受影响      不受影响      kind变→重建
```

**关键区别**：async nudge 和 fallback continuation 创建的是**真实 user 消息**（DB 持久化），不是 transform 钩子的 synthetic 注入。它们不影响 TransformState，只影响下一轮 host 数组的内容（多了一条 Native 消息）。

## 8. 不变式（验收标准）

1. 同一 session 连续两轮 transform，若无任何计数器/ADT 状态变化，synthetic 段的 JS 对象引用逐一 `ReferenceEquals` 成立。host 段内容天然稳定（DB 不变）。整体 JSON 字节一致。
2. CAPS 前缀一旦构建（`Some`），进程存活期间任何后续轮次里永远是同一份引用。重启后自然重建一次。compaction 不触发重建（caps 内容与对话无关）。
3. Backlog 折叠区间只有在 `SessionState.EventCount` 递增时才会整体替换；不递增则同一份引用。compaction 不写 NDJSON → `EventCount` 不变（与 docs/13 phase reset 语义一致）。
4. Top slot（budget-nudge / parallel-hint）由 `TopSlotKind` ADT 驱动。ADT 不变 → 同一份引用；ADT 变 → 重建。只操作数组最后 0~1 个元素。budget nudge 文本固定（`PromptFragments` SSOT），parallel hint 文本固定。
5. Auto-continue 步骤间：若无 todowrite 且未触发 budget nudge，所有计数器/ADT 不变 → 全部 synthetic 段同一份引用 → prompt-cache 跨步骤命中。
6. messagesTransform 钩子对 review 激活/关闭状态零读取、零写入（docs/06：review 状态走 NDJSON 事件 fold）。
7. `FallbackRuntimeState` 和 `FallbackEventBridge` 中不存在任何字符串字面量做状态比较。全部使用 ADT case。
8. docs/10 记载的管线行为保持不变：caps 注入、backlog 投影、context budget nudge、parallel hint 的**触发条件和注入内容**不变，只改**状态管理机制**。
9. 架构测试不回归：`UsesProjectionPolicy`、`noQuadraticListAppend`、`parallelToolPromptSSOTGuard` 等（docs/10）。
10. async nudge（docs/06）和 fallback continuation（docs/12）不受本计划影响：它们创建真实 user 消息（DB 持久化），不经 transform 钩子 synthetic 注入。
11. `TransformState` 是 synthetic 注入段的**唯一进程状态**——不存在第二份"缓存"或"投影"与之竞争。NDJSON 是 durable SSOT，`TransformState` 从中派生。
