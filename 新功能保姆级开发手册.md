# 项目文档自动维护功能保姆级开发手册

本文是给未来开发者直接照着实现的开发手册，不是讨论稿。这里记录已经定稿的产品语义、架构边界、文件格式、并发策略、Opencode 落点、测试路线和容易踩坑的地方。

目标功能：在用户项目的 `./wiki/` 目录维护一个事件溯源风格的 Q/A wiki。模型只在需要时看到 `id + q` 索引，通过 `fetch_wiki(id)` 取 `a`。有项目写入副作用的工作完成后，程序自动 fire-and-forget 拉起 `bookkeeper` 子智能体，把本轮有价值的项目知识写回 `wiki`。

## 0. 已定稿约束

这些约束不要在实现时自行改动：

- 先只实现 Opencode，不做 Mux。
- wiki 存在用户项目根目录的 `wiki/` 下。
- wiki 文件是 NDJSON。
- `snapshot.ndjson` 是周快照文件。
- 每天一个 `YYYY-MM-DD.ndjson`。
- 每个 NDJSON 文件第一行是文件头 header。
- wiki 知识条目只有 `{ id, q, a }`。
- `id` 是 4 位小写 hex，例如 `0a3f`。
- `manager` prelude 只注入 `id + q`，不注入 `a`。
- prelude 放进当前 synthetic `"你好"` 消息里，section 名为 `[项目背景和历史]`。
- 不伪装成 `read` tool，也不要用 synthetic tool call 表达 wiki。
- 查询工具叫 `fetch_wiki(id)`。
- `fetch_wiki(id)` 只返回 `a` 字符串。
- `fetch_wiki(id)` 读 session 启动时的 wiki snapshot cache，不实时读盘。
- 只有 prelude / `fetch_wiki` 是 session snapshot 语义。
- `bookkeeper` 是实时语义，每次任务启动时读取磁盘最新投影。
- 工作区副作用就是记账资格。
- 只读 bash / 只读 executor 通常是垃圾信息，不记账。
- `executor` 增加 `mode = "ro" | "rw"`，其中 `rw` 表示对项目本身有破坏性或写入副作用，只有 `rw` 记账。
- 不需要 `none | read_only | read_write` 三态。
- `bookkeeper` 是 fire-and-forget；主流程不等它。
- `bookkeeper` 通过后台无父 session 执行。
- 端口锁只阻塞后台 wiki job；锁不到就死等；其他人不等它。
- 进程内串行借鉴 `Actors`。
- 跨进程锁用端口锁，不用文件锁。
- `bookkeeper` 和 rewrite 使用同一个 agent、同一个提交工具，只是 prompt 不同。
- `bookkeeper` 提交工具的逻辑 schema 是数组：`[{ id?: string, q: string, a: string }]`。
- 输出空数组表示没有值得写入 wiki 的知识。
- 带 `id` 表示更新实时投影里的已有知识点。
- 不带 `id` 表示追加新知识，由 host 分配 4-hex id。
- 日内普通写入只追加，不改旧行。
- 每日 rewrite 是全量重写，相当于删除旧 entries 后重新追加 canonical entries。
- 除每日 rewrite 外，`id` 应保持稳定。普通 bookkeeper 更新同一知识点时复用 id，周 snapshot rewrite 也不要无理由重分配 surviving id。
- 周整理全量重写 `snapshot.ndjson`，然后删除已经并入 snapshot 的旧日文件。
- 周 snapshot 的截止是上周日 24:00 之前。
- 单条 `q` 只做乐观限长提醒，不做硬性拒绝。

## 1. 用户故事

### 1.1 manager 使用 wiki

当用户新开一个 session，系统读取当前项目 `wiki/`，构建一个 session 内不变的 wiki snapshot cache。

随后在发给 manager 的消息历史前面注入一个 synthetic `"你好"` 用户消息。这个消息里包含原有 caps 上下文，还新增：

```text
[项目背景和历史]

- 0a3f 项目使用 Fable 编译 F# 到 JS，插件入口在 Opencode/Plugin.fs
- b912 Magic Todo backlog 通过 todowrite 的 completedWorkReport 折叠上下文
- 7c01 reviewer loop 使用 return_reviewer 提交 verdict

需要某条历史答案时，调用 fetch_wiki(id)。
```

manager 不直接看到答案 `a`。当需要详情时调用：

```text
fetch_wiki({ id = "0a3f" })
```

返回：

```text
项目使用 Fable 将 F# 编译到 JS。Opencode 主插件入口是 src/Opencode/Plugin.fs，实际装配逻辑在 src/Opencode/PluginCore.fs。
```

### 1.2 自动记账

只要一个工作对用户项目有写入副作用，就在工作结束后由程序自动 fire-and-forget 拉起 `bookkeeper`。

`bookkeeper` 会看到：

- 当前磁盘 wiki 最新投影里的全部 `{ id, q, a }`。
- 本次工作输入 prompt。
- 本次工作输出 / 子任务结果。
- 本次 rw 工具调用摘要。
- 明确指令：只沉淀稳定、可复用的项目知识；不要记录流水账、临时日志、构建噪音。

`bookkeeper` 提交：

```json
[
  { "id": "0a3f", "q": "项目插件入口在哪里？", "a": "Opencode 主入口是 src/Opencode/Plugin.fs，装配逻辑在 src/Opencode/PluginCore.fs。" },
  { "q": "executor 的 rw 模式什么时候使用？", "a": "当 executor 命令会修改项目文件、生成项目产物或改变项目状态时使用 mode=rw。只读诊断使用 mode=ro。" }
]
```

host 负责把第二条补上新 `id` 后追加到今天的 NDJSON。

## 2. 磁盘格式

### 2.1 目录布局

```text
wiki/
  snapshot.ndjson
  2026-06-15.ndjson
  2026-06-16.ndjson
  2026-06-17.ndjson
  2026-06-18.ndjson
  2026-06-19.ndjson
```

约束：

- `snapshot.ndjson` 存放上一个周截止点前已经整理进快照的 canonical 知识。
- 本周日文件保留。
- 今天的日文件可能未 rewrite。
- 已并入 snapshot 的旧日文件必须删除。
- 文件缺失不是错误，按空文件处理。
- `wiki/` 不存在时按空 wiki 处理，首次写入时创建。

### 2.2 日文件 header

第一行必须是 header：

```json
{"type":"wiki_header","version":1,"kind":"day","date":"2026-06-19","rewritten":false}
```

字段：

- `type`: 固定 `wiki_header`。
- `version`: 当前固定 `1`。
- `kind`: 日文件固定 `day`。
- `date`: 文件日期，必须等于文件名日期。
- `rewritten`: 是否已经做过每日 rewrite。

日内普通 append 不修改 header。

每日 rewrite 后，目标日文件全量重写，并置：

```json
{"type":"wiki_header","version":1,"kind":"day","date":"2026-06-18","rewritten":true}
```

### 2.3 snapshot header

`snapshot.ndjson` 第一行：

```json
{"type":"wiki_header","version":1,"kind":"snapshot","through":"2026-06-14"}
```

字段：

- `type`: 固定 `wiki_header`。
- `version`: 当前固定 `1`。
- `kind`: 固定 `snapshot`。
- `through`: snapshot 已覆盖的最后日期。语义是该日期结束时，也就是该日期 24:00 前的知识已经并入 snapshot。

### 2.4 条目行

header 后每行一个 entry：

```json
{"id":"0a3f","q":"项目插件入口在哪里？","a":"Opencode 主入口是 src/Opencode/Plugin.fs，装配逻辑在 src/Opencode/PluginCore.fs。"}
```

字段：

- `id`: 4 位小写 hex，正则 `^[0-9a-f]{4}$`。
- `q`: 问题，给 manager prelude 展示。
- `a`: 答案，只能通过 `fetch_wiki(id)` 获取。

不要向 entry 添加 `createdAt`、`updatedAt`、`source`、`confidence`、`tags`。这些字段看似有用，第一版不要加。需求明确只有 `id/q/a`。

### 2.5 NDJSON 读写纪律

必须遵守：

- append 时只追加完整 JSON 行和末尾 `\n`。
- rewrite 时写临时文件，然后同目录 rename 覆盖目标文件。
- 模型永远不能直接写 header。
- 模型只提交 entries。
- host 负责 JSON 序列化，不能把模型返回的字符串直接拼进文件。
- 读取时如果中途某行 JSON parse 失败，停止读取该文件后续行，不要跳过坏行继续读。
- 最后一行为空是正常情况。
- 如果最后一行非空但 parse 失败，当作半写入行处理，忽略该行和后续内容。
- 文件头 parse 失败时，该文件整体视为不可用，不要读取后续 entry。

这条规则沿用事件日志前缀完整性原则：宁可少恢复一条，也不要恢复建立在坏前缀上的错误状态。

## 3. 投影语义

### 3.1 磁盘实时投影

`bookkeeper` 每次启动时读取磁盘最新投影。投影顺序：

```text
snapshot.ndjson
-> 按日期升序的 YYYY-MM-DD.ndjson
```

只读取 header 之后的 entry 行。

对相同 `id` 使用 latest-wins：后出现的 entry 覆盖前面的 `{ q, a }`。

伪代码：

```fsharp
let project files =
    files
    |> List.collect entriesAfterHeader
    |> List.fold
        (fun state entry -> Map.add entry.id entry state)
        Map.empty
```

这个实时投影用于：

- `bookkeeper` append 模式输入。
- 判断 bookkeeper 输出的 `id` 是否存在。
- 给无 `id` 输出分配不冲突的新 id。
- rewrite 输入。

### 3.2 session snapshot 投影

session 启动时读取一次磁盘投影，保存到内存 cache。

这个 cache 用于：

- manager prelude 生成 `[项目背景和历史]`。
- `fetch_wiki(id)` 查询答案。

session 生命周期内，这个 cache 不随着后台 bookkeeper 或 rewrite 更新。

原因：prelude 给过模型某个 `id`，`fetch_wiki(id)` 必须在本 session 内稳定可用。否则后台 daily rewrite 改了 id 后，manager 刚看到的 id 会马上失效。

### 3.3 两种语义并存

最终语义：

```text
manager prelude snapshot -> fetch_wiki reads same snapshot
rw work completes -> fire-and-forget bookkeeper reads latest disk -> append entries
daily rewrite -> full replace day file, may reassign ids for that day
weekly rewrite -> full replace snapshot, preserve surviving ids when possible, delete merged day files
```

注意：

- `fetch_wiki` 不代表磁盘最新 wiki。
- `bookkeeper` 不使用 session snapshot，它实时读盘。
- 用户从旧 session 复制一个 `id` 到新 session，不保证有效。

## 4. id 规则

### 4.1 格式

`id` 是 4 位小写 hex：

```text
0000
0001
0a3f
ffff
```

不要用大写。

不要加前缀。

不要用 UUID。

不要把 id 扩成 8 位。

### 4.2 分配

host 分配新 id，模型不分配。

算法：

```fsharp
let allocateId existingIds randomHex4 =
    let rec loop attempts =
        if attempts > 65536 then Error "wiki id space exhausted"
        else
            let candidate = randomHex4()
            if Set.contains candidate existingIds then loop (attempts + 1)
            else Ok candidate
    loop 0
```

要求：

- 分配前读取当前实时投影的所有 id。
- 不能和当前投影冲突。
- 如果 65536 空间耗尽，返回明确错误；第一版不扩位。

### 4.3 稳定性

普通路径：

- bookkeeper 带已有 `id`：追加同 `id` 的新 entry，latest-wins 后相当于覆写。
- bookkeeper 不带 `id`：host 分配新 id 后追加。
- 除每日 rewrite 外，不要无理由重分配已有知识的 id。

每日 rewrite：

- 是唯一允许大幅改变当天 entry id 的常规流程。
- 语义上相当于删掉当天文件所有 entries，再写入整理后的 canonical entries。
- prelude / fetch 是 session snapshot，所以活跃 session 不受影响。

周 snapshot rewrite：

- 物理上全量重写 `snapshot.ndjson`。
- 语义上不应无理由重分配 surviving id。
- 多条知识合并时，选择一个 surviving id。
- 已过时或被合并掉的 id 可以消失。

## 5. bookkeeper 设计

### 5.1 agent 定义

新增内建 subagent：`bookkeeper`。

它不是用户可直接调用的普通工具型 agent，而是 host 自动拉起的后台 agent。

在 `src/Opencode/PluginCore.fs` 的 `builtinAgentSpecs` 中增加：

```fsharp
{ name = "bookkeeper"; defaultMode = "subagent"; systemPrompt = ""; defaultMcps = [||] }
```

不要给它浏览器 MCP。

不要让它有文件写权限。

不要让它能调用 `coder` / `investigator` / `executor` / `fetch_wiki`。

它唯一需要的工具是 wiki 提交工具。

### 5.2 提交工具

建议工具名：`submit_wiki`。

逻辑 schema 是 top-level array：

```ts
type WikiDraft = {
  id?: string
  q: string
  a: string
}

type SubmitWikiInput = WikiDraft[]
```

如果 Opencode SDK 对 top-level array 支持有问题，再退一步包成：

```ts
type SubmitWikiInput = {
  entries: WikiDraft[]
}
```

但文档语义仍然视为“提交一个 entries 数组”。不要额外引入 `mode`、`target`、`date` 让模型选择写入方式。写入方式由 host 根据后台 job 上下文决定。

### 5.3 submit_wiki 如何知道 job 上下文

不要让模型提交 `callId` 或 `mode`。

推荐做法：host 创建 bookkeeper session 后，拿到 `childID`，注册：

```fsharp
type WikiJobKind =
    | AppendAfterWork
    | DailyRewrite of date: string
    | WeeklyRewrite of throughDate: string

type WikiJobContext =
    { workspaceRoot: string
      sessionId: string
      kind: WikiJobKind }
```

用 `childID -> WikiJobContext` 放进 `WikiJobStore`。

`submit_wiki` 执行时，从 tool context 里取当前 `sessionID`，查 `WikiJobStore`，决定如何处理 entries。

执行结束后删除 job context。

这样 schema 可以保持纯净，模型只负责交 entries。

### 5.4 append prompt

append 模式 prompt 必须包含：

- 当前实时 wiki 投影的全部 `{ id, q, a }`。
- 本次工作输入。
- 本次工作输出。
- 本次 rw 工具摘要。
- 明确输出规则。

建议 prompt：

```text
You are the project wiki bookkeeper.

You maintain a compact Q/A wiki for this project. The wiki stores durable project knowledge, not a transcript.

Existing wiki entries are listed below. If new work updates an existing fact, reuse its id. If new work introduces a durable fact not covered by existing entries, omit id and the host will assign one. If there is no durable project knowledge to record, submit an empty array.

Rules:
- Submit only stable, reusable project knowledge.
- Do not record command noise, temporary errors, progress updates, or generic coding advice.
- Do not mention this bookkeeper mechanism.
- Keep q short and useful for a future manager prelude.
- Keep a precise enough that fetch_wiki(id) answers the question without needing this transcript.
- Prefer updating an existing id over creating a near-duplicate.
- You must call submit_wiki exactly once.

=== Existing Wiki ===
{json array of id/q/a}

=== Work Input ===
{prompt or direct-turn input}

=== Work Output ===
{assistant/subagent result}

=== RW Tool Summary ===
{structured summary of project-writing tools, if any}
```

### 5.5 daily rewrite prompt

daily rewrite 使用同一个 `bookkeeper` agent 和同一个 `submit_wiki` 工具。

prompt 变成：

```text
You are rewriting one day of the project wiki.

The target day file is being fully replaced. Merge similar questions, remove obsolete entries, and keep only durable canonical knowledge from the target day. This is not an append job.

Rules:
- Submit the complete replacement entries for the target day.
- It is valid to submit an empty array.
- Do not preserve duplicate entries.
- Do not preserve outdated facts when newer facts supersede them.
- You must call submit_wiki exactly once.

=== Snapshot And Context ===
{current projected wiki, useful context only}

=== Target Day Entries ===
{entries from YYYY-MM-DD.ndjson}
```

host 对 daily rewrite 的处理：

- 忽略 target day 的旧 entries。
- 全量写 target day 文件。
- header `rewritten` 置 `true`。
- 可以重新分配 target day entries 的 id。
- 写入必须 temp + rename。

### 5.6 weekly rewrite prompt

weekly rewrite 同样使用 `bookkeeper`。

prompt：

```text
You are rewriting the project wiki snapshot.

The snapshot file is being fully replaced up to the given Sunday 24:00 cutoff. Merge the previous snapshot and all completed day files before or on the cutoff. Keep durable canonical knowledge only.

Rules:
- Submit the complete replacement entries for snapshot.ndjson.
- Preserve ids for surviving entries when possible.
- If several entries merge, choose one surviving id.
- Omit id only for genuinely new synthesized knowledge without a source id.
- Remove obsolete knowledge superseded by newer entries.
- You must call submit_wiki exactly once.

=== Previous Snapshot ===
{snapshot entries}

=== Day Files Through Cutoff ===
{rewritten day entries through cutoff}
```

host 对 weekly rewrite 的处理：

- 全量重写 `snapshot.ndjson`。
- `through` 更新到上周日日期。
- 对无 id entries 分配新 4-hex。
- 对带 id entries 校验 id 格式和去重。
- rename 成功后删除已并入 snapshot 的旧日文件。

## 6. 记账资格

最终规则：工作区副作用就是记账资格。

### 6.1 需要记账

- `coder` subagent。
- `executor(mode="rw")`。
- 直接 `apply_patch`。
- 直接 `write`。
- 直接 `edit`。
- 直接 `patch`。
- `file_edit_replace_string`。
- `file_edit_insert`。
- 未来任何修改用户项目文件的工具。

### 6.2 不记账

- `investigator`。
- `meditator`。
- `browser`。
- `executor(mode="ro")`。
- `fuzzy_find`。
- `fuzzy_grep`。
- `read`。
- `websearch`。
- `webfetch`。
- `reviewer`。
- `bookkeeper` 自己。
- executor summary 子 agent。
- websearch summary 子 agent。
- pre-review reviewer。
- Magic Todo。
- Nudge。
- Caps 注入。
- `fetch_wiki`。

### 6.3 executor mode

`executor` 增加必填参数：

```ts
mode: "ro" | "rw"
```

语义：

- `ro`: 命令只读取、诊断、打印、搜索，不改变项目文件或项目状态。
- `rw`: 命令会写文件、生成产物、运行 formatter、安装依赖、修改锁文件、迁移数据、或以其他方式改变项目状态。

`ro` 不记账。

`rw` 记账。

不要试图从 shell 文本自动推断模式。让调用方显式声明。

第一版可以只在 Opencode executor schema 里加 mode；Mux 之后再补。

## 7. Opencode 模块落点

### 7.1 新增 Kernel 模块

新增：`src/Kernel/Wiki.fs`

职责：纯逻辑。

建议类型：

```fsharp
module VibeFs.Kernel.Wiki

type WikiId = private WikiId of string

type WikiEntry =
    { id: WikiId
      q: string
      a: string }

type WikiDraft =
    { id: string option
      q: string
      a: string }

type WikiHeader =
    | DayHeader of date: string * rewritten: bool
    | SnapshotHeader of through: string option

type WikiFile =
    { header: WikiHeader
      entries: WikiEntry list }

type WikiProjection = Map<WikiId, WikiEntry>
```

建议函数：

```fsharp
val tryParseId : string -> WikiId option
val idValue : WikiId -> string
val parseHeaderLine : string -> Result<WikiHeader, string>
val renderHeader : WikiHeader -> string
val parseEntryLine : string -> Result<WikiEntry, string>
val renderEntry : WikiEntry -> string
val parseNdjson : fileName:string -> text:string -> Result<WikiFile, string>
val renderNdjson : WikiHeader -> WikiEntry list -> string
val projectLatestWins : WikiFile list -> WikiProjection
val preludeLines : WikiProjection -> string list
val buildPreludeSection : WikiProjection -> string option
val applyDrafts : allocate:(Set<string> -> string) -> WikiProjection -> WikiDraft list -> Result<WikiEntry list, string>
val validateDraft : WikiDraft -> Result<WikiDraft, string>
```

注意：

- `Kernel/Wiki.fs` 不读文件。
- 不知道 Opencode。
- 不拉 subagent。
- 不碰端口锁。

### 7.2 新增 Shell 模块

新增：`src/Shell/WikiFiles.fs`

职责：文件系统 IO。

建议函数：

```fsharp
val wikiDir : workspaceRoot:string -> string
val snapshotPath : workspaceRoot:string -> string
val dayPath : workspaceRoot:string -> date:string -> string
val ensureWikiDir : workspaceRoot:string -> JS.Promise<unit>
val readWikiFiles : workspaceRoot:string -> JS.Promise<WikiFile list>
val readProjection : workspaceRoot:string -> JS.Promise<WikiProjection>
val ensureTodayFile : workspaceRoot:string -> today:string -> JS.Promise<unit>
val appendEntries : workspaceRoot:string -> today:string -> WikiEntry list -> JS.Promise<unit>
val rewriteDay : workspaceRoot:string -> date:string -> entries:WikiEntry list -> JS.Promise<unit>
val rewriteSnapshot : workspaceRoot:string -> through:string -> entries:WikiEntry list -> JS.Promise<unit>
val deleteDayFilesThrough : workspaceRoot:string -> through:string -> JS.Promise<unit>
val listDayFiles : workspaceRoot:string -> JS.Promise<string list>
```

新增：`src/Shell/WikiPortLock.fs`

职责：端口锁。

建议函数：

```fsharp
val lockPortForPath : string -> int
val withWikiPortLock : workspaceRoot:string -> Async<'a> -> Async<'a>
```

端口锁算法：

- 用 workspace root 或 wiki dir 绝对路径做 hash。
- 映射到高端端口范围，例如 `49152 + hash % 16384`。
- 用 Node `net.createServer()` 绑定 `127.0.0.1:port`。
- 绑定成功代表拿锁。
- 绑定失败就 `Async.Sleep 1000` 后重试，永远等。
- job 完成后 close server。

注意：

- 这个锁只保护后台 wiki job。
- 主流程不等这个锁。
- 如果端口被无关进程占用，bookkeeper 会死等；这是已接受语义。

## 9. 直接工具路径的 turn 聚合

直接工具路径不能每个工具调用都拉一次 bookkeeper，否则会产生大量垃圾。

推荐 per-session 聚合：

```fsharp
type WikiToolRecord =
    { tool: string
      inputSummary: string
      outputSummary: string
      filePaths: string list }

type WikiTurnRecord =
    { sessionID: string
      mutable userText: string
      mutable assistantText: string
      mutable rwTools: WikiToolRecord list
      mutable dirty: bool }
```

### 9.1 捕获用户输入

在 `chat.message` 或 event 里捕获用户消息文本。

如果 hook shape 不稳定，第一版可以只记录：

- sessionID
- rw tool records
- assistant final text

但尽量捕获用户输入，因为 bookkeeper 需要知道“为什么改”。

### 9.2 捕获 rw 工具

`tool.execute.after` 标记：

```fsharp
turn.dirty <- true
turn.rwTools <- record :: turn.rwTools
```

工具摘要不要放完整大输出。建议：

- tool 名称。
- 文件路径。
- 输入里的关键字段。
- 输出前 4KB。
- syntax check 结果。

### 9.3 捕获 assistant 完成

在 `event` hook 中参考 `NudgeState.isCompletedAssistantMessage`。

当 assistant 完成：

```fsharp
if turn.dirty then
    StartBookkeeperAppend(turn.userText, turn.assistantText, formatRwTools turn.rwTools)
    clear turn
```

如果拿不到 assistant text，仍然可以用 rw tool summary 拉 bookkeeper。不要阻塞主流程。

## 10. 后台 job 与并发

### 10.1 进程内 actor

借鉴 `src/Opencode/Actors.fs` 的 `ExecutorActor`。

新增 `WikiActor`：

```fsharp
type WikiWork =
    { workspaceRoot: string
      work: unit -> Async<unit> }

type WikiActor() =
    member Post(workspaceRoot: string, work: unit -> Async<unit>) : unit
```

每个 `workspaceRoot` 一个 `MailboxProcessor`。

所有 append/rewrite/delete 都经 actor。

### 10.2 跨进程端口锁

actor 内每个 job 执行：

```fsharp
withWikiPortLock workspaceRoot (async {
    do! runMaintenanceIfDue workspaceRoot
    do! runActualWikiJob workspaceRoot
})
```

锁不到就死等。

由于 actor 是后台 fire-and-forget，死等不会影响主流程。

### 10.3 惰性维护

不要后台常驻 timer。

在这些时机触发维护检查：

- session snapshot 初始化后，后台触发。
- append bookkeeper job 开始前，后台触发。
- `fetch_wiki` 不等待维护，但可以顺手触发一次后台维护检查。

维护检查只决定是否启动后台 bookkeeper rewrite session，不阻塞读路径。

### 10.4 每日 rewrite due 条件

当前日期是 `today`。

找到所有 `YYYY-MM-DD.ndjson`：

- `date < today`
- header `kind = day`
- header `rewritten = false`

按日期升序逐个调 daily rewrite。

### 10.5 每周 rewrite due 条件

计算上周日日期 `lastSunday`。

读取 `snapshot.ndjson` header 的 `through`。

如果 `through >= lastSunday`，不需要周整理。

否则：

- 确保 `through < day <= lastSunday` 的 day files 已 daily rewritten。
- daily rewrite 未完成时，先排 daily rewrite。
- daily 都完成后，启动 weekly rewrite。
- weekly rewrite 成功后删除 `date <= lastSunday` 的 day files。

## 11. submit_wiki 执行语义

### 11.1 append after work

输入 drafts：

```json
[
  { "id": "0a3f", "q": "...", "a": "..." },
  { "q": "...", "a": "..." }
]
```

host：

1. 读取当前实时投影。
2. 校验每个 draft。
3. 对带 id draft：
   - id 格式合法且存在于实时投影：保留 id。
   - id 不存在：当作新知识，忽略模型 id，重新分配 id。
4. 对无 id draft：分配新 id。
5. 确保今天 day 文件存在，header `rewritten=false`。
6. 追加 entries 到今天文件。
7. 返回简短成功信息。

不要因为一个 draft id 不存在就整个 job 失败。bookkeeper 看到的是实时投影，但并发 rewrite 可能让 id 消失；把它当新知识即可。

### 11.2 daily rewrite

host：

1. 取 job context 里的 target date。
2. 校验 drafts。
3. 对所有 drafts 分配 entries。
4. daily rewrite 是允许重新分配 target day id 的地方。
5. 全量写目标 day 文件，header `rewritten=true`。

### 11.3 weekly rewrite

host：

1. 取 job context 里的 `throughDate`。
2. 校验 drafts。
3. 带合法 id 的 drafts 尽量保留。
4. 无 id 或冲突 id 分配新 id。
5. 全量写 `snapshot.ndjson`。
6. rename 成功后删除已并入 day files。

## 12. fetch_wiki 工具

### 12.1 schema

```ts
type FetchWikiInput = {
  id: string
}
```

### 12.2 执行

执行时：

1. 从 tool context 取 `sessionID`。
2. 用 `sessionID` 取 session snapshot cache。
3. 校验 id 是 4-hex。
4. 查 cache。
5. 找到则返回 `a`。
6. 找不到返回：`Wiki entry not found in this session snapshot: <id>`。

不要实时读盘。

不要做 fuzzy search。

不要返回 q。

不要返回 JSON。

只返回答案文本或错误文本。

### 12.3 tool 描述

描述必须说清楚 snapshot 语义：

```text
Fetch the answer for a project wiki id from this session's wiki snapshot. The manager prelude lists available ids and questions. This tool returns only the answer text. It does not read the latest disk wiki.
```

## 13. manager prelude 细节

### 13.1 section 格式

如果 wiki snapshot 为空，不生成 section。

非空：

```text
[项目背景和历史]

- 0a3f 项目插件入口在哪里？
- b912 Magic Todo backlog 如何保存？
- 7c01 review loop 如何提交 verdict？

需要某条历史答案时，调用 fetch_wiki(id)。
```

每行格式：

```text
- <id> <q>
```

不要显示 `a`。

不要显示 JSON。

不要超过必要说明。

### 13.2 q 长度

单条 q 乐观限长。

建议在 prelude builder 里显示时截断到 160 字符左右，但不要拒绝写入。

截断只影响 prelude 展示，不改磁盘。

### 13.3 排序

排序建议：

- 按投影中 entry 最后出现顺序。
- 或按 id 字典序。

更推荐最后出现顺序，因为越新的知识越可能有用。

如果实现 `Map` 丢失顺序，可以在 `WikiProjection` 里保存 ordered list，而不是纯 `Map`。

第一版可用 id 字典序，简单稳定。

## 15. 错误处理原则

该功能是辅助记忆，不应该破坏主流程。

### 15.1 主流程不失败

以下失败都不能让用户请求失败：

- bookkeeper session 创建失败。
- bookkeeper 没调用 `submit_wiki`。
- bookkeeper 输出空。
- wiki 目录不可写。
- 端口锁长期拿不到。
- rewrite 失败。
- append 失败。

这些最多在后台 job 内静默结束或返回给后台 session。

### 15.2 fetch_wiki 要明确失败

`fetch_wiki` 是用户可见工具，失败要返回明确文本：

- id 格式错误：`Invalid wiki id: <id>`。
- id 不在 session snapshot：`Wiki entry not found in this session snapshot: <id>`。
- session snapshot 不存在：尝试初始化一次，仍失败则返回 `Wiki snapshot unavailable for this session.`。

### 15.3 文件损坏

读文件时：

- header 坏：跳过整个文件。
- entry 行坏：停止读取该文件后续行。
- 不要修复旧文件。
- 后续 rewrite 可自然修复。

实现惰性维护。

验收：

- 旧日文件 rewritten=false 时启动 daily rewrite 后台 job。
- 周 cutoff 到达时启动 weekly rewrite 后台 job。
- rewrite 成功后旧 day files 删除。
- 主流程不等待维护。

## 18. 容易踩坑

### 18.1 不要让 fetch_wiki 实时读盘

这是最容易错的点。

如果 `fetch_wiki` 实时读盘，后台 daily rewrite 一旦改 id，manager prelude 里的 id 立即失效。

正确做法：session 启动时 cache，fetch 永远查 cache。

### 18.2 不要给 manager 注入答案

prelude 只能是 `id + q`。

把 `a` 放进 prelude 会让上下文失控，也会让 `fetch_wiki` 失去意义。

### 18.3 不要让 bookkeeper 看到过少上下文

bookkeeper 必须看到所有实时 `q+a`，否则它无法判断是覆写旧 id 还是追加新知识。

只给它本次工作输出会造成大量重复问题。

### 18.4 不要把只读命令记账

只读 bash / grep / rg / ls / test 输出通常是临时诊断信息。

只有 `executor(mode="rw")` 记账。

### 18.5 不要把每个工具调用都写 wiki

直接工具路径要按 turn 聚合。

一个 assistant turn 内多次 file edit，只拉一次 bookkeeper。

### 18.6 不要让 bookkeeper 有写文件工具

bookkeeper 只能调用 `submit_wiki`。

磁盘写入必须由 host 实现。

### 18.7 不要把 rewrite 做成 append

daily rewrite 和 weekly rewrite 都是全量重写目标文件。

append 只用于普通 bookkeeper after-work。

### 18.8 不要把端口锁用在读路径

端口锁只保护后台写 job。

session snapshot 读取和 fetch 不等端口锁。

### 18.9 不要让后台失败影响主任务

wiki 是辅助记忆。bookkeeper 或 rewrite 失败不能让用户当前任务失败。

## 20. 最小可用版本边界

如果必须压缩第一版，最小闭环是：

1. `Kernel/Wiki.fs`。
2. `Shell/WikiFiles.fs`。
3. manager prelude snapshot。
4. `fetch_wiki(id)`。
5. coder 完成后 fire-and-forget bookkeeper append。
6. executor `mode=rw` 后 append。

可以暂缓：

- 直接 rw 工具 turn 聚合。
- daily rewrite。
- weekly rewrite。

但既然本需求明确要求“直接做到直接工具路径”，正式实现时不要把直接工具路径遗漏。

## 21. 最终验收标准

功能完成时必须满足：

- 新 session 的 manager 能在 synthetic `"你好"` 消息里看到 `[项目背景和历史]`。
- 该 section 只含 `id + q`，不含任何 `a`。
- manager 能调用 `fetch_wiki(id)` 取回该 session snapshot 中的 `a`。
- 修改磁盘 wiki 后，同一 session 的 `fetch_wiki(id)` 仍返回旧 snapshot 的答案。
- 新 session 能看到磁盘最新 wiki。
- coder 完成后会 fire-and-forget 拉起无父 bookkeeper session。
- `executor(mode="rw")` 会触发 bookkeeper。
- `executor(mode="ro")` 不触发 bookkeeper。
- 直接文件写工具在 assistant turn 结束后触发一次 bookkeeper。
- bookkeeper 只拥有 `submit_wiki`。
- `submit_wiki` 能 append 0-N 条。
- 无 id entry 会被 host 分配 4-hex id。
- 带已有 id entry 会以同 id 追加，latest-wins 后覆盖。
- day 文件第一行 header 正确。
- daily rewrite 全量替换 day 文件并设置 `rewritten=true`。
- weekly rewrite 全量替换 snapshot 并删除已并入 day files。
- 所有 wiki 写 job 经进程内 actor 和端口锁。
- bookkeeper / rewrite 等后台失败不影响主流程。
- `pnpm build && pnpm test` 通过。
