# mux vibe-fs dedup 不工作 — 调查进度

## 症状
连续读同一文件两次,第二次仍返回原文,未被替换成 `[No Change Since Previous Read/Write]`。

## 结论:根因已定位(非去重算法本身的 bug)

去重算法是对的,编译产物也是新的,不是这两处问题:
- `vibe-fs/src/Kernel/Dedup.fs` `deduplicate`:精确匹配 + 长超集匹配,逻辑正确。
- `vibe-fs/src/Mux/Dedup.fs` `deduplicateReadOutputsWithSeen`:匹配 `type="dynamic-tool"` /
  `toolName ∈ {read,file_read}` / `state="output-available"` / `output`(字符串或 `{content}`),
  与生产持久化形状(`mux/src/common/orpc/schemas/message.ts` 的 `DynamicToolPartAvailableSchema`)完全一致。
- 编译产物 `vibe-fs/build/src/Mux/Dedup.js`(Jun 15 08:03)与源码一致,**不是 stale**。

### 真正的根因:去重时机错位(时序 bug)

1. 去重只在 `mux/src/node/services/messagePipeline.ts:125` 的 `prepareMessagesForProvider` 里跑一次,
   而该函数在 `streamText` **启动前**只执行一次(`aiService.ts:2138`)。
2. Agent 在**同一轮**里重复读文件,是发生在 `streamText` 内部的**多步工具循环**里。
   这些 step 只通过 `streamManager.ts:1430` 的 `prepareStep` 回调暴露,
   而 `prepareStep` **从不调用去重**(只做 workflow 剥离 + 媒体抽取)。
3. 因此:同一轮内的第 2 次 read,其 tool-result 在去重逻辑跑完之后才被 SDK 追加进消息流,
   永远不会被去重 → 第二次返回原文。**与用户症状完全吻合。**

### 次要问题:seed 自重叠(跨轮会过度去重)
- `aiService.ts`:`conversationHistory: messages` 与喂给 `prepareProviderRequestMessages(messages,...)`
  的是**同一个数组引用**(agentSession.ts:3531 两个字段传同一份)。
- `messagePipeline.ts:125` 用 `collectReadOutputs(conversationHistory)` 做 seed,
  而被去重的 window 是该 history 的**子集** → 跨轮场景下 window 内首次出现也会被误标记(过度去重)。
- 当前没暴雷,是因为同一轮内根本没走到这一步(被上面的时序 bug 抢先了)。修复时序后要一并修这个 seed 逻辑。

## tool-result 在 ModelMessage 里的运行时形状(prepareStep 看到的)
字符串结果(如 read):
```
{ type: "tool-result", toolCallId, toolName: "read",
  output: { type: "text", value: "<文件内容>" } }
```
非字符串(对象)结果:`output: { type: "json", value: {...} }`
`role: "tool"` 消息的 `content` 是 `Array<ToolResultPart>`。
注意:这与 MuxMessage 持久化形状(`part.output` 直接是字符串/`{content}`)**不同**,
要在 `prepareStep` 去重需新增一个针对 ModelMessage 形状的提取/写回路径(读 `output.value`)。
