# mux host 徒手测试问题修复方案

## 结论

这份徒手测试报告有两类信息混在一起：

1. 真缺陷：当前 `head` 仍会失败，必须修。
2. 版本漂移 / 认知偏差：报告所测二进制比当前源码旧，或把“顶层 helper 已存在”误当成“真实 mux 链路已接线”，不能按表象误判。

先做一次基线校准，再动代码。否则极易把已经修好的点再修一遍，或漏掉“源码里有 helper，但真实链路没接上”的真缺陷。

## 基线校准

先重新构建并重新安装当前 `vibe-fs` + `mux`，再复跑徒手测试。原因：当前源码与报告至少有 4 处不一致，另有 1 处是“helper 已存在，但真实链路未生效”的潜伏缺陷。

已在当前 `vibe-fs` 源码存在的能力：

- `read` 目录回退：`src/Mux/HostTools.fs`
- `fuzzy_grep` 暴露：`src/Mux/Plugin.fs`、`src/Mux/HostTools.fs`
- `fetch_wiki` / `return_bookkeeper` 暴露：`src/Mux/Plugin.fs`
- `/loop` / `/loop-review` slash command：`src/Mux/SlashCommands.fs`
- plugin command 接到 mux orpc：`../mux/src/node/services/vibeFsBinding.ts`、`../mux/src/node/orpc/router.ts`

因此第一步不是修代码，而是确认手测二进制是否落后于源码。

## 问题分层

### A．确认是真缺陷，必须修

#### 1．`websearch` 摘要子代理 agentId 错配

现象：

- 报错 `Task.create: unknown agentId (executor). Runnable agentIds: desktop, exec, explore`

根因：

- `src/Mux/HostTools.fs` 把 Mux 子代理硬编码成 `"executor"`
- 但 Mux 当前真正可运行的 agentId 是 `"exec"`

受影响点不止一处：

- `summarizeWhenNeeded` 大输出摘要：`src/Mux/HostTools.fs`
- `websearchTool` 搜索摘要：`src/Mux/HostTools.fs`

最小修复：

- 统一把 Mux 内部委派到执行型子代理的 agentId 改成 `"exec"`
- 不要只修 `websearch`，否则大输出 `executor` 摘要路径仍会潜伏报错

建议改动文件：

- `src/Mux/HostTools.fs`

必须补的测试：

- 新增集成测试，断言 `websearch` 摘要委派使用 `exec`
- 新增集成测试，断言 `executor` 大输出摘要也使用 `exec`

验收：

- `websearch` 在有结果且触发摘要时返回总结，不再抛 unknown agentId
- 大输出 `executor` 也能正常摘要

#### 2．`todo_write` 只接了原生 todo，没接 Magic Todo backlog

现象：

- schema 缺 `completedWorkReport`
- todo item 缺 `priority`
- append-only backlog 语义不可用

根因：

- `../mux/src/common/utils/tools/toolDefinitions.ts` 的原生 `todo_write` 仍是 mux 自己的老 schema
- `../mux/src/node/services/tools/todo.ts` 也只持久化 `todos`
- `vibe-fs` 的 Mux 适配层当前只给 `todo_write` 加了 nudge wrapper，未接入 `MagicTodo` 的报告捕获、backlog 缓存、消息投影链
- `src/Mux/MessageTransform.fs` 当前只做 review / caps / wiki prelude，不做 Magic backlog projection

这不是单点 schema 缺字段，而是整条语义链没接通。

最小正确方案：

1. 继续优先在 `vibe-fs` 侧完成，不优先改 mux 核心。
2. 通过现有 wrapper 覆盖 `todo_write` 的 description + schema，使 LLM 看到 `completedWorkReport`、`priority` 与 append-only 语义。
3. wrapper 执行时剥离 `completedWorkReport`，避免把未知字段直接传给 mux 原生 `todo_write`。
4. 把剥离出的 `completedWorkReport` 存入 `MagicSessionStore`。
5. 在 `src/Mux/MessageTransform.fs` 引入与 Opencode 类似的 Magic backlog 重建与投影。
6. host 语义应按 Opencode 风格处理，而不是 Mimocode 风格，因为 mux 暴露的是 `todo_write`，不是 `task`。

建议改动文件：

- `src/Mux/Wrappers.fs`
- `src/Mux/MessageTransform.fs`
- `src/Shell/MagicSessionStore.fs`（如需复用现有存储即可不动）
- `src/Kernel/MagicTodo.fs`（仅当现有 schema 复用不够时）
- `../mux/src/node/services/vibeFsBinding.ts` 一般不用再改，现有 wrapper 合并顺序已支持 schema 覆盖

不建议优先改的文件：

- `../mux/src/common/utils/tools/toolDefinitions.ts`
- `../mux/src/node/services/tools/todo.ts`

原因：

- 这些属于 mux 核心原生 todo 工具；按仓库约束，应优先把能力放在本仓库适配层，而不是侵入 mux 本体

必须补的测试：

- `todo_write` wrapper 暴露 schema 含 `completedWorkReport`
- `todo_write` wrapper 暴露 todo item 含 `priority`
- execute 前能剥离 `completedWorkReport`，但仍捕获到 `MagicSessionStore`
- 历史回放后 backlog 可重建
- message transform 后能看到 Magic projection，而不是只剩裸 todo 调用

验收：

- `todo_write` 能接收完整 Magic Todo 入参
- 原生 mux todo 存储继续工作
- 历史折叠后 backlog 仍可恢复

#### 3．tree-sitter native pack 在 Node v25 下失配

现象：

- `@kreuzberg/tree-sitter-language-pack` native subpath exports 不匹配

根因：

- `src/Shell/TreeSitterShell.fs` 直接 `resolve("@kreuzberg/tree-sitter-language-pack/ts-pack-core-node.{suffix}.node")`
- 该路径依赖包的 `exports` 暴露方式；Node v25 下这里不再稳定

最小修复优先级：中。

建议方案：

1. 先做代码侧兼容加载，不要先靠降级 Node 兜底。
2. `tryGetPack` 改成分层加载：
   - 先尝试包根导出
   - 再尝试当前 native subpath
   - 两者都失败时返回明确原因
3. 保持失败可见，不要 silent swallow；当前语法检查 wrapper 已经是安全失败，不需要再加兜底黑魔法。

建议改动文件：

- `src/Shell/TreeSitterShell.fs`

测试与验证：

- 单元层很难稳定模拟 Node exports 细节，建议补最少的 loader 单测 + 一次 Node v25 环境徒手回归
- 验证 `[syntax-check]` marker 仍出现，且正常返回 diagnostics 或明确失败原因

验收：

- Node v25 下编辑 wrapper 不再报当前 native pack load failed

#### 4．`readOnlyWarning` 需补真实集成锁，不急着改实现

现象：

- 徒手测试说 `executor` ro 模式下 `cat /etc/passwd` 未带 warning

现状代码：

- `src/Kernel/Executor.fs` 已把 `cat` 纳入 `readOnlyReadCommands`
- `prependSafetyWarningForExecution` 理论上会给 shell ro 输出前置 warning

判断：

- 更像“手测二进制落后于源码”或“实际走的不是 `vibe-fs executor` 路径”
- 当前还不能据此直接改实现

最小处理：

1. 先在当前 head 重新构建后复测。
2. 无论是否复现，都补一条 Mux 集成测试，锁住真实插件路径，而不是只锁 Kernel helper。

建议测试：

- `muxExecutorRoCatPrependsWarningSpec`

只有在当前 head 复测仍失败时，才进一步追查：

- `integrateVibeFsTools` 是否被原生 `bash` / `executor` 同名冲掉
- 实际调用是否经过 `summarizeWhenNeeded`
- UI 是否展示时吞掉了 warning 前缀

#### 5．`read` 去重 helper 已存在，但真实 mux 消息链没接线

现象：

- 用户实测连续 `read` 同一内容时，LLM 侧没有看到 `[No Change Since Previous Read/Write]`

现状分层：

- `src/Mux/Plugin.fs` 顶层确实已暴露 `collectReadOutputs`、`deduplicateReadOutputsWithSeen`、`deduplicateModelReadOutputsWithSeen`
- `tests/IntegrationToolTests.fs` 也已有顶层 helper 测试
- `../mux/src/node/services/messagePipeline.ts` 会在 provider 前调用 `transformVibeFsMessages`
- `../mux/src/node/services/vibeFsBinding.ts` 只是把调用委托给 `registration.messagesTransform`
- 但 `src/Mux/MessageTransform.fs` 当前只做 review replay、caps、wiki prelude，完全没有调用 read dedup

根因：

- 之前只证明了 dedup helper 本身可用
- 没证明真实 mux `messagesTransform` 已把 dedup 接到最终 LLM 可见消息链
- 因此当前 head 上会出现“helper 测试为绿，但真实 mux 上下文仍不去重”的分裂状态

最小修复：

1. 不改 `../mux` 主链路入口，修复面留在本仓库。
2. 在 `src/Mux/MessageTransform.fs` 的 `messagesTransform` 中补 read dedup。
3. 去重要以最终送给模型的消息数组为准，不能只在顶层 helper 自测里成立。
4. 必须沿用现有对象级替换逻辑，保留 `file_read` / `read` 当前输出形状，不要把 JSON 输出粗暴降成字符串。

建议改动文件：

- `src/Mux/MessageTransform.fs`
- 如需复用局部 helper，再小幅整理 `src/Mux/Plugin.fs`

必须补的测试：

- `tests/IntegrationToolTests.fs`
  - 新增 `muxMessagesTransformDedupsRepeatedReadSpec`
  - 直接走 `muxMessageTransform`
  - 历史里先放一次 `read` / `file_read` 输出，当前窗口再放同内容 read
  - 断言 transform 后最终消息里的重复 read 被替换成 `[No Change Since Previous Read/Write]`
- 现有顶层 helper 测试保留，但不再把它当作真实接线证明

验收：

- mux 实测重复 `read` 时，LLM 侧即时看到去重 marker
- 顶层 helper 测试继续为绿
- `wiki` / `caps` / review replay 注入顺序不被破坏

## 设计取舍

### `todo_write` 为什么不优先直接改 mux 核心

因为仓库约束已明确：

- 最好改本仓库
- 其次改 binding
- 最差才改 mux 本体

而 `vibeFsBinding.ts` 当前已经支持 wrapper 覆盖 schema 与 execute。`todo_write` 的能力缺口本质是插件语义没接上，不是 mux 原生 todo 工具本身必须重写。

### `websearch` 为什么要顺手修 `executor` 摘要路径

因为两个点共用同一类错误：

- Mux 运行 agentId 体系是 `exec` / `explore`
- `"executor"` 只是在 `vibe-fs` 工具名里成立，不在 mux task runtime 里成立

只修一个，另一个会留下同类 latent bug。

### tree-sitter 为什么不建议先靠降 Node 版本处理

降版本只是环境规避，不是根因修复。当前代码把 native subpath 当稳定 API 用，本身就脆。应先把 loader 设计修到对包导出结构变化更稳，再决定是否额外写环境约束。

## 最终验收标准

### 必须转绿

- `websearch` 可用
- 大输出 `executor` 摘要可用
- 重复 `read` 对 LLM 即时显示去重 marker
- `todo_write` 接收 Magic Todo schema，并能保留 `completedWorkReport`
- tree-sitter 在 Node v25 下恢复工作，或至少有明确、稳定、可诊断的兼容加载结果

### 允许不改实现，只需复测确认

- `read` 目录读取
- `fuzzy_grep` 暴露
- wiki 工具暴露
- slash command 路径
- `readOnlyWarning`（若当前 head 已正常）

### 必须修文档预期

- read 去重要验证“LLM 实际看到的消息”，不是只看原始 tool result
- wiki 工具有目录门控
- review 流程依赖 slash command / pluginCommands 路径

## 一句话实施策略

先重建复测，砍掉版本漂移噪音；再只修 4 条真链路：`exec` agentId、`todo_write`→Magic Todo、tree-sitter loader、mux read dedup 接线；最后把徒手测试指南改成与实际架构一致。
