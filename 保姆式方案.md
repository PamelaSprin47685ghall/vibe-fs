# OMP 宿主适配保姆式方案

## 0. 目标

- 路线固定：`F# -> JS`，不做 `TS` 重写。
- 事实源固定：复用 `vibe-fs` 的 `Kernel` / `Shell`，`vibe-omp` 只做参考对照，不做业务回流。
- 结构固定：业务语义进 `Kernel`，副作用进 `Shell`，`OMP` 只保留最薄宿主适配层。

## 1. 范围

### v1 要做

- `fuzzy_find` / `fuzzy_grep`
- `websearch` / `webfetch`
- `runner` / `runner_wait` / `runner_abort`
- `editor` / `greper` / `reverie` / `browse`
- `submit_review` / `submit_review_result`
- `/loop`、`before_agent_start`、`tool_result`、`agent_end`、`session_start`、`session_shutdown`、`input`
- `pi` 基址解析、`createAgentSession`、`prune` 猴补丁

### v1 不做

- 不接 `knowledge graph`
- 不做 `WorkBacklog`/`BacklogProjection` 折叠
- 不把 `todo_read` / `todo_write` 升成新的核心语义；仅保留宿主原生能力
- 不做 `vibe-me-engine` 依赖回流

## 2. Parity 矩阵

### Review 运行时

- `Plugin.fs` 模块加载时只做一次 `let reviewStore = Shell.ReviewRuntime.createReviewStore ()`，并把同一实例闭包给 `ReviewTools` / `SessionLifecycle`；不在 `src/Omp` 再造第二份 Map。
- `submit_review`、review child、`/loop`、`session_shutdown` 共享同一 store。
- `session_shutdown` 必须 deactivate 父会话 review，并 dispose/purge pending child。
- `/loop` activate 时直接调用 `activateReview(sessionId, task, createdAt)`，`createdAt` 统一取 UTC 毫秒时间戳；`addChild` / `setPendingReview` / `resolvePendingReview` 的顺序与 `vibe-omp/src/loop.ts` 对齐。
- OMP review child 明确不走 `ChildAgentRegistry` / `return_reviewer` 路径；编排 SSOT = `vibe-omp/src/loop.ts`，状态 SSOT = `Kernel.ReviewSession` + `Shell.ReviewRuntime`。
- `reviewInstructionsOmp` 放在 `Kernel.ReviewPrompts` 同级或 `Kernel.LoopMessages` 旁，正文只要求调用 `submit_review_result`；accept = `feedback: null` 或空白，reject = 非空字符串；正文不得出现 `return_reviewer` / `agent_report` / `verdict: PASS|REJECT`。

### Review 映射

| 旧 TS 行为 | F# 落点 |
| --- | --- |
| `Promise.race(deferred, waitForIdle)` + grace + `REVIEWER_NUDGE_PROMPT` | `Kernel.ReviewSession` + `Kernel.ReviewPrompts.reviewInstructionsOmp` 的 OMP nudge 文案 |
| `attachReviewChild` → `addChild` + `setPendingReview` | `Shell.ReviewRuntime` 同名 API |
| `detachReviewChild` → `resolvePending(..., terminated)` + `unlockReview` | `ReviewResult.Terminated` + `unlockReview` |
| child 工具集 `read` + `submit_review_result` | 已在白名单表，保持 |

### Review 首 prompt 模板

- `reviewInstructionsOmp` + `=== Change Report ===` + `report` + `=== Affected Files ===` + `affectedFiles` + 可选 `=== Original Task ===` + `task`。
- `task` 为空时省略 `=== Original Task ===` 段。
- `report` 与 `affectedFiles` 原样拼接，不改写 verdict 术语。
- 初始 prompt 必含 `submit_review_result`，且不得含 `return_reviewer`。

### Review 子循环算法

1. `ReviewTools.fs` 创建 child 后，先 `addChild(parentSessionId, childSessionId)`，再 `setPendingReview(childSessionId, resolve)`。
2. `await child.session.prompt(initialPrompt)`。
3. `nudges = 0`，while `nudges < 3`：
   - `race(deferred, child.session.waitForIdle())`；
   - 若 resolve，直接返回；
   - 若 idle，首次 grace `6000ms`，后续 `10000ms`；
   - `race(deferred, timeout)`；若未 resolve，`prompt(REVIEWER_NUDGE_PROMPT)`；`nudges += 1`。
4. 超过上限后返回 `terminated`，feedback 用当前 assistant text 兜底。
5. 子循环 finally 中统一 `resolvePendingReview(childSessionId, { terminated: true })`、`unlockReview(childSessionId)`、`abort`、`dispose`。
6. `submit_review` 外层 finally 无条件 `unlockReview(parentSessionId)`。

### Review 原语关系

- `Kernel.ReviewSession.decideAfterRound` / `promptParts` 只负责「轮次结果 → prompt 片段」的纯转换，不负责 OMP 的 `waitForIdle + grace` 竞态。
- `waitForIdle` 命中 `NoResult`，grace 结束仍未 resolve，则继续下一次 nudge；`Resolved` 直接结束。
- `unlockReview(childSessionId)` 只作为对齐 `vibe-omp/src/loop.ts` 的幂等收尾，child 不另行抢锁。

### OMP nudge 状态机

- `agent_end` 只做命令式编排，不把 `Kernel.Nudge.decide/update` 当总分发器。
- runner 分支的输入固定为 `hasActiveRunner=true`、`isLoopActive=false`，loop 分支固定为 `hasActiveRunner=false`、`isLoopActive=true`。
- 若任何分支内部调用 `Kernel.Nudge.decide/update`，`todos` 输入恒为 `[]`；open todo 只做只读投影，不参与 OMP `agent_end` 的分发与去重。
- loop 分支还必须满足 `ctx.sessionManager` 存在且 `!ctx.hasPendingMessages?.()`；否则不发 loop nudge。
- open todo 只作为投影输入和去重信号，不在 OMP `agent_end` 发 `NudgeTodo`。
- runner / loop 可各自做一次 `update` 去重，但仅限各自分支内部，不允许一次 `decide` 包办两类提醒。
- 这是相对 `vibe-omp` 的有意分叉：OMP 允许 open todo 与 review active 并存时仍按 runner/loop 规则发提醒，不让 todo 抑制 loop nudge。

### Nudge 存储与清理

- OMP 使用 `Omp/NudgeRuntime.fs` 作为薄 coordinator：模块级 state 只包一层 `Kernel.Nudge` / 去重窗口 / last action。
- `SessionLifecycle.fs` 是 `session_shutdown` 唯一清理点：同时清 `FuzzyIteratorStore`、`ReviewRuntime`、`NudgeRuntime`、runner job map。
- `clearNudgeSession` 不在 `ReviewTools.fs`、`RunnerTools.fs` 重复注册。

### Nudge parity

- 与 `vibe-omp/src/nudge.ts` 相比，OMP v1 不把 `todos` 喂给 `Kernel.Nudge` 总分发；这是有意收敛到 runner/loop 两段式的差异，不是遗漏。
- 若未来要恢复 `todo` 抑制 loop 的行为，需单列 v2 方案，不回填 v1。

### 子会话白名单

| 子会话 | 显式 `toolNames` | 备注 |
| --- | --- | --- |
| `editor` | `read`、`edit`、`write`、`find`、`fuzzy_find`、`fuzzy_grep`、`lsp` | 旧 `vibe-omp` 编辑子代理 |
| `greper` | `read`、`find`、`fuzzy_find`、`fuzzy_grep` | 旧 `vibe-omp` 搜索子代理 |
| `reverie` | 空集 | 只注入 `files`，不额外挂工具 |
| `browse` | `browser` | host 没有 `browser` 时直接报错，不静默降级 |
| `submit_review` review child | `read`、`submit_review_result` | 复用 `loop.ts` 语义 |
| `runner` runner child | `runner_wait`、`runner_abort` | 只负责后台等待 / 中止 |

### 子代理 prompt 路径

- OMP v1 走 parity 路径，但 `editor` / `greper` 不直接复用 `Kernel.Subagent.formatPrompt` 的多意图 schema；改用 OMP 单字符串适配层，把单字段 `intent` 变成最小 prompt 输入。
- `editor` / `greper` 的适配层走 OMP 专属 prompt 常量：`editorPromptOmp`、`greperPromptOmp`。
- `browse` 走 `browserPromptOmp`：只保留任务文本与 OMP 工具说明，不写 `stealth-browser-mcp` 叙述；host 没有 `browser` 就直接报错。
- `reverie` 先用 `Shell.WorkspaceFiles` 读文件，再拼 `meditatorPrompt`。
- 不接 `Kernel.Subagent.formatPrompt` 的多意图 schema，也不把 OMP 的单字段 `intent` 展开成 `objective/background/targets` 作为 v1 主路径。
- OMP 专属 prompt 常量放在 `Kernel.SubagentPrompts` 的 OMP 分支或单独 re-export 文件里，`src/Omp` 不手写长文案。

### 事件

| 事件 | 行为 | 复用事实源 |
| --- | --- | --- |
| `before_agent_start` | caps 注入 + strip host 前置提示 | `Kernel.CapsFormat` + `Shell.WorkspaceFiles` + `Shell.CapsFileCache` |
| `tool_result` | 语法诊断追加 | `Kernel.TreeSitterKernel` + `Shell.TreeSitterShell` |
| `agent_end` | nudge | `Shell.ReviewRuntime` + `Omp/MessagingCodec.fs` + `Omp/SessionLifecycle.fs` + `Shell.FuzzyIteratorStore` + `Shell.PromiseQueue` | runner/loop 两段式手写编排；`Kernel.Nudge` 仅作分支内去重，不作总分发 |
| `session_start` | 工具裁剪 | `Kernel.OmpSessionTools`（新增） |
| `session_shutdown` | 清理 session 级缓存 | `Shell.FuzzyIteratorStore` / nudge store / runner store |

### 对照旧实现

- 子会话白名单与 `vibe-omp/src/subagents.ts`、`vibe-omp/src/loop.ts`、`vibe-omp/src/runner.ts` 一一对照。
- `browse` 对应 `vibe-omp` 的 host browser 检查，不引入静默 fallback。
- `session_shutdown` 要清理 iterator scope、review runtime、nudge coordinator、runner job map 的 session 级状态。

## 3. 依赖边界

- `OMP` 侧 F# 代码只允许引用 `VibeFs.Kernel.*` 和 `VibeFs.Shell.*`。
- 明确禁止 `vibe-me-engine` 回流；TS 旧实现只做协议参照，不做依赖。
- 禁止 `OMP` 反向引用 `VibeFs.Opencode.*` / `VibeFs.Mux.*`。
- 禁止在 `OMP` 重抄 Kernel 的长提示词、方法论目录、权限矩阵、审查文案。
- `Kernel.OmpSessionTools.fs` 必须和 `HostTools.fs`、`Config.fs` 同一层级纳入 `vibe-fs.fsproj` 的 Kernel 段，不能只加 `src/Omp`。
- `HostTools` 的 `Host` DU 不扩展；OMP 专属规则只进 `Kernel.OmpSessionTools` 和 `src/Omp`。

### nudge 优先级

- **SSOT A**：`agent_end` 的 OMP 编排固定为命令式两段式：先 runner 分支，再 review loop 分支；**不得**一次调用 `Kernel.Nudge.decide`/`update` 后按返回值统一分发。
- `todo` 相关相位只作为 nudge 投影输入；v1 不在 `agent_end` 发 `NudgeTodo`，也不接 WorkBacklog 折叠，不注册、不包装、不回写 `Kernel.WorkBacklog`。
- `Kernel.Nudge` 的 open-todo 分支保留给 Kernel 级通用路径，不作为 OMP `agent_end` 的直接分支。

## 4. 命名与映射

- `OMP` 对外子代理名保留旧名：`editor`、`greper`、`reverie`、`browse`。
- 内核语义名固定为：`coder`、`investigator`、`meditator`、`browser`。
- 映射只发生在 `Omp/SubagentTools.fs`，不进入 Kernel。
- `websearch` / `webfetch` 的实现模块名写成 `Shell.OllamaClient`；`WebSearchApi` 只作为历史语义别名，不作为源码文件名。
- 不要按 README 的历史叙述去找 `WebSearchApi.fs`；实际源码文件名是 `src/Shell/OllamaClient.fs`。

### 子代理参数面

- `editor` / `greper` / `browse` 保持 vibe-omp 旧形状：单字段 `intent`。
- `reverie` 保持 `intent` + `files[]`。
- 不升级为 Mux 的 `intents[]` 多意图模型。
- `intent` 最终映射到 Kernel 的 canonical 角色与 prompt 组合，不把 OMP UX 名改写进 Kernel。

## 5. 工具裁剪

- 主会话判定：active tools 中命中任一子代理名即判为主会话。
- 主会话恒剥离：`bash`。
- 主会话剥离 child-only：`find`、`edit`、`write`、`lsp`、`fuzzy_find`、`fuzzy_grep`、`runner_wait`、`runner_abort`、`submit_review_result`、`search`、`glob`、`ast_edit`、`ast_grep`、`browser`。
- 保留：`read`、`editor`、`greper`、`reverie`、`browse`、`runner`、`submit_review`、`websearch`、`webfetch`、`todo_read`、`todo_write`。
- 裁剪规则写成 `Kernel.OmpSessionTools`：`ompSubagentToolNames`、`ompChildOnlyToolNames`、`ompAlwaysStripToolNames`、`filterOmpMainSessionActiveTools`。
- 判定规则：`activeTools` 与 `ompSubagentToolNames` 有交集 ⇒ 主会话；否则按子会话处理。

## 6. 模块划分

建议新建 `src/Omp/`：

1. `PiResolve.fs`：解析 `@oh-my-pi`，拿 `SessionManager`。
2. `Schema.fs`：`typebox` schema。
3. `Codec.fs`：`obj` ↔ 强类型记录。
4. `MessagingCodec.fs`：session entries ↔ Kernel 消息；同时提供 `entries -> todos / lastAssistantMessage` 的 nudge 输入投影。
5. `ChildSession.fs`：`createAgentSession`、`runSubagent`、abort 竞速。
6. `FuzzyTools.fs`：`fuzzy_find`、`fuzzy_grep`。
7. `WebTools.fs`：`websearch`、`webfetch`。
8. `RunnerTools.fs`：`runner`、`runner_wait`、`runner_abort`。
9. `SubagentTools.fs`：`editor`、`greper`、`reverie`、`browse`。
10. `ReviewTools.fs`：`submit_review`、`submit_review_result`、`/loop`；review child 只挂 `read` + `submit_review_result`。
11. `MessageTransform.fs`：`before_agent_start`、`tool_result`。
12. `SessionLifecycle.fs`：`session_start`、`session_shutdown`、`agent_end`、`input`。
13. `Tools.fs`：集中注册。
14. `Plugin.fs`：唯一默认导出工厂。
15. `PruneGuard.fs`：`PI_BASE` / pruning 补丁隔离。

### Runner 映射

- 对外工具名固定 `runner` / `runner_wait` / `runner_abort`。
- `runner` 参数固定 `language`、`program`、`dependencies`、`what_to_summarize`，不引入 `timeout_type` / `mode`。
- `runner` 的执行后摘要仍可 spawn 子会话，但只做结果压缩，不改变对外参数面；摘要固定走 `Kernel.SubagentPrompts.executorSummarizerPrompt` 的 OMP 占位参数：`timeout_type = "omp-runner"`，`mode = "rw"`。
- `runner_wait` / `runner_abort` 只用于后台任务，和 OpenCode 的 `executor` 命名不合并。
- `runner` 的对外 schema 只对齐旧 `vibe-omp` 的参数面；与 Kernel 侧 `executor` / 摘要 prompt 的差异要在方案里显式保留。
- `hasRunningRunnerJob` 的 OMP SSOT = `Shell.SessionExecutor` 活跃任务 + OMP session-local runner map；`session_shutdown` 必须同时清理两者。

## 7. 产物与入口

- Fable 产物沿现有单工程管线生成，`OMP` 入口编译到 `build/src/Omp/Plugin.js`。
- `Plugin.fs` 只导出一个默认工厂，形态与 `kunweiExtension(pi)` 一致，内部用 `WeakSet` 保幂等。
- 宿主加载时直接取 default export，不从 TS 旧包回拉运行时。
- `vibe-fs.fsproj` 里 `src/Omp/*.fs` 放在 `src/Shell/*.fs` 之后、`tests/*.fs` 之前；`tests/ArchitectureTests.fs` 追加 OMP 目录检查。
- 若 npm 包要并列暴露 OMP 入口，再加 `./omp` 子路径导出；否则只保留 `build/src/Omp/Plugin.js` 作为宿主入口。
- `Kernel.OmpSessionTools.fs` 也要插入 `vibe-fs.fsproj` 的 Kernel 段，位置靠近 `HostTools.fs` / `Config.fs` / `ToolCatalog.fs`。
- `PruneGuard.fs` 对 `patchDisablePrune().catch(() => {})` 保持同等失败静默语义。
- `Plugin.fs` 里 review store / nudge coordinator / prune patch 只初始化一次，不进测试热路径。

### 核心断言

- `fuzzy` iterator 单次消费。
- `webfetch` 阻断 SSRF。
- `caps` 幂等且有预算。
- `session_start` 剥离稳定。
- `agent_end` 优先 runner 再 loop。
- `submit_review_result` 三态稳定。
- `editor` / `greper` / `reverie` / `browse` 的 child prompt 含对应 `SubagentPrompts` 子串；其中 `browse` 只验证 OMP 版工具说明，不验证 stealth-browser-mcp。
- review child 初始 prompt 含 `submit_review_result` 且不含 `return_reviewer`。
- `runner` 事件链与 `vibe-omp/src/runner.ts` 的参数面一致。
- review child 只挂 `read`、`submit_review_result`，且 `submit_review_result` 只接受 `feedback: null|string`。
- `editor` / `greper` 测试只断言 OMP 单字符串适配层前缀，不要求完整 `CoderIntent` / `InvestigatorIntent`。
- `agent_end` 测试覆盖 runner gate、`hasPendingMessages` gate、review active gate 三个分支。

### 细节约束

- `entries -> todos / lastAssistantMessage` 只做 nudge 投影，不升级为 WorkBacklog 折叠。
- `MessagingCodec` 归一化 `SessionEntry`：兼容 `message.content` 与 `info` + `parts` 两种形态，`lastAssistantMessage` 与 `readAssistantText` 同构；`todos` 只产出扁平 open 状态列表，不引入新 phase 模型。
- `runner` 对外参数固定 `language`、`program`、`dependencies`、`what_to_summarize`；子会话只挂 `runner_wait` / `runner_abort`。
- `fuzzy_find` / `fuzzy_grep` 的 iterator scope 绑定 `sessionId || workspaceId`，`session_shutdown` 要清理该 scope。
- `vibe-omp/src/index.ts` 的 `CHILD_ONLY_TOOLS` 作为对照基线；若 v1 主动增收 `ast_edit` / `ast_grep`，需在方案中标注为「相对旧实现收紧」而非默认。

## 9. 迁移顺序
