# 新功能保姆级实现方案

## 背景
- 最根本的需求是：把 `../editplus` 的 `read` / `edit` 核心能力移植进当前仓库。
- 四入口点分流只是为了把这套能力按宿主场景打开或关闭，不是需求本体。
- 当前仓内与本需求直接相关的 `Opencode` server 入口点只有两个：`src/Opencode/Plugin.fs` 与 `src/Opencode/PluginMimo.fs`。
- 这次需求不是在现有入口点里加运行时开关，而是把入口点扩成四个：普通版 `Plugin` / `PluginMimo`，以及启用 editplus 的 `PluginPlus` / `PluginMimoPlus`。
- editplus 不是一个小补丁，而是一整套新的文件交互协议：`read` 不再按 `filePath + offset + limit` 读，而是给文件行分配稳定 tag；`edit` 也不再按 `oldString/newString` 做精确替换，而是按 tag 区间编辑。
- 这套协议天然是有状态的，因为 tag 只有放在一个 session 的连续 read/edit 历史里才有意义；但 opencode 重启后仍可能加载旧 session，所以状态不能只放内存里，必须能从历史消息 replay 恢复。
- 当前代码库里，`MessageTransform.fs` 已经承担“从 session 历史重建运行时投影状态”的职责，因此 editplus 状态恢复也应接在这一层，而不是另起一套旁路机制。
- 另外，`opencode` 的 plugin loader 对入口点文件和 package 导出有明确约束：一个模块里多导出的 legacy function plugin 会被全部尝试加载；一个 package 的 `server` 导出位也只有一个。因此这次“开不开 editplus”最稳妥的表达方式就是四个独立入口文件，而不是一个文件里多导出或一个 package 根入口里做运行时分支。

## 整体背景
- 这个仓库是 `vibe-fs`，本质上是一个用 F# 写、经 Fable 编译成 JS 的 `opencode` 插件/运行时实现。
- 代码分层大致是 `Kernel`（纯域逻辑与类型）、`Shell`（文件系统/执行器/外部副作用）、`Mux`（统一工具与包装）、`Opencode`（面向 opencode 宿主的集成层）。
- 这次需求只应该落在 `Opencode` 层，因为要改的是“宿主实际暴露给模型的工具集合”和“宿主消息历史的重建逻辑”，不是通用内核本身。
- 当前 `Opencode` 层已经有一套共享的入口构造器 `PluginCore.fs`，两个现有入口点都只是把 host 不同地传进去；这说明“入口点层分流”本来就是该层最自然的扩展方式。
- `Plugin` / `PluginMimo` 继续代表现有行为，`PluginPlus` / `PluginMimoPlus` 代表带 editplus 的变体，这样做的好处是：不用在同一个入口里塞运行时分支，也不用让下游猜一个布尔配置。
- editplus 的核心价值是把“文本匹配编辑”改成“稳定 tag 定位编辑”，它解决的是 LLM 编辑里最脆弱的那一段：旧文本稍微变一点，精确替换就会失效；tag 方案把定位和内容分离，编辑只依赖已读历史而不是旧字符串。
- 因为这个协议依赖历史连续性，所以它天然要求 session-level 状态；但 opencode 的实际使用场景包括重启后恢复旧 session，因此状态必须可从历史消息重建，这也是为什么 replay 必须挂在消息 transform / history projection 层，而不能只存在工具实现内部。

## 一句话
- 只改 `Opencode`。
- 通过四个入口点分流是否启用 editplus：`Plugin` / `PluginPlus` / `PluginMimo` / `PluginMimoPlus`。
- `write` 保持不动。
- `edit` 只走 tag 协议，不保留 `oldString` / `newString`。
- 状态不落盘，靠 session history replay 恢复。

## 已确认事实
- 当前 `Plugin.fs`、`PluginMimo.fs` 只是把 `pluginFor opencode` / `pluginFor mimocode` 作为默认导出。
- `opencode` 的 legacy loader 会把一个模块里所有导出的函数都当成候选插件；同文件多导出会被逐个执行，不能把两个入口点塞进同一个文件。
- `opencode` 的新 loader 也有“单个 plugin 文件/单个 default export”的约束；默认导出对象只能有一个 `server()` 或 `tui()` 入口。
- 因此四个入口点必须是四个独立文件，每个文件只导出一个默认工厂。
- 现有 `build/package.json` / `package.json` 的根导出指向 `build/index.js`，而 `package.json` 的 build 脚本当前把 `build/src/Mux/Plugin.js` 复制成 `build/index.js`，所以“包根”现在不是 `Opencode` 入口。
- 结论：四入口点更适合通过文件路径或独立导出路径显式选择，不要指望一个包根自动承载四个 server 入口。

## 分发结论
- 如果消费者是源码/文件插件路径，四入口点最简单：直接选 `src/Opencode/Plugin*.fs` 编译出的对应 JS 文件。
- 如果后续要做 npm 包分发，当前 loader 对一个 package 的 `server` 入口只有一个槽位；四个变体要么做四个独立包，要么做四个独立文件路径并用 file plugin spec 指向它们。
- 不建议把“Plus 是否启用”做成运行时布尔参数；本次改法就是“入口点本身就是开关”。

## 核心设计
- 入口点只决定“是否启用 editplus”。
- `Plugin` / `PluginMimo` 走现有工具集，不额外暴露 `read/edit`。
- `PluginPlus` / `PluginMimoPlus` 在现有工具集之外，额外覆盖同名 `read/edit`。
- `write` 仍旧由现有实现提供，不要碰。
- `EditPlusState` 只保存可重建状态：tag registry、mtime、文件锁、当前文件段映射。
- session 重启后，不从磁盘快照恢复，而是从消息历史 replay。

## editplus 核心算法必须翻译的点
- `numToTag` / `tagToNum`：base52 tag 编码。
- `splitLines`：保留 CRLF / CR / LF 的行尾。
- `formatTag`：按 tag 输出 `tag|line`。
- `resolveTag`：把 tag 解析成当前行位置，并区分 expired / stale / external。
- `readRange`：`begin` + `endExclusive` / `endInclusive` 区间解析。
- `TagRegistry`：
  - 给每个文件分配稳定 tag 段。
  - 维持 `zList` / `xList` 之类的前后映射。
  - 编辑后重排受影响段，未受影响 tag 保持稳定。
  - 外部修改触发 stale / external 错误。
- `withLock`：同文件串行写入。
- `appendSummary`：错误时自动附带当前文件摘要，方便重读再试。

## read 行为
- 输入协议只认 editplus 的 tag 协议：`path`、`begin`、`endExclusive`、`endInclusive`。
- 没有 range 时返回完整 tag 视图；大文件可返回结构化摘要视图。
- 有 range 时只返回指定区间。
- 目录读可以保留目录列表能力，但不要再回到旧分页语义。
- `read` 不再支持旧的 `filePath` / `offset` / `limit` 兼容口子。

## edit 行为
- 只接受 `begin`、`endExclusive` / `endInclusive`、`content`。
- 不再支持 `oldString` / `newString`。
- `content` 为空表示删除。
- 插入 = `begin == endExclusive`。
- 编辑后更新 registry，并返回新 tag 信息。
- 跨文件、反向区间、过期 tag、外部修改都必须报错。

## 状态恢复
- 现有 `src/Opencode/MessageTransform.fs` 已经承担“从 history 重建状态”的角色，editplus 应接在这里。
- 重启后，从 `MessagingCodec.decodeMessages` 的历史消息里 replay 所有 `read/edit` 相关 tool parts。
- replay 目标是把 tag registry 重建到“当前会话最后一次成功 read/edit 之后”的状态。
- 不要额外落盘快照，不要做内存 fallback；历史就是单一真源。

## replay 依据
- `src/Opencode/MessagingCodec.fs` 的 `ToolState` 已经包含 `status`、`output`、`error`、`input`。
- replay 只需要看：
  - `toolName` 是否为 `read` 或 `edit`
  - `state.status` 是否为 `completed`
  - `state.input` 中的 `path` / `begin` / `endExclusive` / `endInclusive`
  - `state.output` 的实际结果文本
- `src/Kernel/Messaging.fs` 已有 `ToolPart(toolName, callID, state, raw)` 形状，足够回放。

## 结构摘要
- `editplus` 原版 `read` 会用 tree-sitter 生成结构摘要；`vibe-fs` 里已有 `TreeSitterShell.fs` 能检测语言并做 syntax parse。
- 推荐把结构摘要能力接到现有 `TreeSitterShell.fs`，新增一个纯函数来抽取结构起始行，然后在 `read` 输出时做稀疏显示。
- 如果实现时间不足，最低限度也要先保证“完整 tag 视图 + range 读取”正确；结构摘要可作为第二阶段，但不建议长期跳过。

## I/O 与边界条件
- 保留 BOM / CRLF / LF 的原样写回。
- 读写时不要默默改行尾。
- 文件大小上限、输出截断、异常信息都要明确。
- `read` 的输出最好参考 editplus：大文件不直接整页倒出来，避免 host 输出爆炸。

## 与现有 host 集成的注意点
- `read/edit` 一旦变成 tag 协议，UI / 记录层如果只认旧输入字段，可能看不到正确的定位信息。
- 当前宿主对 `read` 的定位字段偏向 `filePath`，对 `edit` 的定位字段偏向 `filePath` / `patchText`，这意味着 Plus 版如果完全只用 `path` / tag，前端的 file chip 可能退化。
- 本次要求是不做兼容，但实现时最好在内部记录里保留“解析后的绝对路径”和“编辑范围”供展示层使用，而不是为了旧 API 去兼容参数。

## 入口点实现建议
- `src/Opencode/Plugin.fs`：`pluginFor opencode false`
- `src/Opencode/PluginPlus.fs`：`pluginFor opencode true`
- `src/Opencode/PluginMimo.fs`：`pluginFor mimocode false`
- `src/Opencode/PluginMimoPlus.fs`：`pluginFor mimocode true`
- `PluginCore.fs` 里把 `editPlusEnabled` 传给工具注册与 message transform。
- `Tools.fs` 里只在 `editPlusEnabled = true` 时把 `read/edit` 加进 `tool` object，其他工具不变。

## 为什么要分入口点，不用编译期开关
- 现在开关的语义是“谁加载哪个入口点”，不是“同一入口根据配置分支”。
- 这样能避免一堆 runtime 兼容分支，也避免把行为选择藏进全局配置。
- 入口点就是开关，最直接，也最容易测试。

## 测试清单
- `Plugin` 不包含 editplus `read/edit`。
- `PluginPlus` 包含 editplus `read/edit`。
- `PluginMimo` 不包含 editplus `read/edit`。
- `PluginMimoPlus` 包含 editplus `read/edit`。
- `read` 正常输出 tag 视图、结构摘要、range 读取。
- `edit` 能做替换 / 插入 / 删除。
- stale / external change 报错正确。
- restart 后 replay 可恢复旧 tag 状态。
- `write` 的现有测试全部不动且继续通过。

## 需要特别补的回归点
- `tests/IntegrationPluginTests.fs`：四入口点 shape / tool 注册差异。
- `tests/IntegrationToolDefSpecs.fs`：确认 Plus 版 tool definition 里出现新的 `read/edit`。
- `tests/IntegrationWikiPreludeSpecs.fs`：确保 message replay 不破坏 wiki/caps 注入。
- `tests/IntegrationAfterHookSpecs.fs`：`write` 相关记账行为保持不变。
- 新增 editplus 专项测试：
  - tag 编码
  - range read
  - edit replace/insert/delete
  - stale/external change
  - 重启后 replay

## 实现顺序
1. 先翻 `EditPlusState`，把 tag registry / lock / stale detection 做成纯 F#。
2. 再翻 `EditPlusTools`，把 `read/edit` 工具接出来。
3. 再改 `PluginCore.fs` / `Tools.fs`，按入口点开关是否注册 plus 工具。
4. 再补 `PluginPlus.fs` / `PluginMimoPlus.fs`。
5. 再接 `MessageTransform.fs` 的 replay。
6. 最后补测试并跑 `pnpm build && pnpm test`。

## 验收标准
- 四个入口点都存在且语义明确。
- 只有 Plus 入口点启用 editplus。
- `write` 完全没变。
- 旧 session 重启后能继续使用历史 tags。
- 不需要额外运行时开关。
- `pnpm build && pnpm test` 通过。

## 额外结论
- `opencode` 这边的工具加载语义决定了：四个入口点必须是四个独立默认导出文件，别把两个入口点塞一个模块里。
- 如果后续要做 npm 分发，单个 package 的 `server` 入口只有一个槽位，必须另想打包策略；本次先按文件入口实现最稳。
